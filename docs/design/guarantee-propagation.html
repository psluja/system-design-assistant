Guarantee Propagation

Consistency, ordering and delivery semantics as computed verdicts along every request path — the invisible production bugs no diagram tool checks, made visible with 100%-certain lattice arithmetic.
SDA design document · 2026-07-03 · status: for owner review — not started
About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a client-side tool in which a software architect draws a system architecture on a canvas and the tool computes the design: load, latency, availability, cost — judged against declared requirements. This document proposes a new computed dimension: qualitative guarantees.

guaranteea qualitative promise a path makes about data: how fresh a read is (consistency), whether messages keep their order (ordering), whether a message can be duplicated or lost (delivery).
latticea small ordered set with a "weakest-wins" combining rule — the mathematical structure that makes these labels computable with certainty.
flowone request path through the design, from its traffic origin to its terminal.
verdictthe tool's computed judgment of a declared requirement: ok, violation, or honestly "unknown" with what would resolve it.

1The bug class this catches

A user posts a comment, refreshes, and the comment is gone — the write went to the database writer, the read came from a replica still catching up. Two order events arrive swapped — the stream passed through a fan-out that keeps no order. These failures are invisible on every diagram and surface only in production. They are also — and this is the key insight — mechanically derivable from the wiring, because guarantees only ever degrade along a path.

2The model — three independent dimensions, each a tiny lattice

DimensionValues (strongest → weakest)Combining rule along a path
Consistency (read freshness)strong → eventualweakest hop wins; a cache hop with no declared invalidation contributes eventual
Orderingtotal → per-key (partition/group) → noneweakest hop wins; fan-out contributes none
Deliverytwo independent flags: may-duplicate, may-losemonotone OR — once a hop can duplicate (at-least-once) or drop (at-most-once), the path can

Every component's ports declare their contribution as sourced data, exactly like every other catalog number: an Aurora writer provides strong; a read replica provides eventual (replication lag is documented behavior); Kafka preserves per-key order and is at-least-once (may-duplicate); SQS standard is none/may-duplicate; SQS FIFO is per-key with effective exactly-once — and we already model its 300 msg/s price. The engine sees only opaque ordered tokens and the meet operation; what the tokens mean is entirely content, preserving domain-agnosticism.

3The 100%-computability contract (the owner's condition)

Everything in this feature is either certain or declared — nothing is inferred.
Certain (computed)Why it is 100%
the path meet, the verdict, the root-cause hopfinite-lattice arithmetic on declared tokens — deterministic, no probability, no heuristics; soundness is a THEOREM (monotonicity: adding a hop can never strengthen a guarantee), pinned by property tests and a differential against an independent DataScript relational implementation (two engines must agree, our standard rigor)
Declared (an assumption, badge-tracked)How it stays honest
each component's contribution (replica = eventual, FIFO = per-key…)sourced from official documentation, carried as data with provenance — rows in the design document's assumptions register like every other number
architect overrides ("my cache invalidates on write")explicitly set on the instance → badged architect; their truth is the architect's responsibility, stated in the document
Refused (would be a guess)What happens instead
read-your-writes via session stickiness; consumer idempotency; correctness of custom invalidation logicthese depend on application code the model cannot see → the verdict reads unknown naming exactly which declaration would resolve it. NOT included in v1's computed lattice — no fake certainty, per the owner's rule

4UX/UI — how the architect sees and touches it

SurfaceBehavior
Declaring a requirementthe SAME gesture as every SLO: in the flow's Requirements section (web Inspector / VS Code native tree) pick "Ordering ≥ per-key" or "Consistency = strong" or "Duplicates: not allowed". No requirement anywhere → the whole feature is silent (no-filler rule).
Canvas, with a requirement declaredthe flow's edges show a slim guarantee STRIP under the wire (distinct from transform pills): teal while the running meet still satisfies the requirement, switching to red FROM THE DEGRADING HOP onward — you literally see where the promise breaks. Hover names it: "ordering: per-key → none — a fan-out topic keeps no order".
The degrading hopgets the standard violation treatment: red halo, a Problems row "flow users→billing: ordering degraded below requirement at topic1", and a REMEDIATION with computed trade-offs: "switch queue to FIFO — restores per-key; documented ceiling 300 msg/s (your flow: 45 msg/s ✓, +$X/mo)". The remediation reuses compare_options machinery — suggestions are computed, never advised from air.
Inspector (component)a read-only "Guarantees" block on every port showing its declared contribution with the provenance badge (documented/est./architect) + the override control for the declarable cases (e.g. cache invalidation declared = consistency contribution becomes strong-on-hit, badge flips to architect).
System panelper-flow guarantee summary line: "consistency eventual (replica1) · ordering per-key ✓ · duplicates possible (sqs1)" — the parenthesized node is the root cause, clickable to reveal.
Generated design documenta "Guarantees" section: per-flow table (requirement / computed / root cause), and every declared contribution appears in the assumptions register. Unknown verdicts print with their resolving declaration — honest to the review board.
MCP (AI agents)set_guarantee_slo, guarantee results in evaluate/verdicts with the same guided errors; list_components/describe_component show declared contributions.
Synthesis (level-3, R3+)a declared guarantee requirement becomes a clingo constraint the topology enumerator must satisfy — the same seam that already enumerates protocol-legal completions now also rejects a candidate whose propagated meet would fall below the requirement (e.g. "Ordering ≥ per-key" filters out standard-SQS wirings, admits FIFO). So synthesize/compare_options return only designs that MEET the qualitative promise, cost-ranked as today — guarantees as first-class synthesis constraints, not a post-hoc filter. R1 ships the lattice + forward propagation the constraint reads; wiring it into the ASP model is R3+ scope (owner-confirmed).
