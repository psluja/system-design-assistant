Request Classes

Multi-commodity flows over a shared topology — the owner-commissioned answer to “systems connected each-to-each, synchronously”. Today SDA computes over ONE implicit river: traffic flows along every wire. A full mesh (service A calls B for orders; B calls A for reports) then reads as a per-request cycle and is refused — even though reality is two acyclic classes of request that merely happen to share the same drawing. This document adds a request class: a named flow with its own origins and its own wire membership, so per-class acyclic paths run over a shared, possibly cyclic, topology. The default is a single implicit class containing every wire — so every existing design means exactly what it always did, at zero migration cost.
SDA design document · 2026-07-04 · request classes (multi-commodity flows) · status: for owner review — not started
About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas and the tool computes the design — load, latency, cost, feasibility — against declared requirements, live on every edit. Underneath, SDA is a domain-agnostic engine that turns the drawing into a typed-property graph and a family of solvers, over which a content pack layers all cloud meaning as data. The engine propagates each quantity (throughput, latency, availability, cost…) along the wires with a per-key algebra, checks the result against declared bands, and — run backwards — optimizes, repairs, or explains. A separate discrete-event simulation (the DES) gives true percentiles over time. This document extends the flow model: it lets one topology carry several independent request flows, so a mesh whose drawing has cycles but whose traffic does not is computed honestly instead of being rejected.

request classa named flow: its own set of origins (nodes that inject traffic for the class) and its own wire membership (the edges the class actually traverses). One class's paths are acyclic even where the shared drawing is not. The default class contains every wire and every origin — exactly today's single river.
flow / origin / terminala flow is a request's journey from an origin node to a terminal node along the wires. Today a design's flows are its connected components (content/sda/src/system.ts requestFlows); a class refines this — several flows can share one component.
multi-commodity flowthe standard name (from network optimisation) for several distinct commodities routed over one shared graph, each with its own sources/sinks, all sharing the edges' and nodes' finite capacity. A request class is exactly a commodity.
SLO / band · verdictan SLO is a declared requirement on one property of one place (e.g. latency ≤ 200 ms), internally a band. A verdict is the tool's honest read-back: ok / warning / violation / unknown. Uncertainty is a value, never a guess.
flow key · ρ · overflowa flow key is a quantity that travels the wires and is reshaped by port transforms — today exactly one: throughput (content/sda/src/registry.ts line 118, flow: true). ρ = offered load ÷ capacity; at ρ ≥ 1 a queue grows unbounded. overflow = offered − served (the rejected rate).
cell networkthe engine's internal projection of the drawing: one in / out / local cell per (node, key), solved to a least fixpoint (engine/solve/src/network/build.ts). The MiniZinc optimiser and the native solver both read the SAME cells; classes must extend the cells, not fork the math.

1The workflow this serves — the each-to-each mesh

The owner named the case directly: real back-ends are wired each-to-each. An orders service calls an accounts service synchronously to place an order; the accounts service calls the orders service synchronously to build a report. On the canvas that is two wires, A→B and B→A. There is no cycle in the traffic — an order request never comes back to its own caller — but there is a cycle in the drawing. SDA today has exactly one notion of “how load moves”: it moves along all wires at once, as one river. So the drawing's cycle becomes a flow cycle, and the honest machinery refuses it.

This is not a bug in the flow model; it is the flow model reaching the edge of what a single commodity can express. The fix the owner asked for is the standard one from network optimisation: stop pretending one river, and let the topology carry several commodities — here, request classes — each with its own origins and its own subset of wires, each acyclic on its own, all sharing the same physical nodes and their finite capacity. The name for the future extension was already reserved in the latency-semantics work (§2 below); this document designs it.

The design center, stated as one sentence. A class is a named flow = (its origins, its wire membership). Everything the engine already does — propagate a flow key along wires, cut at async edges, roll up latency/availability/cost, propagate guarantees, simulate percentiles — is repeated once per class over that class's wires, then composed at the shared nodes. The default is one implicit class over every wire, which is byte-for-byte today. The whole feature is additive: old designs are new designs with one class.

2The single river today — where it lives, and why the mesh is refused

To extend the flow model without breaking it, we first pin exactly how it works. Every claim here is verified against the named file and line.

2.1 One river: every wire carries the flow

FactIn the code today
The engine builds the incoming adjacency from every edge in the graph, indiscriminately — there is no notion of “which flow” an edge belongs to.engine/solve/src/network/build.ts lines 255–267: a single loop over graph.edges pushes each into incoming/outgoing.
A flow key's inflow is the fan-in of every incoming edge's transformed contribution: in(N,K) = Σedges f(out(up,K)); served out(N,K) = combine(local, in).flowInflow (build.ts 312–326) and the per-key assembly (build.ts 373, 392–398). The async cut is a per-edge filter: ups = incoming.filter(u => !(onAsyncEdge==='cut' && u.async)) (build.ts 364).
throughput is the ONLY flow-flagged key; its algebra is min down a path, sum at a fan-in, carried across async.registry.ts 118: aggregate:{ series:'min', fanIn:'sum', onAsyncEdge:'carry', flow:true }. The offered load is inflow(throughput)+self(originRps); overflow = max(0, offered − self(throughput)), applied uniformly by withOverflow (content/sda/src/behaviors.ts line 62, 279).
A design is partitioned into flows by connected component — so a mesh is ONE flow with one source and one terminal, never two.requestFlows (system.ts 89–113): union-find over the wires, one {ids, source, terminal} per component. The each-to-each mesh is a single component ⇒ a single flow.

2.2 Why the mesh is refused (three honest mechanisms, one root cause)

Because there is one river, an each-to-each pair is a genuine flow cycle, and three independent parts of the honest machinery react to it — none of them can express “two acyclic classes”:

MechanismWhat it does with the mesh todayWhere
Search rejects (optimize / repair / explain)Both the incumbent MiniZinc compile and the native solver reject a cyclic dependency among the freed cells outright: “search over a cyclic flow is not supported”. The mesh cannot be optimized or repaired at all.incumbent hasCycle (engine/solve/src/minizinc/search.ts 209–211, 284); native hasCycleAmongDerived (engine/solver-contract/src/native/model.ts 107–109, 177–214).
Latency folds cycle-guard (degenerate, not rejected)The response and cumulative-latency folds stop at a re-visited node and return its own term — so a mesh produces a number, but a silently truncated one that answers no real question.queueing.ts realCumulativeLatency line 159, responseLatency line 199 (both: if (onPath.has(id)) return own(id)).
Guarantee walk cuts cycles (per path, already)Guarantee propagation already enumerates simple source→terminal paths, cutting a node it has revisited — so it is the one part of the engine that already thinks per-path rather than one-river.enumerateFlowPaths (engine/solve/src/guarantee/propagate.ts 118–152), propagateFlow (163–165).
The honest read-back today is a contradiction the tool should not tolerate. The scalar forward pass reaches some fixpoint on the mesh (a relay's min(cap, …) settles), the latency folds return truncated numbers, and search refuses entirely. Three surfaces, three different behaviours, one drawing — and none of them is the answer the architect wants, which is: “there are two request classes here; here is each one's load, latency and cost.” Request classes replace all three reactions with one correct one.
The pre-authorisation from latency-semantics v2. The v2 latency design (docs/design/latency-semantics-v2.html §4.4) already listed this exact topology — “full-mesh / any-to-any, each request using an acyclic subset” — and named the fix positively: “a well-defined response depends on WHICH request we mean: the perspectives are per-request-class… a multi-commodity (request-class) flow problem: one flow per class, each acyclic, sampled independently.” It scoped itself to a single commodity and deferred this document. Request classes are that deferred extension; when they land, v2's own suffix sampler (§6 here) “runs once per class” exactly as v2 predicted.

3The model — a class is a named flow the meta-model addition

A request class is declared as data, in the same spirit as everything else in SDA: no code, references by id. It carries two facts and nothing more.

// A request class — pure DATA, added to the design document (export schema bumps 3 → 4).
interface RequestClass {
  readonly id: string;                              // "order", "report" — the class's name on the canvas
  readonly origins: readonly { node: NodeId; rps: number }[]; // where THIS class injects load (its per-class originRps)
  readonly wires: readonly EdgeId[];               // the edges this class traverses (its membership)
}
// Absent ⇒ the design has ONE implicit class over EVERY wire and every node's originRps — today, bit-for-bit.
PropertyPositive definitionReduces to today when…
Default: one implicit class
zero migration		 A design with no declared classes has a single implicit class whose wires = all edges and whose origins = every node's existing originRps (registry.ts 89). The cell network, the DES network and every verdict are then identical to today. always — this is the whole design corpus as it stands.
Per-class acyclicity Each class's wire membership, as a directed subgraph, must be acyclic. The engine's cycle guard (§2.2) moves from “the whole topology” to “each class's edge subset”. A mesh is legal iff every class carves an acyclic slice of it. a single class over an already-acyclic drawing — the guard fires on the same graph as today.
Shared nodes, separate paths The classes overlap on nodes (the physical services are the same) and may overlap on wires (a shared DB edge can carry both order and report load). What differs per class is the route and the origins. one class — every node is crossed by exactly one river, so “shared” is vacuous.
Why not derive classes automatically from reachability? Tempting, and wrong: in a mesh, plain reachability from an origin cannot separate the two acyclic subsets — that ambiguity is the entire reason the mesh is hard. The class's wire membership is the minimum information that disambiguates it, so it is declared, not inferred. (An assist that proposes a membership from a chosen origin→terminal pair is a fine UX affordance — §7 — but the truth of record is the declared set.)

4Per-class arithmetic — fold once per class, contend at the node

This is the heart of the design and the source of its three hardest tensions. The rule is simple to state and subtle to honour: load composes per class along the class's wires; capacity is shared and contended at the node; latency, response, lag and guarantees are read per class along the class's path. Everything is a per-class repeat of an existing fold, joined at the shared node by one new term.

4.1 Load composition — per-class flow cells, one shared-capacity split

Today the flow key lives in cells in:N:K, out:N:K, local:N:K (build.ts 40–44). The generalisation indexes the flow cells by class — in:N:K:C, out:N:K:C — while capacity, cost and every non-flow key stay unindexed (one physical node). Per class, the inflow fold is exactly today's flowInflow, but summed only over that class's edges:

// Per-class offered load — today's flowInflow (build.ts 312–326), restricted to the class's wires E_C:
offered(N, C) = inflow(N, throughput, C) + originRps(N, C)          // class C's arrivals at N
             = Σe ∈ E_C into N f_out(e) · out(up(e), throughput, C)   // grouped by in-port, f_in per port — unchanged

// THE ONE NEW TERM — the shared node contends across classes (processor-sharing default):
totalOffered(N) = ΣC offered(N, C)                              // all classes queue at the SAME servers
served(N, C)    = offered(N, C) · min(1, capacity(N) / totalOffered(N))  // each class's share of a shared ceiling
overflow(N, C)  = offered(N, C) − served(N, C)                        // per-class rejected rate
overflow(N)     = ΣC overflow(N, C) = max(0, totalOffered(N) − capacity(N))  // = today's overflow, exactly

Two properties make this the honest default and keep it behaviour-preserving:

  • Unsaturated is exact and separable. When totalOffered(N) ≤ capacity(N) — the design target that the search's ρ-headroom already enforces — the split factor is 1, so served(N,C) = offered(N,C): each class passes through untouched, the fold is fully separable per class, and there is no cross-class coupling. This is the regime almost every real design sits in.
  • Saturated is processor-sharing, and it reduces correctly. When the node is overloaded, its finite capacity is split in proportion to each class's offered load — the M/M/c processor-sharing / PASTA default, the honest choice when no priority is declared. With one class the factor cancels and served = min(capacity, offered) — exactly today's out = min(local, in) and overflow = max(0, offered − cap) (behaviors.ts 62). Total overflow is class-count-independent.
Tension #1 (the hardest): the saturated split is non-separable, which threatens per-class monotonicity. Raising class C's load lowers the served rate of other classes at a shared saturated node (they contend). So per-class served throughput is not monotone in cross-class load once totalOffered > capacity — and monotonicity is precisely the property the native solver's branch-and-bound relies on (§5). The resolution is a scope, honestly drawn: the coupling term is inactive in the unsaturated region, which is where the search's headroom constraint keeps every candidate design; the saturated split is a boundary the scalar reports as a proportional estimate and the DES measures as truth (like every other queue-dynamics quantity). Priority classes (weighted, not proportional) are a named future extension, not R1 — §9.

4.2 Latency, response & lag — one perspective per class crossing a node

Latency semantics v2 already established that a node has one response perspective, and that a node crossed by more than one request has one perspective per request-class (v2 §4.4). Request classes make that literal. The split is clean because a shared queue's per-hop wait is class-blind while the path composition is per class:

Tension #2, resolved: acyclicity is a per-class property, so the cycle guard MOVES rather than loosens. Under one river the guard sits on the whole topology (search rejects any cycle; the folds truncate at a revisit). Per class, the guard runs on E_C: a class whose own path cycles (A→B→A within one class — a synchronous self-wait) is still refused, honestly, exactly as v2 §4.4 keeps refusing it — there is nothing finite to measure. What becomes legal is only the case where every class is individually acyclic. The refusal does not weaken; it re-targets from the drawing to the commodity.

4.3 Guarantees — the lattice walk already follows class paths

Guarantee propagation is the part of the engine that already thinks per-path: enumerateFlowPaths walks simple source→terminal paths and cuts revisited nodes (propagate.ts 118–152), and propagateDimension meets tokens along each (46–63). Per class, the only change is which edges the walker may traverse: restrict enumerateFlowPaths to E_C and run it from the class's origins. The lattice meet, the first-drop root-cause attribution and the “touched-unknown ⇒ unknown” honesty are all unchanged — they were already per-flow. A per-class guarantee requirement (“the order class must stay read-your-writes end to end”) is judged on the order class's paths alone, and its root-cause hop is named on the order class's wires.

5Solver implications — honestly size · monotonicity · the oracle

Classes touch the numeric core, so the solver story must be told without hand-waving: how big the model grows, which assumptions survive, and what the differential oracle must gain before the native solver is trusted on a multi-class design.

5.1 Size — cells × classes, but only for flow keys

Only the flow key's cells gain a class index. A design with |N| nodes, one flow key and |C| classes grows its flow cells from O(|N|) to O(|N|·|C|); every non-flow key (latency, availability, cost, the pool/queue/limit keys — the large majority of the registry) stays single. The fixpoint is still one dependency-ordered sweep per acyclic class-slice (build.ts 197–233), so evaluate stays within the <16 ms hot-path budget for realistic class counts. The class count is the growth axis to bound (a sanity cap — §8), not the node count.

5.2 Monotonicity — preserved per class in the headroom regime, lost at a shared saturated node

The native solver (engine/solver-contract/src/native/model.ts) is exact because, after freeing the tunables, every cell is a deterministic acyclic function of the knob vector, and the capacity/flow family is monotone — the property the oracle's generator deliberately keeps its designs inside (engine/solver-contract/src/harness/generator.ts 137–139: “every axis except declined-* stays INSIDE the monotone capacity/flow class, so the native solver and the incumbent MUST still agree”). Per class, each class's fold is the same monotone min/sum — monotone per class. The one place it breaks is the shared saturated split of §4.1: served(N,C) decreases in a sibling class's load, a non-separable coupling that lands multi-class-under-saturation in the same “outside the monotone class” category as the existing declined-* axes. In the unsaturated region the search's ρ-headroom constraint enforces (inflow ≤ factor·capacity, the Headroom request field), the coupling is inactive and per-class monotonicity holds — so optimize/repair remain exact where they search.

Tension #3 — the headroom lesson, stated so it is not re-learned the hard way. The differential oracle certifies the native solver only over what its generator produces. TASK-79's suite is trustworthy precisely because its coverage axes were made explicit and every axis's monotonicity was pinned; an unexercised structure can pass the whole green suite and still be wrong. A multi-class design — especially one with a shared saturated node, where per-class monotonicity fails — is exactly such an unexercised structure today. Therefore the oracle harness must grow a CLASS AXIS — generating multi-class designs, including shared-saturation contention — and that axis must be green (native agrees with the MiniZinc referee, or the case is an honest declined-*) BEFORE the native solver is ever bound for a multi-class design. The generator's axis list (generator.ts 141–152) is the exact seam to extend; do not switch the runtime solver on classes until the axis exists and passes. This is the “gaps are the goal” discipline applied to a new structural dimension.

5.3 MiniZinc emission — the same per-class cells, one linearisation caveat

The MiniZinc projector emits a var + equality per non-constant cell (engine/solve/src/minizinc/search.ts compile 146–254); indexing the flow cells by class multiplies the vars by |C| but changes nothing structural — each per-class equality is the same arithmetic. The single caveat is the saturated split's division by totalOffered: in the unsaturated region the search stays in (headroom), the factor is 1 and the constraint is linear; the saturated proportional split is non-linear and is not emitted as a forward equality — it is treated like tail latency and overflow-under-load: a DES-measured truth, with the scalar carrying the proportional estimate for display. This keeps the emitter linear where it must be and honest where it cannot.

6The DES — tag jobs with class, key reservoirs by (node, class) per-class time truth

The simulator already forks a request at fan-out, tags forks with retry generation, records a per-node response suffix and per-lineage lag stamps. Classes ride on the same machinery: a job carries its class; routing follows the class's edges; the response and lag reservoirs gain a class key.

7UX & MCP — declare a class, colour its river surfaces

The architect must be able to say “these wires are the order class, those are the report class” with the least possible ceremony, and read each class back distinctly. The interaction is a class lens: a filter over the canvas that colours each class's river and scopes the numbers shown.

  • Declaring membership. A class is {id, origins, wires}. The primary gesture: pick a class in the lens, then click wires to toggle their membership (or draw an origin→terminal path that the tool proposes as a membership, which the architect accepts/edits). A wire may belong to several classes (a shared DB edge serves both) — membership is a set, not a partition.
  • Per-class origins. A class's origins are declared on the class, defaulting from the nodes' existing originRps. This reconciles with today's node-local originRps: the implicit class inherits each node's full originRps; when the architect splits classes, each class takes a share and the node's total is their sum (a validation, not a new field on the node).
  • The default view. No classes declared ⇒ the canvas shows the single implicit river, identical to today. One or more declared ⇒ the lens defaults to “all classes” (each river in its own colour) with a picker to isolate one; the selected node's card shows per-class chips.

7.1 MCP — least input, self-correcting

// Additive tools; omitting classes leaves every existing tool behaving exactly as today.
declare_class({ id: "order", origins: [{ node: "orders_svc", rps: 800 }], wires: ["w3", "w5"] })
set_class_membership({ class: "report", wire: "w7", member: true })   // toggle one wire
// existing tools gain an OPTIONAL class scope; absent ⇒ the implicit "all" class (today's behaviour):
set_slo({ node: "accounts_svc", key: "latency", max: 200, class: "order" })  // per-class latency SLO
evaluate({ class: "report" })                                       // read the report river's roll-up

The tool self-corrects like the rest of the MCP surface: an unknown class id lists the declared classes; a wire id that is not an edge lists the edges; declaring a class whose membership cycles names the offending back-edge and refuses that class (not the design). The set_guarantee_slo / set_lag_slo flow tools (already source→terminal keyed, tools.ts 360–417) gain the same optional class scope.

8Boundaries & non-goals

A class whose own path cycles stays refusedPer-class acyclicity is enforced on E_C (§4.2). A synchronous self-wait inside one class has no finite response — honestly rejected, as v2 §4.4 already rejects it. Classes legalise a cyclic drawing, never a cyclic commodity.
Class-count sanityCells and reservoirs grow linearly in |C| (§5.1, §6). A soft cap with a warning keeps the hot path fast and the canvas legible; beyond it the tool asks whether the design really has that many distinct commodities.
Priority / weighted contention is futureR1's shared-node split is proportional processor-sharing (§4.1). Weighted priority classes (a gold class pre-empting a bronze one) are a named extension — they change only the split weights, not the model shape.
Per-class service time is futureR1 keeps one service time per node, so the per-hop wait is class-blind (§4.2). Different work per class at a shared node (a heavier report query) is a refinement to the M/M/c station, deferred.
Synthesize / enumerate stay class-agnosticclingo enumerates topologies (structure); classes are a load overlay on a fixed topology. synthesize() evaluates each enumerated topology class-aware; enumerate itself is unchanged — it builds the graph, classes describe the traffic over it.
No engine domain leakageA class is opaque ids (a class id, node ids, edge ids) — the engine gains a per-class index on flow cells and a per-class walk; it still knows nothing of orders, reports, or AWS. The cloud meaning stays content.
