Load stages just landed. The owner's brief for the next round is not "more peak features" — it is that everything should hook into everything: „wszystko spinało się ze wszystkim… jak to wpływa na promises na nodach? jak zobaczę które nody nie wyrobiły przez peaki?". This document scans every shipped feature, grounds each connection in code, and proposes the ONE elegant form the integration should take — consolidate, don't accrete. About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool that computes an architecture drawn on a canvas — its throughput, latency, availability and cost — against declared requirements, live on every edit, and can run backwards to size or repair a design. A domain-agnostic engine does the math (a fixpoint solver for the forward pass, a discrete-event simulator for tails, a native search for optimization); all cloud meaning is content. Load stages (the feature just shipped, load-stages.html) let a traffic origin declare that its load changes over time — a launch spike, a diurnal rhythm, a quarterly season — as a k6/Gatling-style table of steps wrapped in periodic cycles. This document plans how that time-varying load should tie into the features it currently sits beside rather than inside. Definitions, stated positively:

baseline levelthe generator's steady rate — the ×1 the shape's multipliers are relative to. It is the same reconciled assumedRps cell that worlds and Monte-Carlo already manipulate (load-stages.ts:112 generatorRate). A design with no cycles evaluates byte-for-byte as today.
λ(t) / the shapethe instantaneous rate level · Πcycles multiplier(t) — the exact scalar the sweep reads per window (load-stages.ts:112). Mean and peak are DERIVED from it: derivedMean bills, derivedPeak judges (load-stages.ts:180, :194).
Tier-1 sweepthe cheap analytic core: partition the auto-derived season into windows, and at each window override every origin's assumedRps to its λ(t) and evaluate the steady M/M/c response (time-sweep.ts:144, using applyScenarioToGraph). It yields a TimeSweep: the ρ-envelope, the worst window, the cost integral (mean bill) and the fraction of the span in violation (time-sweep.ts:47–62).
rhoByNode / worst windowper window, ρ = offered/capacity for every station (time-sweep.ts:37); worstWindowIndex is where cycles superimpose worst. peakLoadByNode(sweep) projects each node's own worst ρ and WHEN it peaks — a NodePeak {rho, atS} (time-sweep.ts:207, :192). This is the per-node peak read.
Tier-2 transientthe expensive proof: on the one worst window, a targeted DES measures the true backlog build-up/drain and returns a StressVerdict — survives?, recovery time, peak backlog, windowed p99, lost requests, retry amplification (two-tier.ts:64–82), basis measured (transient).
R4 (the foundation) in progressa concurrent agent is wiring the per-node peak into the presenter: the canvas ρ chip, a "Load" verdict row in the Inspector, and the System ρ rows now read peakReadOf(steadyRho, NodePeak) and append "· peak 141% ⚠ @18:00" (peak-view.ts:56, :70; node-detail.ts:168). This document treats R4 as done and plans everything downstream of it.

1The one idea — a peak is a derived world

Every integration below is one question asked of one feature: does it read the steady graph, or the peak-overridden graph?

The load-stages feature already made the load-bearing choice for us. To evaluate a shaped design, the Tier-1 sweep does not invent a new evaluation path — at each window it takes the origin's instantaneous rate and overrides node|assumedRps to that scalar, then calls the ordinary forward evaluator (time-sweep.ts:141–147). That override is byte-identical to what a named world does: applyScenarioToGraph substitutes a fact-assumption input cell (scenario.ts:154), and Monte-Carlo does the same per draw (uncertainty.ts:363). The worst window is therefore nothing more exotic than a scenario whose demand vector is the peak λ(t).

The consolidation.

SDA already has three consumers of "overlay a demand and re-judge": worlds (evaluateWorlds, scenario.ts:215), Monte-Carlo (runUncertainty, uncertainty.ts:350) and robust improve (robustOptimize, robust.ts:154). Each already accepts an AssumptionScenario. The elegant integration for peaks is to express the worst window as one derived scenario — the "peak world" — and let it flow through the seams that already exist, exactly as the auto-trio (derived-scenarios.ts:95) flows through them. No new evaluation engine; a new demand source. That reframes the whole matrix: most rows are "feed the peak scenario into a seam that already takes a scenario."

The counter-current, and the honesty this document must keep: a peak is not always a fact-assumption override. Two connections resist the "peak = world" reduction and need real design — (a) the cost integral is a whole-season mean, not a single overlaid point; (b) Tier-2's survival is a measured dynamic, not a re-evaluation. Those are flagged as they arise.

3The foundation this rests on — R4, and the gap it leaves

R4 is doing exactly the right first thing: it turns the sweep's peakLoadByNode into a per-node display so the canvas, Inspector and System panel answer the owner's „które nody nie wyrobiły przez peaki" directly — a node calm at the mean but saturated at 18:00 now reads "ρ 141% · saturated · at peak (18:00)" (peak-view.ts:77; node-detail.ts:168–170). That is the visible half.

The gap R4 does not close (and this plan's spine).

R4 threads the peak into the presenter — the view-models the two shells render. It does not touch realAwareVerdicts (verdict.ts:77), the single canonical verdict list that MCP evaluate, the design doc (doc-model.ts:95), the worlds matrix (scenario.ts:272) and Monte-Carlo SLO confidence (uncertainty.ts:395) all consume. So today the human sees a peak violation on the canvas while the AI reading MCP evaluate, and the generated deliverable, still see the steady green. That violates the project's own "web is a dumb renderer / one truth" invariant (memory: web-is-a-dumb-renderer). Closing it — promoting the peak from a presenter read into the shared verdict list — is integration #1 and the enabler for half the matrix.

4Integration #1 — per-node promises inherit peak-awareness TOP 3

The elegant design

Give realAwareVerdicts an optional peak read and let it do, in the shared list, exactly what R4 does in the presenter. Today the function recomputes each scalar latency SLO against the node's real (steady) response and raises a saturation violation for every steady ρ≥1 tier (verdict.ts:112–128). The one-form extension: accept an optional Map<string, NodePeak> (the sweep's peakLoadByNode) and, for a node whose peak ρ≥1 while its steady ρ<1, emit a saturation violation carrying a basis: 'peak' tag and the clock instant — the identical judgement R4 already makes, now in the list every surface reads.

Because the sweep already overrides assumedRps to λ(t) and re-runs the ordinary evaluator (time-sweep.ts:144), the peak-overlaid verdicts are just realAwareVerdicts run on the worst-window graph. The cleanest form is: verdicts are a function of a demand overlay; the steady list is the base overlay, the peak list is the worst-window overlay, and a node's shown verdict is the worse of the two (a max over demand, exactly as R4's peakReadOf takes the peak only when it exceeds steady — peak-view.ts:60).

Feasibility (code-grounded)

Direct. realAwareVerdicts is pure over (graph, value, queues) (verdict.ts:77); the worst-window graph and its value/queues are already built inside the sweep loop (time-sweep.ts:145–150) — they are simply not returned per node today (only ρ is kept). Two honest sub-tasks: (1) have the sweep retain the worst window's verdicts (or re-derive them on demand from the retained overlay); (2) merge them into the canonical list with a peak basis. No new solver, no new addressing.

Cost & tension

R2. The tension is labelling: a verdict that fails only at 18:00 must never read like a permanent failure. The list must carry basis: 'peak' + the instant so a renderer/agent can say "holds at the mean, saturates at the daily peak (18:00)" — the same two-basis honesty load-stages already keeps between analytic and transient (two-tier.ts:60). Get this wrong and every shaped design screams red at the AI. This is the single most important honesty decision in the whole plan.

5Integration #2 — Improve sizes for the worst window TOP 3

The highest-value connection. "Give me the cheapest configuration that survives the daily peak" is the question a staff architect actually asks, and SDA is one derived scenario away from answering it.

The elegant design

Robust improve already sizes a design to hold every SLO across a set of worlds, by running the native search once per world and taking each provisioning knob's maximum (robust.ts:154, :107). A world is just a fact-assumption overlay. So: derive a "peak world" whose demand overrides are each origin's derivedPeak (load-stages.ts:194) and hand it to robustOptimize/robustRepair as one more world. The result — "size db.concurrency to 40, bound by world peak" — falls out of the existing bindingWorld machinery (robust.ts:110), and enacts through the existing apply_solution path (search.ts:363). The surface is the worlds argument that already exists on repair/optimize (search.ts:266, :314) — add a reserved value like worlds:"peak" that resolveWorlds (search.ts:228) maps to the derived peak world.

Feasibility & the monotonicity question (the task's specific ask)

Does worst-window sizing stay in the native monotone class, or escalate to the MIP? It stays native. The native search is exact and fast precisely because the design is a monotone function of the provisioning knobs, and it decides feasibility at one box corner (ALGORITHMS.md → native/search.ts). The demand is a fixed input to each solve, not a knob — a peak demand is simply a larger constant, so a peak-world solve is no harder, and no less monotone, than a base-world solve. The escalation trigger is budget coupling (a cost ceiling fighting the objective — search.ts:291), which is orthogonal to whether the demand is peak or steady. So sizing for the peak does not by itself force the reference MIP.

Where non-monotonicity can bite is the robust combine, and robust already guards it: after the knob-wise max it re-verifies every world and DECLINES honestly (did-not-converge, naming the world) if the max-combine left the monotone class (robust.ts:124–128). The peak world inherits that guard for free.

Cost & the design decision

R2. One real choice: the single global worst window vs. each node's own worst instant. peakLoadByNode gives every node its own peak instant (time-sweep.ts:207) — a fan-out node may peak when a downstream node does not. Sizing to the single global worst window (one peak-demand vector) is cheaper and usually right; sizing each node to its personal worst instant is the true robust answer and is the per-window generalization of robust's knob-wise max (max over time windows instead of over worlds). Recommendation: ship the single-global-worst-window peak world first (one extra world, trivially correct), and treat per-window-max as a later exactness upgrade. Flagged as an open question (§16).

6Integration #3 — worlds gain a shape, and a peak column TOP 3

The elegant design — two moves

(a) A world overrides the SHAPE, not only the level. Today a world overrides fact-assumption scalars — including a source's demand (scenario.ts:80, :90). "Black Friday" is not a bigger baseline; it is a different shape — a taller, wider spike. The elegant form: extend ScenarioOverride so a world may also carry cycles for an origin (its own load-stages table), overlaid the same way the level is. A "Black Friday world" then IS the diurnal design with the spike preset swapped in, evaluated through the exact same overlay. This keeps the promise that a world is a concrete point in the assumption space — the point just now includes the shape.

(b) The worlds matrix gains a peak column. WorldSummary already carries peakRho — but it is the peak ρ across nodes at the single evaluated point (scenario.ts:291), not across time. When a world is shaped, each world's row should instead report its worst-window ρ and instant (run the Tier-1 sweep per world, read rhoEnvelope's max), and its cost integral beside its steady cost. The matrix becomes: per world, cost (mean bill) · worst-window ρ · when · survives?.

Feasibility

Move (a) is a schema-additive change to ScenarioOverride (scenario.ts:53) plus teaching applyScenarioToGraph to overlay a generate transform's cycles (it overlays fixed input cells today — scenario.ts:327; a transform lives on the port, so this is a genuinely new overlay target, honestly the harder half). Move (b) is pure composition: evaluateWorlds (scenario.ts:215) already iterates worlds; run timeSweep per world (it is sync-capable via the DI'd EvaluateGraph — time-sweep.ts:64) when that world is shaped and fold the peak read into summarizeWorld (scenario.ts:260).

Cost & tension

R1 for the peak column over already-shaped designs; R2 if move (a) (shape-per-world) is included. Tension: cost. Running a full Tier-1 sweep per world multiplies the ambient cost by the world count — acceptable for the ≤3 auto-trio, but the matrix should sweep lazily (only shaped worlds, coarse livePointsPerCycle — load-stages.ts:43) to stay ambient.

7Integration #4 — the capacity envelope, computed at peak

The elegant design

The envelope answers "how much sustained load can each origin carry with every SLO green?" by freeing the origin's demand and maximizing it subject to the SLOs (envelope.ts:246). For a shaped design the honest edge is not "how high can the baseline go" but "how high can the baseline go such that the PEAK still holds." Because peak = baseline × peakMultiplier (a constant for a fixed shape — load-stages.ts:194), the peak-aware maxRps is simply the steady maxRps divided by the shape's peak multiplier. That is a one-line post-scaling of the existing inversion — beautifully cheap — and the reported edge becomes "baseline ≤ 550 rps (peak ×1.8 → 990 rps at the SLO edge)."

Feasibility & tension

Two grades. The cheap grade (scale the existing edge by the peak multiplier) is exact when the binding SLO is capacity/ρ, which is monotone in demand — sound, R1. The exact grade (free the baseline but constrain the peak-overlaid graph inside the solver) needs the native model to evaluate at λpeak, i.e. the peak overlay must be the graph the inversion constrains — feasible (envelope.ts:311 builds the graph it inverts) but more invasive, R2. Tension: the queueing knee (envelope.ts:264) is where reality bites — at peak the knee arrives at a lower baseline, and reporting both ("knee at baseline 400 / capacity edge at 550") is the honest, and genuinely useful, output.

8Integration #5 — Monte-Carlo over baseline × shape

The elegant design

Monte-Carlo turns declared ranges into distributions (uncertainty.ts:350). The natural marriage with peaks: range the baseline LEVEL (the uncertain quantity) while the shape stays deterministic (the known pattern), and report a new confidence — "the daily peak holds in X% of sampled baselines." Each draw multiplies its sampled baseline by the fixed peak multiplier and judges the peak. This is the cleanest possible statement of the owner's uncertainty-meets-peaks question: not "will the mean hold" but "how sure are we the peak holds."

Feasibility

The seams are already there. Ranges live on assumedRps (the baseline). The batch evaluator overlays each draw (uncertainty.ts:366). Crucially, runUncertainty already accepts an optional scenario to center the sample on an active world before drawing (uncertainty.ts:332, :363) — feed it the peak world and every draw is judged at peak demand, and the existing SLO-confidence loop (uncertainty.ts:398–403) becomes peak confidence with no new code path. The only genuinely new piece is one metric spec — "peak ρ at the busiest tier" — added to metricSpecs (uncertainty.ts:428) so the tornado can attribute which ranged input drives peak strain.

Cost & tension

R2. Tension: v1 Monte-Carlo drives the forward pass only, and DES-only keys are honestly excluded (uncertainty.ts:342). "Does the peak survive (Tier-2)" is a dynamic, not a forward read, so MC can answer "does the peak ρ stay <1 in X% of baselines" (analytic) but must NOT claim to sample the transient survival — the same honesty line load-stages already draws between Tier-1 and Tier-2.

9Integration #6 — the system cost promise reads the day's integral

The elegant design

The system cost promise judges the whole-design monthly bill against a ceiling (system-promise.ts:111, :134). Today it reads systemSummary(...).totalCostUsdMonth — the steady snapshot. For a shaped design the honest bill is the cost integral: the mean whole-design cost over the season, which the Tier-1 sweep already computes as costIntegral (time-sweep.ts:183, "the mean bill"). The elegant integration: when a shape exists, the system cost promise reads the sweep's integral instead of the steady total. A 120-second spike inside a quarter barely moves the bill — and the promise should say so, honestly, rather than either over-charging at peak or ignoring the season.

Feasibility & tension

R1, and mostly a wiring choice. The integral is a scalar the sweep already returns; systemPromiseVerdicts takes a value lookup (system-promise.ts:111), so the caller passes the integral-derived total when shaped, the steady total otherwise. Tension — and it is a real one: this must stay consistent with the backward search's budget. The system promise and the search share ONE sum today (systemBandsOf → Objective.total, system-promise.ts:166; search.ts:287). If the verdict reads the integral but the optimizer still holds the steady total, they diverge — the anti-drift invariant breaks. So this connection is only honest if it lands together with a decision about which bill the search budgets to (recommend: the search budgets the steady/provisioned ceiling — capacity is reserved at peak — while the verdict may additionally show the pay-per-use integral; the two costs are already separate models in the doc, doc-model.ts:435).

10Integration #7 — tail & lag SLOs judged at the peak window

The elegant design

Tier-2 already runs a DES over the worst window and measures the windowed p99 during the peak (p99DuringMs — two-tier.ts:73, :305). A percentile SLO is set on a terminal and today judged by the steady-state simulate run (simulate.ts:59). The elegant unification: a tail SLO should be judged against the worst-window p99, not only the steady tail — the peak is exactly where a p99 deadline is at risk. The mechanism is already 90% built: Tier-2 produces the windowed p99; it simply is not yet routed into checkTailBands. The same holds for lag SLOs — a CDC/replication deadline is most likely to blow during the peak backlog, and the two-pass lag verdict (lag-slo.ts:94) can take the Tier-2-measured lag at the worst window as its measured basis.

Feasibility & tension

R2. Feasible: Tier-2's per-window metrics are already surfaced through simulate's loadStages.transient (simulate.ts:155–166). Tension: which run judges the tail? A design can have both a steady tail (the 50k-completion measured run, simulate.ts:53) and a worst-window tail (Tier-2). The honest answer is to report both with their bases — "steady p99 210ms ✓ · peak-window p99 640ms ✗ @18:00" — never collapsing them, matching the two-basis discipline. This is the tail-SLO twin of integration #1's labelling problem.

11Integration #8 — a "peak behaviour" section in the deliverable

The elegant design

The generated design doc is a pure function of the verified model (doc-model.ts:710). It has a closed, out-of-domain-free section canon (doc-model.ts:588). Peaks earn one new in-domain section — "Peak behaviour" — carrying the ρ-envelope series, the worst window (node + instant + ρ), which tiers saturate at peak, the cost integral, the %-of-span in violation, and the Tier-2 survival verdict (survives / recovers-in / peak backlog / lost requests). Every one of those is a scalar/series the sweep and two-tier already return; the doc-model's job is only to shape them into a section, its established discipline ("DATA in, never computed here" — doc-model.ts:44).

Feasibility & tension

R1. Feasible and idiomatic: it mirrors the optional guarantees/scenarios sections exactly — a new SectionKey, present ONLY when a shaped generator exists, spliced into sectionOrder like scenarios is (doc-model.ts:868, :595). Natural placement: right after §5 capacity (it IS the capacity story over time). Tension: the no-filler rule — a flat design must produce byte-identical docs (the golden-test contract, doc-model.ts:126), so the section is strictly gated on a non-empty sweep, exactly as guaranteeVerdicts gates §7b (doc-model.ts:838).

12Integration #9 — the reliability advisor flags a peak-only saturator

The elegant design

The reliability advisor turns computed series-product availability into an AWS nines tier and a sourced remedy (reliability.ts:97). It is currently blind to time — availability is a static product. The creative connection: a node that saturates only at peak is an availability risk the steady read cannot see — at ρ≥1 the queue is unbounded and requests time out, so effective availability at peak is below the nameplate SLA the advisor quotes. The elegant, honest form is a one-line advisory: when peakLoadByNode shows a hard dependency saturating at peak, the advisor notes "the computed availability assumes ρ<1; node X saturates at 18:00, so peak availability is lower — add capacity or shed load before adding redundancy." It reuses the advisor's existing "weakest dependency" wording (reliability.ts:102).

Feasibility & tension

R1, additive. Tension: keep it advisory, not a recomputed number — the advisor's credibility rests on sourced AWS figures (reliability.ts:12), and "peak availability = 1 − time-in-saturation" would be an invented number. The connection is a flag ("your nines assume no saturation; here is where that assumption breaks"), not a new availability computation. Lower value than #1–#3, but it closes a genuine blind spot for near-zero cost.

13Integration #10 — per-class shapes (forward-compatible only)

Request classes are frozen and partial: forward/scalar per-class ships, but per-class tails and backward search are honestly declined until per-class DES routing lands (FEATURES.md; simulate.ts:38). The peak connection is latent and clean: a Cycle rides a per-port generate transform, and class membership is per-wire, so nothing structurally prevents a report class carrying a quarterly shape while a user class carries a diurnal one — the sum-across-generators rule (load-stages.ts:24) already composes them. Recommendation: defer, but note the invariant now — when per-class DES arrives, the worst window becomes per-class (each commodity has its own λ(t)), and the sweep's origin detection (time-sweep.ts:84) generalizes to per-class origins. No work today; a one-paragraph forward-compat note in the request-classes doc so the seam is not accidentally closed.

16Open questions for the owner

1. Peak vs mean labelling — the honesty line. A design that saturates only at 18:00 is neither "green" nor "red." Confirm the desired wording contract: a distinct peak basis everywhere ("holds at mean, saturates at daily peak @18:00"), never a bare violation. This is the single most consequential decision (§4).

2. Single global worst window, or each node's own worst instant? Sizing/verdicts against one peak-demand vector is cheap and usually right; per-node worst-instant is the exact robust answer (§5). Ship the former first, upgrade later — agreed?

3. Which bill does Improve budget to? If the cost promise reads the day's integral but the search budgets the steady/provisioned ceiling, they must be reconciled to preserve the one-sum anti-drift invariant (§9). Recommendation: search budgets the reserved (provisioned) ceiling; the verdict additionally shows the pay-per-use integral. Confirm.

4. Does Improve size for the peak by default, or on opt-in? Default-to-peak is the safest design ("survive what you declared"); opt-in (worlds:"peak") is the least surprising for existing agents. Which?

5. May a world override the SHAPE (cycles), not only the level? The "Black Friday world" is the motivating case (§6). It is more invasive (a new overlay target — a port transform, not an input cell). Worth R2, or is level-only enough for v1?

6. Is the delivery-guarantee-at-peak connection in scope? It is real and grounded (lost requests at peak stress an at-least-once claim, §14) but subtle. Surface it, or leave delivery guarantees peak-agnostic for now?
