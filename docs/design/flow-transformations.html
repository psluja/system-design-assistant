Flow Transformations

Per-port traffic transfer functions — replacing the naive “every node relays 1:1” assumption with declared, solver-verifiable flow arithmetic.
SDA design document · 2026-07-02 · status: approved for build (round after universal origins)
About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas — services, databases, queues, connected by wires — and the tool computes that design instead of merely drawing it: it derives each component's load, latency, availability and cost, checks them against the architect's requirements, and can search for better configurations. This document is self-contained; earlier design documents are optional further reading, not prerequisites.

component / manifesta building block on the canvas (a database, a queue, a service). Its behavior is described purely by JSON data — numbers and rules — never by code.
port & wirecomponents connect through named ports (in/out); a wire is one connection between two ports. Traffic flows along wires.
forward evaluationthe instant calculation that runs on every change: given the declared traffic, compute each node's load, latency, availability, cost.
DESdiscrete-event simulation — a second engine that replays the system event by event over time; it produces true latency percentiles (p95/p99), which averages cannot give.
MiniZinc / MIPan exact optimization solver used when the tool runs 'backwards': find the cheapest configuration that still meets the requirements.
SLO & verdictan SLO is a requirement the architect declares (e.g. 'latency ≤ 250 ms'); a verdict is the tool's computed judgment of that requirement: ok, violation, or honestly 'unknown'.

1The bug being fixed

Today every outgoing edge of a node carries the same rate the node received (capped by capacity). Real components transform traffic: one request can spawn a hundred log lines, a thousand events can collapse into one aggregate.

The same gap hides aggregators (100 : 1), cache-miss pass-through (1 : 0.2 toward the DB), event fan-out (1 : N), sampling, and dead-letter splits. Prior art exists in performance-engineering research — Layered Queueing Networks (visit ratios), Palladio, JMT — but no interactive design canvas computes this live. It is a genuine differentiator.

2The design in one sentence

Any port — IN or OUT — may declare a transfer function from a closed, solver-verified set. A node becomes: arrivals → fin → service (capacity/queue) → fout per port → edges. Default everywhere: ratio(1) — today’s behavior, bit for bit.

Ownership decides the side (owner’s correction, 2026-07-02): emission belongs to the sender (“I emit 100 log lines per request” — ratio(100) on the generator’s OUT port), consumption shape belongs to the receiver (“I aggregate whatever arrives, 100 : 1, from any sender” — batch(100) on the aggregator’s IN port; putting it on every sender would assign the property to the wrong component). Transforms are pure data on the manifest port (components stay JSON; the engine stays domain-agnostic — an opaque function tag + parameters, arithmetic only). A per-instance override knob follows the same pattern as every config value.

Deliberate boundary — queues and throttles. Transforms give the rate arithmetic of a queue (batch consumption, drain ratios) and cap(r) IS a steady-state throttle (the excess lands in the existing overflow — the 429 mechanism). But a queue is memory over time — backlog growth, waiting time, drops after the buffer fills — and a memoryless function must not pretend otherwise: dynamics stay in the DES (queueMode: drainRate/maxBacklog). Semantics preserved: a throttle rejects (overflow verdict), a queue defers (DES) — the component declares which it is.

3The function set — what fits without a revolution

The dividing line is the backward solver: optimize / repair / synthesize run on MiniZinc + COIN-BC, a linear (MIP) solver. Anything linear or piecewise-linear rides today’s machinery; genuine non-linearities would demand a different solver class — a revolution we refuse for v1.

Observation that keeps this cheap: every real case the owner named — 1:1 events, 1:100 logs, 100:1 aggregation — is linear. The “activation function” intuition is right about shape declared per port, but infrastructure traffic arithmetic is ratios and caps, not sigmoids. A closed set keeps the honesty contract testable: each function gets a JS↔MiniZinc differential test; an arbitrary expression could not be verified this way.

5What this unlocks beyond the bug

Mid-chain origination completes. The universal-origins round (shipped) recorded one honest limit: a node that both relays and self-originates cannot add its origin to the emitted flow, because per-node throughput doubles as capacity. Per-edge emission is exactly the seam that removes it — origin injection becomes an additive term of the same edge arithmetic.

Aggregation pipelines stop lying. Kafka consumers, metric pipelines, CDC batchers — the 100:1 stages finally show their true downstream pressure, and optimize sizes the tail tiers for the real rate.

6Honesty & compatibility contract

Defaultratio(1) everywhere — every existing design and file evaluates bit-for-bit identically (pinned by a golden test).
Declared, never guessedTransforms come from the manifest (sourced) or the architect’s explicit override — the engine never infers a ratio.
DifferentialEach function of the set ships with a JS↔MiniZinc differential test and a DES cross-check (scalar mean vs simulated mean).
Refused for v1Arbitrary expressions and non-linear shapes — would silently break exact optimization; revisit only with a named, sourced use case.
