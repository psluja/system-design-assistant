Scale-up Cold Start — the time a fleet takes to grow

The owner ratified modelling the time a scaling fleet takes to grow: scaleUpColdStart, ONE honest lag knob (seconds) on a scaling fleet — the whole delay from a unit being needed to it serving, as a single number, not four separate detect/decide/launch/warm stages (the uwspólnianie principle — consolidate what exists). maxUnits stays the scaling ceiling and requiredUnits stays the computed demand-driven size; scaleUpColdStart adds only the missing time dimension — how fast the fleet grows toward what the load needs. It lives where transients live: the Tier-2 DES, whose server count becomes time-varying c(t). This closes exactly the gap behaviors.ts:75–76 already flags as honestly uncovered.

About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool that computes an architecture drawn on a canvas — its throughput, latency, availability and cost — against declared requirements, live on every edit, and can run backwards to size or repair a design. A domain-agnostic engine does the math (a fixpoint solver for the forward pass, a discrete-event simulator for tails and transients, a native search for optimization); all cloud meaning is content. An auto-scaling fleet (Fargate, Cloud Run, a Kubernetes HPA, an EC2 ASG) is modelled today as a pool that is always already the right size for the load it faces — capacity appears the instant demand does. That is a convenient fiction: real fleets take time to add units, and while they lag, work piles up. This document designs the honest version of that time, as the owner ratified it. Definitions, stated positively:

fleet / unitan auto-scaling compute pool (a role shared across Fargate, Cloud Run, k8s HPA, EC2 ASG) and one member of it (a task / instance / pod). Its capacity is maxUnits × per-unit, where per-unit throughput is concurrency / (perRequestDuration/1000) (behaviors.ts:294–296).
maxUnits unchangedthe scaling ceiling — the most units the fleet may ever run (an ASG max size, an HPA maxReplicas, an account task quota). A design commitment, a config knob (registry.ts:70; declared per fleet at common.ts:217,234,251,268). It sets how big the fleet can get, not how fast.
requiredUnits unchangedthe computed demand-driven size: inflow / per-unit — how many units this instant's offered load needs (behaviors.ts:297). Derived, never authored; billing already reads it (cost = costPer(requiredUnits), behaviors.ts:299). It says how big the fleet should be now, not whether it has got there yet.
scaleUpColdStart this documentONE lag, in seconds: the time from a unit becoming needed (requiredUnits rises) to that unit serving requests. It rolls detection + decision + launch + warm-up into a single honest number — the whole delay, not its parts. A fleet config knob, a sibling of maxUnits/concurrency on the same manifests (§2). 0 = today's instant fiction, bit-for-bit.
c(t) — time-varying servers this documentthe DES station's server count made a function of time: it tracks requiredUnits(t) with the scaleUpColdStart lag, capped at maxUnits. Today it is a single fixed number (network.ts:24–30) — pinned at the ceiling for the whole run. This is the one real engine change (§3).
Tier-1 steady sweep / Tier-2 transientthe two-tier load-stages evaluation (load-stages.html §10): Tier 1 is the cheap quasi-static analytic sweep over the season (time-sweep.ts); Tier 2 is the expensive DES that proves the one worst window over time (two-tier.ts, engine/sim/transient.ts). Cold start is a transient, so it lives in Tier 2 (§4).
per-request cold start distinct, existsa DIFFERENT thing already shipped: the DES coldStart distribution, a bimodal per-request service penalty (one request hits a cold container → slow) (coldstart.test.ts:8–9; doc-8 §Cold starts). scaleUpColdStart is per-fleet (not enough units yet → requests WAIT). Orthogonal; reused, not duplicated (§5).

1The workflow this serves — a fleet takes time to grow, and today the tool pretends it does not

A staff architect sizes an auto-scaler for two things at once: the height of the peak (can the fleet get big enough?) and the speed of the rise (can it get big enough in time?). SDA answers the first today and is silent on the second — it models every fleet as an autoscaler with infinite reaction speed. That silence is a lie of omission, and it is the exact gap the engine already admits to.

The tool's own comment says it plainly. Where the Lambda account-concurrency ceiling is modelled, the code stops and records what it will not model: "BURST behaviour (a +500-per-10s / +5,000-rps-per-10s scale-up ramp) is a TRANSIENT, not a steady-state ceiling, so it is NOT modelled here — recorded in doc-8 as honestly uncovered (the numeric hot path is steady-state; a ramp is a DES/time question)" (behaviors.ts:75–76). The coverage map carries the same open item: "Transients (warm-up, ramp) → DES → warm-up exists; richer transient scenarios open" (doc-8). This feature closes precisely that line — the ramp is the DES/time question, and the DES is where it belongs.

The stakes are concrete. A checkout fleet with maxUnits: 100 can serve a Black-Friday peak of 4,000 rps — Tier-1 confirms the ceiling clears the load, ρ < 1, green. But if the flash sale drives demand from 400 to 4,000 rps in ninety seconds and the fleet needs three minutes to launch and warm the tasks it now needs, the queue floods for those first minutes: timeouts, drops, a retry storm — a failure that survives at steady state, survives the peak if instant, and dies only on the ramp. Today no SDA surface can see it. After this feature, the Tier-2 transient does, and says so per node.

The owner's framing is a discipline as much as a feature: model the whole delay as one honest number. A real autoscaler's lag is a chain — metrics scrape interval, alarm evaluation, a scaling decision and cooldown, instance provisioning, container pull, application warm-up. Splitting that into four authored knobs would be false precision (nobody knows their alarm-evaluation window to the second) and four ways to disagree. One number — "from needed to serving takes ~N seconds" — is what an architect can actually estimate, defend, and vary. That is the uwspólnianie stance this document carries throughout: consolidate the delay into one knob, reuse the transient machinery whole, change the engine in exactly one place.

2The knob the owner ratified — one lag on the fleet, beside maxUnits

scaleUpColdStart is a fleet property, not a demand property. Demand shape (cycles) lives on the traffic generator; how fast the serving side grows to meet it lives on the fleet — right where maxUnits and concurrency already live, read by the same sizing relations.

2.1 Where it is declared — mirroring maxUnits / concurrency (confirmed)

The four demand-driven fleets each declare their scaling knobs as plain config and share one sizing-relation set: compute.fargate (common.ts:207, maxUnits: 100 at :217), compute.cloudrun (:234), compute.k8s (:251), compute.asg (:268) — every one closing with relations: sizingRels() (behaviors.ts:295–300). scaleUpColdStart joins as a fifth sibling config knob on those same manifests, exactly as maxUnits is a config knob: a new registry key beside maxUnits (registry.ts:70) with a node-local, no-flow aggregate (a fleet's reaction time is nobody else's — like maxUnits at registry.ts:168), and a small helper on the sizing family so the four fleets declare it uniformly rather than each hand-copying a literal.

// content/sda/src/registry.ts — a new key beside maxUnits (registry.ts:70), node-local like it:
scaleUpColdStart: Key('scaleUpColdStart'), // seconds from a unit being NEEDED to it SERVING (0 = instant, today)

// content/sda/src/common.ts — a sibling config on each demand-driven fleet, beside maxUnits:
{ key: k.maxUnits,          value: 100, unit: '1' },  // the CEILING (unchanged)
{ key: k.scaleUpColdStart,  value: 0,   unit: 's', est: true }, // NEW: the fleet's growth lag; est. per platform

The default is 0: a fleet with no declared lag is byte-for-byte today's instant autoscaler — the sacred-identity discipline load-stages already holds (a flat generator ≡ steady generate(level)). Because the knob is a fleet fact, not part of the generate Transform (whose payload is level + cycles + disable, graph.ts:80–84), the demand model and the growth model stay cleanly separated — one authored on the source, one on the fleet.

The knob in one sentence.

scaleUpColdStart adds a time axis to a fleet that already has a size axis: maxUnits caps how big, requiredUnits says how big it should be now, and scaleUpColdStart says how long it takes to get there. Two of the three already exist; this feature supplies only the third, and only inside the one engine that reasons over time.

3The one real engine change — a time-varying server count c(t)

Everything else in this design is content and reuse. There is exactly one change to the domain-agnostic engine, and honesty demands it be named sharply: the DES uses a fixed server count for a whole run; scaleUpColdStart makes that count a function of time.

3.1 Today: the server count is a constant, pinned at the ceiling

A DES station declares servers: number — a single fixed integer (network.ts:24–30). The content projector sets it to the fleet's full size: serverCount(node) = concurrency × fleetOf(node), and fleetOf = replicas ?? maxUnits ?? 1 (graph-read.ts:20,25–28) — i.e. concurrency × maxUnits, the fleet at its ceiling, for the entire simulation (sim.ts:253,263). The simulator reads that constant in the two places server count matters: admission to service, if (s.busy < s.def.servers) (des.ts:501), and utilisation, s.areaBusy / (s.def.servers · T) (des.ts:797). The analytic twin reads the same serverCount (queueing.ts:22, c = concurrency servers, queueing.ts:5–8) so the two agree under the differential test — a shared definition that must not silently break.

So today's DES models an autoscaler that is always at maxUnits: infinitely many warm servers standing ready from t = 0. Under a rising load that never queues for lack of capacity — the fleet was already as big as it could ever be.

3.2 The asymmetry this fixes — demand already varies in time; serving does not

The revealing thing is that the DES is already half time-varying. The arrival side carries a rateProfile — a periodic λ(t) multiplier — and the simulator generates a non-homogeneous Poisson process by exact inversion, one uniform per arrival (network.ts:68–77; des.ts:628–631). Load-stages' cycles lower to exactly this profile. So demand rises and falls over time already; the serving capacity is frozen at the ceiling. scaleUpColdStart restores the symmetry: the serving side gets its own time dependence, and the two finally race each other honestly — which is the whole physical content of a scale-up.

3.3 The change — c(t) chases requiredUnits(t) with the lag, capped at maxUnits

Give a station an optional server schedule instead of only a constant. When present, the count admitted to service at time t is

servers(t) = min( maxUnits,  ceil( requiredUnits( t − scaleUpColdStart ) ) ) × concurrency

i.e. the fleet at time t serves the size the load needed scaleUpColdStart seconds ago — a pure dead-time reading of "the time from needed to serving", capped by the ceiling. requiredUnits(t) is not new math: Tier-1's sweep already overrides each origin's assumedRps to its instantaneous λ(t) and re-evaluates the forward pass, which recomputes requiredUnits = inflow/per-unit per window (time-sweep.ts:156–164; behaviors.ts:297). So the c(t) target is the Tier-1 requiredUnits envelope, shifted right by the lag — the two tiers read one demand model, never two. Only the admission check (des.ts:501) and the utilisation denominator (des.ts:797) need to consult servers(t) instead of the constant; when no schedule is present the field is absent and every read is bit-for-bit today's.

Honest caveat — we model ONE lag, not a vendor's exact autoscaler.

A dead-time chase of requiredUnits(t−τ) is a faithful model of the one thing every autoscaler shares — capacity lags demand — and nothing more. It is not a target-tracking vs step-scaling controller, does not encode cooldown windows, does not reproduce a specific vendor's step size (Lambda's burst quota, an ASG's per-adjustment increment) or a warm-pool prefill. Those are exactly the four stages the owner ruled out consolidating into false precision. The claim the tool makes is honest and bounded: "with a growth lag of ~τ seconds, here is the backlog and the tail while the fleet catches up." Where a design needs vendor-exact scaling dynamics, that is out of scope and should read unknown, never a fabricated controller — the same line the tool holds everywhere.

4Where it lives — Tier-1 is untouched; Tier-2 is the whole home

Cold start is a transient by definition, so it appears in exactly the tier that reasons over transients — and it slots into machinery that already measures the very thing it produces.

4.1 The steady sweep is correctly blind to it

Tier-1 is the quasi-static analytic sweep: at each window it reads the equilibrium M/M/c response for that window's demand (time-sweep.ts:150–193, basis analytic (quasi-static) at :201). At equilibrium the fleet is the size the load needs — that is what equilibrium means — so a growth lag has, correctly, no effect there. Cold start is the departure from quasi-static; a sweep that assumes quasi-static must not see it, and the owner's ruling that Tier-1 stays unaffected is exactly right. Tier-1 continues to answer "can the ceiling serve the peak?"; it was never the tier that answers "in time?".

4.2 The transient DES already measures backlog build and drain

Tier-2 runs the untouched transient DES over the worst window and reads a survival verdict (two-tier.ts:201,338). That runner already produces precisely the readings a lagging fleet generates: a per-station backlog time series with its peak and when it happened (StationBacklog, transient.ts:84–92), a per-window arrivals/served/failed/p99 bucket (TransientWindow, transient.ts:61–82), a drain time from the disturbance's end back to baseline (drainTimeS, transient.ts:190–213), and a divergence test for "still growing at the end" (two-tier.ts:270–284). Feed that runner a station whose c(t) lags, and the backlog it builds while capacity is short is measured by the machinery as-is. No new metric, no new observation path: the feature supplies a time-varying server count and the existing transient reads the consequence.

The wiring is small and local. The Tier-2 job builder (tier2Job, two-tier.ts:201–253) already constructs the worst-window network and lowers each shaped origin's λ(t) to a windowed rateProfile. It gains one step: for a fleet station with scaleUpColdStart > 0, attach the c(t) schedule sampled from Tier-1's requiredUnits envelope over the same neighbourhood, shifted by the lag and clamped to maxUnits. The runner (runTier2, two-tier.ts:338) then plays demand-up against capacity-lagging, and transientVerdict (two-tier.ts:287–331) reads the same survival story it reads today — now including the failures a slow ramp causes.

6The integration matrix — a fleet property the transient respects EVERYWHERE, not a peak add-on

The owner's hard requirement: scaleUpColdStart must fit the WHOLE product. It is a property of a fleet, so it bites wherever demand rises faster than the fleet can grow — a world switch, a capacity edge, a robust sizing, any spike, any per-node verdict — not only in the headline spike-vs-scale-up race. 6.1 Worlds — does the fleet catch the step?

A world overrides fact-assumptions, demand among them, and the sweep/verdict paths already re-evaluate the overlaid graph (scenario.ts:153, the same override Tier-1 uses at time-sweep.ts:159–160). Today "switch to the peak world" is judged quasi-statically: evaluate the graph at world B's demand, done. With scaleUpColdStart, the honest question about a transition — from an operating world to a surge world — is transient: the fleet begins at A's size and must reach B's requiredUnits across the lag. That transition is expressible in the existing grammar (a two-stage step cycle), so a "world-switch transient" is a Tier-2 run over an A→B step, needing no new overlay target. Value is high because world switches are exactly where a real system meets a step it did not ramp into gently.

6.2 Capacity envelope — the edge you can reach in time

The envelope frees an origin's demand and maximises it subject to the SLOs, reporting the capacity edge and the queueing knee (envelope.ts:5–6,52–54). That answers "how high can sustained load go". scaleUpColdStart adds an orthogonal axis: "how fast can load rise and still be served" — because a step to the very edge is not served instantly by a lagging fleet. The clean form is a companion figure beside the maxRps edge: the fastest rise (rps per second) the fleet can track without the Tier-2 backlog breaching the SLO, computed by sweeping the ramp slope against the lag. Medium value — a real but secondary reading next to the level edge.

6.3 Robust Improve — survive the height AND the slope

Robust sizing solves each world and takes the knob-wise max, sound because provisioning knobs (incl. maxUnits) are monotone (robust.ts:46–49). Sizing maxUnits for the peak height stays in that monotone class — a peak is just a larger fixed demand. But surviving the ramp is a transient feasibility question the forward pass cannot see: the remedy is often not a higher ceiling but a higher floor (a minimum warm-unit count so the fleet starts nearer the peak) or a smaller lag (a warm pool). Recommendation: v1 sizes maxUnits for the peak height via the existing robust path, then runs a Tier-2 ramp-catch check on the sized design and, if it still floods, reports the honest remedy ("survives the peak height; the ramp is too slow — raise the warm floor or reduce the cold-start lag") rather than silently over-sizing the ceiling. High value: "cheapest fleet that survives the spike AND scales fast enough" is the question a staff architect actually asks.

6.4 Per-node verdicts — the failure only the transient can see

This is the sharpest payoff. realAwareVerdicts judges each node against its steady ρ and its worst-window ρ, folding a peak in as the same saturation (verdict.ts:78–95). A scale-up failure is invisible to both: at steady state ρ<1 (green), and at the peak instant — evaluated as if the fleet were already that big — ρ<1 too (green). The failure exists only in the time between: the fleet had not grown yet. That is not a ρ≥1 read at any instant; it is a backlog that builds and (maybe) drains. So it must ride the Tier-2 StressVerdict (two-tier.ts:65) — the measured (transient) basis — never collapsed into the quasi-static saturation list. The node's honest verdict becomes: "holds at the mean · holds at the peak if capacity were instant · fails during scale-up (backlog N at the worst window, drains in ~M s / does not drain)". This is the killer diagnosis the feature exists to produce.

7The three hardest tensions

7.1 Determinism of c(t) in the seeded DES

The DES's credibility rests on byte-for-byte reproducibility: a fixed seed fixes the output exactly, via a single RNG stream and a totally ordered event set (des.ts:13–15). A time-varying server count must not break that. The clean resolution is the open-loop dead-time model above: c(t) is a pure, deterministic function of time — Tier-1's requiredUnits envelope shifted by the lag — sampled like the arrival rateProfile already is, drawing nothing from the RNG. It stays byte-deterministic trivially, and reuses the exact seam the arrival side proved. The tempting alternative — a closed-loop autoscaler that reacts to the DES's own observed backlog/arrivals — is more realistic but introduces feedback (like retry storms) that can be non-monotone and is harder to size and reason about; it also remains deterministic (the reaction is a function of seed-determined history) but is genuinely harder. Recommendation: ship open-loop dead-time (deterministic, cheap, honest about capacity lag); treat closed-loop reactive scaling as a later, clearly-flagged upgrade. The tension is real because open-loop assumes the autoscaler knows what was needed τ ago — a slight optimism the caveat in §3.3 must state.

7.2 Is scale-DOWN modelled?

The owner named scaleUpColdStart — the up direction, which is the risk axis (too-slow growth ⇒ backlog ⇒ failures). Scale-down is a different lag and rarely a failure: removing units when demand falls does not build backlog, it only briefly over-provisions (a small cost, not an outage). v1 should model scale-up lag only and treat scale-down as instant — the conservative choice, because it makes each repeating peak start from a low fleet and pay the full cold start again (the worst case). But there is a genuine subtlety for multi-cycle demand: if a diurnal fleet scales down fully in the overnight trough, tomorrow's morning ramp pays the cold start afresh; if it retained a warm floor, it would not. That is precisely why the robust-improve remedy (§6.3) is a warm floor, and why scale-down speed / a min-warm-units knob is the natural v2. Decision: instant scale-down in v1, explicit; a scale-down lag or warm floor deferred, flagged (§9).

7.3 Per-unit cold start vs fleet-lag granularity

Two physically different "cold starts" now coexist, and conflating them would be a lie. The existing DES coldStart distribution is per-request: enough units exist, but a given request lands on a just-launched container and pays a slow service time — a tail effect (coldstart.test.ts:8–9). scaleUpColdStart is per-fleet: there are not enough units yet, so requests wait — a backlog effect. They are orthogonal and both real; they even compose (a unit the fleet just added serves its own first requests cold). v1 models the fleet as a single homogeneous c(t) count of warm servers (the dominant effect: queueing when under-provisioned) and leaves the per-request penalty to the existing distribution, unchanged. Modelling each newly-added unit as cold-for-its-first-N-requests — coupling the two — is a faithful refinement but a real increase in state (per-unit warmth, not just a count); deferred, and named so the seam is not accidentally closed.

8Ranked build rounds

Ordered so the smallest honest slice ships first and each round leans on the last. The engine change and the knob are the spine; everything downstream reads them.

  1. The knob + the engine change. R1 Add the scaleUpColdStart registry key and fleet config (default 0), and give the DES Station an optional deterministic server schedule read by the two server-count touchpoints (des.ts:501,797). Gate it: absent ⇒ bit-for-bit today, pinned by a sacred-identity test. Differential-check that a constant schedule equals the fixed count exactly. Nothing user-visible yet.
  2. Tier-2 plays c(t). R1 In tier2Job (two-tier.ts:201), attach the lagged requiredUnits(t) schedule to each shaped fleet station over the worst-window neighbourhood; runTier2 and transientVerdict read the resulting backlog unchanged. This alone closes the doc-8 gap end-to-end.
  3. Per-node scale-up verdict. R2 Surface the "fails only during scale-up" verdict on the Tier-2 StressVerdict basis (§6.4), dual-basis honest ("holds steady · holds at peak-if-instant · fails on the ramp"), and route it to the per-node surfaces the peak read already uses. The highest-value diagnosis.
  4. Robust Improve ramp-catch. R2 After sizing maxUnits for the peak height (existing robust path), run a Tier-2 ramp-catch check and report the honest remedy (warm floor / smaller lag) when the height is sufficient but the slope is not (§6.3).
  5. Worlds transition + envelope rise. R2 The A→B world-switch transient (§6.1) and the "fastest sustainable rise" companion to the capacity edge (§6.2) — both compositions over existing sweeps, no new solver.
  6. Deferred refinements. defer Closed-loop reactive scaling (§7.1), a scale-down lag / warm-floor knob (§7.2), and per-unit-warmth coupling with the existing coldStart distribution (§7.3) — each a faithful upgrade, each named so its seam stays open.

9Open questions for the owner

1. Dead-time or first-order lag? The design proposes c(t) = requiredUnits(t − scaleUpColdStart) — a pure dead-time, which maps exactly to your words "the time from needed to serving". A first-order lag (du/dt = (target − u)/τ) is smoother but its τ is not literally the delay. Confirm dead-time as the honest primary (§3.3, §7.1).

2. Open-loop first, closed-loop later? Open-loop (a deterministic function of time, reusing the rateProfile seam) ships determinism for free; closed-loop (react to the DES's measured backlog) is more realistic but feedback-coupled. Agreed to ship open-loop and flag closed-loop as an upgrade (§7.1)?

3. Scale-down: instant in v1? Modelling only scale-up lag, with instant scale-down (the conservative worst case, each peak pays full cold start), and deferring a scale-down lag / warm-floor knob to v2 (§7.2). Confirm.

4. Fleet-count granularity, not per-unit warmth? v1 models a homogeneous c(t) count and leaves the per-request coldStart distribution untouched, rather than tracking each new unit as cold-for-its-first-N-requests (§7.3). Enough for v1?

5. Does the bill follow the lagging size, or the required size? During the ramp the fleet is briefly smaller than requiredUnits, so the honest instantaneous cost is marginally lower. v1 keeps cost = costPer(requiredUnits) unchanged (§2.2); an integral-of-c(t) bill is a possible refinement. Leave cost as-is for now?

6. Where is the knob's default value sourced? The lag is genuinely platform-dependent (a warm Lambda vs a cold EC2 ASG differ by orders of magnitude). Default 0 (instant, today) is the safe non-lying default; should the four fleets ship an est. per-platform starting value (badged est., architect-tunable), or stay 0 until the architect opts in?
