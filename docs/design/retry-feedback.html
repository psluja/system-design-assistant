Retry Feedback & Goodput Collapse

The last big fidelity gap: past saturation, real systems don’t plateau — they collapse. Retries as declared data, simulated honestly, with Erlang-A as the theory anchor.

About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas — services, databases, queues, connected by wires — and the tool computes that design instead of merely drawing it: it derives each component's load, latency, availability and cost, checks them against the architect's requirements, and can search for better configurations. This document is self-contained; earlier design documents are optional further reading, not prerequisites.

saturation / ρ (rho)the ratio of offered load to a component's capacity; ρ ≥ 1 means more work arrives than the component can serve.
goodputthe rate of REQUESTS THAT ACTUALLY SUCCEED, as opposed to raw attempts; the difference is retries and failures.
DESdiscrete-event simulation — SDA's engine for questions about time: it replays the system event by event, so queues, timeouts and retries emerge naturally.
scalar (forward) passthe instant steady-state calculation that runs on every canvas change; it is deliberately simple and honest — where dynamics matter it says 'unknown' and defers to the simulation.
Erlang A / Erlang Cclassical queueing-theory formulas: Erlang C gives waiting times for a multi-server queue; Erlang A extends it with customers who abandon the queue — the theoretical anchor for timeouts.
SLO & verdicta declared requirement (e.g. 'throughput ≥ 5,000/s') and the tool's computed judgment of it: ok, violation, or honestly 'unknown'.

1The gap (named by the holistic validation)

Our load-sweep matched Erlang C to fractions of a millisecond — up to saturation. Past it, SDA shows served throughput flat at capacity. Real systems show a hump: clients time out, retry, and the retries are new load attacking an already-saturated tier. Useful work (goodput) falls below capacity while total attempted load explodes. This is the mechanism behind most cascading outages — and it is exactly what doc-8 records as our known idealization.

2The model — retries as caller-side DATA

A retry policy is a fact of the CALLER’s code, so it is declared on the calling node (client or service), as three plain config keys — components stay pure data:

KeyMeaningDefault
timeoutMsper-attempt deadline: the caller abandons an attempt whose queue-wait + service exceeds it0 = none (today’s behavior, bit-for-bit)
retryCountadditional attempts after the first (0 = fail immediately on timeout)0
retryBackoffMsdelay before re-injection (fixed; jitter is a DES seed detail, noted honestly)0
Where it executes: the DES, and only the DES. Retry feedback is non-monotone (load ↑ → timeouts ↑ → load ↑) — the definition of a “question about time” under doc-8 §0. The scalar pass stays out: with a retry policy present, scalar goodput reads unknown and points at simulate — the same honest pattern percentile SLOs already use. No fixpoint hacks, no invented steady-state.

New simulation outputs: goodputRps, errorRate, amplification (attempts ÷ arrivals), all fed into verdicts the way DES already feeds tailLatency — a throughput SLO on a retrying path is judged against goodput, not raw completions.

4The theory anchor — Erlang A

M/M/c with exponential abandonment is the Erlang-A model — closed-form abandonment probabilities exist. That gives this feature the same rigor the queueing latency got (Erlang C matched to 0.1 ms): DES abandonment rates are differential-tested against Erlang-A at known (ρ, θ=1/timeout) points, and two property tests pin the semantics: timeoutMs=0 ∧ retryCount=0 ≡ today bit-for-bit, and the hump property — past saturation, goodput must be strictly LOWER with retries than without (the model may never claim retries help a saturated system).
