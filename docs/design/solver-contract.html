The Solver Contract

The heart of the project, made explicit: one small package that is the canonical documentation home, the abstract domain models, and the executable specification that every solver — the incumbent generic ones and our own domain solver to come — must pass, identically. Distilled from the code that exists, not invented.
SDA design document · 2026-07-03 · TASK-79 phase 0 · status: for owner review — not started
About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas and the tool computes the design — load, latency, cost, feasibility — against declared requirements. Underneath, SDA is a domain-agnostic engine that turns the drawing into a typed-property graph and answers questions about it with a handful of solvers. Today three of those solvers are heavy third-party WebAssembly components (a numeric optimizer, a logic enumerator); the long-term plan (TASK-79) replaces their runtime use with a purpose-built solver and keeps the generic ones forever as test-time referees. This document designs phase 0: the boundary — the contract — that makes such a swap a one-line change and provably behavior-preserving.

capabilityone question the engine can answer about a design: evaluate it, optimize a knob, repair it, explain why it can't be met, enumerate legal topologies, evaluate a batch. Each is its own small interface.
adapter / implementationa concrete thing that answers a capability. The incumbent adapter wraps MiniZinc / clingo / the JS hot path; the second adapter will be our own solver. Both satisfy the same interface.
conformance suitean executable test battery shipped with the contract that every adapter runs against — exactness, never-hang, honest non-convergence, determinism, cancellation. Passing it is what "implements the capability" means.
composition rootthe single place in each app that decides which adapter is bound to which capability. Everything else depends only on the interface (dependency inversion), never on a concrete solver.
referee modebinding both adapters at once behind a comparison wrapper that runs them side by side and flags any divergence as a top-priority bug — how we prove the new solver matches the old.
honesty stateSDA's rule that uncertainty is a value, never a guess: an answer may be a solution, or infeasible, or did-not-converge — and those three are never conflated.

1Why the contract is the heart

SDA's whole promise — "the tool must not lie" — rests on the solvers. The value of a verified design is exactly the trustworthiness of the thing that verified it. Today that trust lives implicitly, scattered across a facade, three injected function types, a CLI adapter, a browser twin, and the doc-4 charter prose. It works, but the idea is not in one place, and there is no single artifact that a second implementation could be built against and measured by.

The owner's framing (2026-07-03, "musimy to mieć") is a centralization of the idea into one separated package, @sda/solver-contract, that is three things at once:

The contract package is…What that means concretely
The canonical documentation homethe abstract domain models — request shapes, result shapes, the honesty states — carry their meaning as doc-comments on the interface itself. The contract IS the documentation: there is no truth about "what optimize returns when it can't converge" that lives anywhere but on the type. Design docs (this one, doc-4) reference the contract; they do not re-state it.
The abstract domain modelsone home for OptimizeRequest, OptimizeResult, Shortfall, SolveOutcome, EnumerateResult… — the vocabulary in which every solver question is posed and every answer is read, independent of MiniZinc syntax, clingo atoms, or cell-network internals.
The executable specificationa conformance suite ships in the package. "Adapter X implements capability C" is not a claim in a README — it is a green test run. The incumbent adapter is the first to pass it (pinning today's behavior); our solver is the second, built against it.
The one rule that keeps this honest: an interface exists only if two real implementations exercise it. Phase 0 does not invent abstractions for solvers we might write. Every capability in the contract is distilled from an implementation that ships today and is destined for a second implementation (our domain solver, or — for enumerate — a future in-house enumerator). No speculative generality; the abstraction earns its place by being needed twice.

3The contract — one interface per capability

Each capability below is a distilled interface with its request/result domain models and honesty states, written so the doc-comment IS the specification. Each is justified by naming its two implementations — the rule from §1. Signatures preserve today's shapes; the contract renames toward capability nouns and lifts the models out of minizinc/ into the shared package.

The central design tension, resolved once here: the sync hot path vs the async cold solvers. Evaluate is synchronous today (a <16 ms in-process fixpoint) and must stay so — the canvas re-evaluates on every edit. Optimize/Repair/Explain/Enumerate are asynchronous (they await a WASM worker). The contract does not force one shape on both. Each capability declares its own natural return type: Evaluate returns Result<…> directly; the search capabilities return Promise<Result<…>>. This is deliberate interface segregation — there is no god-interface a solver must implement wholesale. A solver adapter implements exactly the capabilities it provides, and a consumer depends only on the capabilities it calls. The future domain solver may answer Evaluate synchronously and Optimize asynchronously from one module — that is fine; it satisfies two separate interfaces.

3.1 Evaluate — the hot path sync

/** Forward evaluation: solve the design's typed-property graph to its least fixpoint and
 *  judge every declared band. SYNCHRONOUS and pure — this runs on every canvas edit, so it
 *  must not await. Uncertainty is a VALUE, never a throw: a build problem is a Result error;
 *  a failure to settle is `converged:false`; a band no value can decide is verdict `unknown`. */
export interface Evaluate {
  (req: EvaluateRequest): Result<Evaluation, readonly string[]>;
}
export interface EvaluateRequest { readonly graph: Graph; /* registry is bound at adapter construction */ }
export interface Evaluation {
  readonly converged: boolean;              // false ⇒ did-not-converge / NaN (honest)
  value(node: NodeId, key: Key): number | undefined;
  readonly verdicts: readonly Verdict[];  // each: ok|warning|violation|unknown|did-not-converge
}

Two implementations: (1) the cell-network least-fixpoint evaluate() shipping today; (2) the domain solver's CPU forward pass (TASK-79 layer 2, "CPU proves"). Verified against engine/solve/src/engine.ts and fixpoint/solve.ts (converged = settled && !hasNaN).

3.2 Optimize — search a knob against SLOs async

/** Run backwards: choose values for the freed `tunables` that make every SLO hold while
 *  minimizing/maximizing `objective`. ASYNCHRONOUS (awaits a worker). The result triad is
 *  load-bearing and MUST NOT be conflated: a proven-optimal `solved`, a proven `infeasible`
 *  (no assignment can satisfy the SLOs — use ExplainInfeasible for the shortfall), or an
 *  honest `did-not-converge` (hit the hard time bound — simplify or set knobs manually). */
export interface Optimize {
  (req: OptimizeRequest): Promise<SearchResult<OptimizeSolution>>;
}
export interface OptimizeRequest {
  readonly graph: Graph;
  readonly tunables: readonly Tunable[];        // {node,key,min,max} — a freed config input
  readonly objective: Objective;                 // {node,key,direction:'min'|'max'}
  readonly headroom?: Headroom;                  // {key,factor} — ρ ≤ factor, finite-latency sizing
}
export interface OptimizeSolution {
  readonly assignments: readonly { node: NodeId; key: Key; value: number }[];
  value(node: NodeId, key: Key): number | undefined;
}
// The honesty triad, shared by every search capability. `did-not-converge` ≠ `infeasible`.
export type SearchResult<T> =
  | { readonly kind: 'solved'; readonly value: T }
  | { readonly kind: 'infeasible' }
  | { readonly kind: 'did-not-converge' };

Two implementations: (1) the MiniZinc projection optimizeModel + COIN-BC/HiGHS; (2) the domain solver's CPU branch-and-bound over the same cell network. The three-kind result is distilled straight from SolveOutcome (minizinc/cli.ts) and the facade's two distinct error constants INFEASIBLE / DID_NOT_CONVERGE (facade.ts lines 13–14) — the contract promotes those constants into a typed kind so the distinction cannot be lost in a string.

A model-fidelity note the contract must carry, not hide: the facade prunes to reachableTunables and evaluates violated bands before calling the solver (facade.ts lines 74–82). That pruning is a performance optimization of the incumbent adapter, not part of the capability's meaning — so it stays inside the adapter, below the interface. The contract specifies only the observable behavior (best objective subject to SLOs); the conformance suite pins that, and the domain adapter is free to prune differently as long as it returns the same optimum.

3.3 Repair — the minimal edit async

/** Given a design that violates its SLOs, find the SMALLEST change to the tunables that
 *  makes every band hold (minimize Σ|new − current|). Answers "what is the least edit that
 *  fixes this", not "what is the cheapest design". Same async triad; a `solved` with an
 *  EMPTY change list means the design already holds — no edit needed. */
export interface Repair {
  (req: RepairRequest): Promise<SearchResult<readonly Change[]>>;
}
export interface Change { readonly node: NodeId; key: Key; from: number; to: number; delta: number; }

Two implementations: (1) repairModel (L1-minimizing MIP); (2) the domain solver's local-search repair. Verified against facade.ts lines 111–123 and minizinc/search.ts repairModel.

3.4 ExplainInfeasible — the exact shortfall async

/** Explain WHY the SLOs cannot be met: re-encode every band as SOFT and minimize total
 *  penalty. A positive penalty names exactly WHICH SLO fails and BY HOW MUCH — strictly
 *  more informative than a boolean UNSAT or an unsat-core. The relaxed model is always
 *  satisfiable, so the only non-`solved` outcome is `did-not-converge` (a timeout). */
export interface ExplainInfeasible {
  (req: ExplainRequest): Promise<SearchResult<readonly Shortfall[]>>;
}
export interface Shortfall {
  readonly node: NodeId; key: Key;
  readonly bound: 'floor' | 'ceiling' | 'point';
  readonly amount: number;  // how far the band is missed (0 ⇒ satisfied)
}

Two implementations: (1) relaxedModel soft-penalty MIP; (2) the domain solver's slack computation over the same bands. Verified against facade.ts lines 97–109 and relaxedModel (minizinc/search.ts lines 408–444).

3.5 Enumerate — generate legal topologies async

/** GENERATE candidate discrete structures (not check one): return every valid selection —
 *  exactly one option per slot such that all adjacent choices are compatible and placement
 *  rules (requires/conflicts) hold. Domain-agnostic: slots/options/compat are opaque ids.
 *  UNSAT ⇒ an EMPTY list (not an error). Deterministic order for reproducible callers. */
export interface Enumerate {
  (req: EnumerateRequest): Promise<EnumerateResult>;
}
export interface EnumerateRequest {
  readonly problem: SelectionProblem;   // slots · adjacencies · compatible · requires? · conflicts?
  readonly limit?: number;             // 0 / omitted ⇒ all answer sets
}
export type EnumerateResult =
  | { readonly kind: 'enumerated'; readonly selections: readonly Selection[] }
  | { readonly kind: 'did-not-converge' };  // NEW — a bound the raw runner lacks today (see §"open questions")

Two implementations: (1) enumerateSelections over clingo-wasm (RunAsp); (2) a future in-house series-parallel enumerator (TASK-79 exploited-structure), OR — nearer term — the second implementation is the node vs browser pair of RunAsp providers, which already exercise the same SelectionProblem → Selection[] contract through two independent WASM wirings (app/mcp/src/clingo-node.ts and app/web/src/clingo.ts). Verified against engine/solve/src/asp/clingo.ts.

Honesty gap surfaced by distillation: today answerSets throws on a clingo ERROR, and enumeration has no time bound (asp/clingo.ts line 28). This violates the "search never throws / never hangs" discipline that the MiniZinc path already honors (app/mcp/src/search.ts lines 24–30). The contract specifies the correct shape — a did-not-converge kind and no-throw — and the incumbent Enumerate adapter must close that gap to conform. This is exactly the value of writing the contract: it makes the one inconsistent seam visible and gives it a fix target with zero ambiguity.

3.6 EvaluateBatch — N evaluations, one call async · not yet built

/** Evaluate the SAME design under many input scenarios in one call. The batch seam is layer 1
 *  of the domain solver ("GPU proposes") and the engine behind Monte-Carlo uncertainty. SEEDED:
 *  byte-reproducible. A CPU backend (loop of Evaluate off-thread) is the first implementation;
 *  a WebGPU backend the second. Per-scenario states are exactly Evaluate's. */
export interface EvaluateBatch {
  (req: EvaluateBatchRequest): Promise<readonly Evaluation[]>;
}
export interface EvaluateBatchRequest {
  readonly graph: Graph;
  readonly scenarios: readonly Scenario[];  // each: drawn overrides for ranged config inputs
}

Two implementations: (1) the CPU worker loop over Evaluate (Monte-Carlo R1); (2) the WebGPU/WGSL batch proposer (TASK-79 layer 1). This interface is included in phase 0 only as a declared seam — it has no implementation to pin yet, so its conformance clauses are written but skip-marked until the first backend lands (honoring "no interface without two impls" by marking it not-yet-active rather than pretending). Distilled from docs/design/uncertainty-monte-carlo.html §2.

3.7 Simulate — the time engine sync · seeded contract-adjacent

The DES is a separate engine (doc-4 §3b): it computes dynamics and true percentiles, not a fixpoint over the property graph. It is not a solver of the cell network and TASK-79 does not replace it — the referee story (numeric differential vs MiniZinc) does not apply. Phase 0's decision: Simulate does not join the solver contract. Its already-clean seam is the TailProvider (verdict/tail.ts) that decouples engine/solve from engine/sim; that seam stays where it is. Including Simulate would be a god-interface reflex — the honest call is to list it, explain why it is out, and keep the contract to the cell-network solver family. (If a second DES implementation ever appears, it earns its own small contract then — not speculatively now.)

4The conformance suite — the executable specification

The suite ships in the package and every adapter runs it. It is parameterized by the adapter under test: conformanceOf(adapter) returns a describe-block the adapter's own test file invokes. "Implements capability C" means: this suite is green for C.

ClauseWhat it assertsExecutable test shape
Exactness On a corpus of designs with known answers, the adapter returns the same solution the reference does (the optimum, the shortfall, the selection set). This is the anti-lie clause. for (design of corpus) expect(adapter.optimize(design)).toEqual(reference[design]) — the reference being the pinned incumbent output (phase 0) and later the MiniZinc referee (phase 1's generated suite).
Hard time-bound (never hangs) A pathological input returns within the budget — as a best-so-far or an honest did-not-converge — but always returns. A tool that can hang is poison for an agent flow (memory: solver-time-bounded). const t=Date.now(); const r=await adapter.optimize(degenerate); expect(Date.now()-t).toBeLessThan(BUDGET+margin); expect(r.kind).toBeDefined(). Uses the known 319 s-degeneracy family as fixtures.
Honest non-convergence infeasible and did-not-converge are returned as distinct kinds and never swapped; a genuinely-infeasible design reads infeasible, a timed-out one reads did-not-converge. Two fixtures: a proven-UNSAT design ⇒ expect(r.kind).toBe('infeasible'); a within-budget-impossible design ⇒ expect(r.kind).toBe('did-not-converge'). Also: no capability ever throws on a no-solution.
Determinism under seed Same input + same seed ⇒ byte-identical output, run twice. Enumerate returns its canonical order; EvaluateBatch is reproducible. expect(await run(input,seed)).toEqual(await run(input,seed)) and, for enumerate, expect(selections).toEqual([...selections].sort(canonical)).
Cancellation A solve handed an already-aborted AbortSignal settles promptly without producing a stale result; an in-flight solve aborted mid-run rejects/returns cancelled rather than resolving late onto a superseded design (the epoch-guard discipline, web-workers-concurrency). const c=new AbortController(); c.abort(); const r=await adapter.optimize({...,signal:c.signal}); expect(r).toBe(CANCELLED) — cancellation is a new parameter the contract introduces (see §"open questions": today cancellation is staleness-guarded at the call site, not passed to the solver).
Why the suite lives WITH the contract, not with the engine: the suite is the operational definition of the interfaces' doc-comments. If a clause and a comment ever disagree, that is a contract bug — keeping them in one package makes that impossible to miss. The engine's own tests (property, differential, golden) stay where they are and test the incumbent implementation; the conformance suite tests the boundary, and is what the second implementation is graded against.

5The composition root — one binding site, referee mode

Every app has exactly one place that decides which adapter answers which capability. Nothing else names a concrete solver. Switching implementations — for one capability or all — is a single change there.

// app/mcp/src/composition.ts  (and the browser twin in app/web/src/composition.ts)
import type { SolverBindings } from '@sda/solver-contract';

export function bindSolvers(mode: RuntimeMode): SolverBindings {
  const incumbent = makeIncumbentAdapter({ solveMzn: nativeSolveMzn, runAsp: nativeRunAsp });
  switch (mode) {
    case 'incumbent': return incumbent;                     // today, and the runtime default
    case 'domain':    return makeDomainAdapter();                 // phase 3 — one line flips the runtime
    case 'referee':   return referee(incumbent, makeDomainAdapter()); // both bound; divergence ⇒ P0
  }
}

The referee wrapper implements every capability by calling both bound adapters, returning the trusted (incumbent) answer, and asserting equivalence:

const referee = (trusted: SolverBindings, candidate: SolverBindings): SolverBindings => ({
  optimize: async (req) => {
    const [a, b] = await Promise.all([trusted.optimize(req), candidate.optimize(req)]);
    if (!equivalent(a, b)) reportDivergence('optimize', req, a, b); // P0 — the tool would lie
    return a;  // the trusted answer is what ships; the candidate is only measured
  },
  /* …one wrapper per capability, each independently switchable… */
});

Transition period: both implementations coexist in the tree; only one is in the runtime per the bound mode. Referee mode is a CI/dev configuration, never the shipped default. equivalent() is capability-specific and tolerance-aware: exact for Enumerate's selection sets and Repair's changes; float-tolerant for Optimize's objective (two optima may differ within ε yet both be valid), with the same value read back through value(node,key) so the comparison is on observable outputs, not solver internals.

6Bundle separation — designed day one

The prize (TASK-79) is dropping the 17.3 MB MiniZinc wasm (plus its data ≈ 17.8 MB) from the runtime bundle once the domain solver ships — while keeping the incumbent adapter in CI forever as the referee. The separation must be structural from day one, or it becomes impossible to extract later.

MechanismHow our builds honor it
The referee adapter is a separate entry, dynamically imported The incumbent adapter (which pulls in the MiniZinc/clingo loaders) is reached only via await import('@sda/solver-contract/incumbent') from the composition root, never a static top-level import. The domain adapter is the static default. This is the pattern already proven in the tree: app/web/src/app.tsx lines 897–898 dynamically import('./mzn') / import('./clingo'), and the wasm is fetched only on first solve (app/web/src/mzn.ts apiP ??= import(...)). Bundle separation is not new machinery — it is making today's lazy-load the only path to the heavy adapter.
Vite (web) — the wasm is a public asset, the loader a lazy chunk The 17.3 MB app/web/public/minizinc/minizinc.wasm is a public/ asset resolved at runtime by URL (mzn.ts new URL('minizinc/minizinc.mjs', document.baseURI)), so it is never in the JS graph — it ships or not by whether public/minizinc is vendored. The loader module lands in its own dynamic chunk (Rollup code-splits on import()). Phase-3 removal = drop the vendor step + never bind the incumbent adapter at runtime; the referee CI build re-adds the vendor step behind a flag.
esbuild (VS Code MCP) — a build flag drops the entry The VS Code extension bundles with esbuild (app/vscode/esbuild.mjs). The incumbent adapter entry is excluded from the runtime bundle via a define/flag (--define:SDA_REFEREE=false) that dead-code-eliminates the dynamic import, or by marking @sda/solver-contract/incumbent external and shipping it only in the CI build. The wasm files (app/vscode/dist/*.wasm) are copied by the build script conditionally on the same flag.
CI keeps the referee forever The conformance + differential jobs build in referee mode (flag on), so the incumbent adapter and its wasm are present in CI regardless of what the shipped bundle contains. The generic solvers never leave the repo — only the runtime bundle.
The day-one requirement is a discipline, not a feature: no module outside the incumbent adapter entry may statically import a solver loader. A lint rule / dependency-cruiser check in the contract package enforces it — if app.tsx ever imports ./mzn at top level, CI fails. That single rule is what guarantees the 17.8 MB stays extractable in phase 3.

7Migration plan — zero behavior change, pinned by the whole suite

Phase 0 moves the existing implementations behind the contract with no observable change. The safety property is: at every step, the full existing test suite (100 tests) plus the new conformance suite stay green, and the incumbent adapter's outputs are byte-identical to today's.

8Non-goals (phase 0)

No new solver logicPhase 0 writes zero optimization/enumeration algorithms. It only relocates types, wraps existing code, and adds the suite + composition root. The domain solver is phases 1–3.
No god-interfaceNo single Solver interface that every adapter must implement wholesale. Capabilities are segregated; an adapter implements only what it provides, a consumer depends only on what it calls (interface segregation, dependency inversion — SOLID throughout).
No speculative abstractionsAn interface exists only where two real implementations exercise it. EvaluateBatch is declared but skip-marked until its first backend; Simulate is deliberately left OUT of the contract (one implementation, a different engine).
No performance workPhase 0 changes no hot-path timing. The <16 ms evaluate budget is preserved by keeping Evaluate synchronous and in-process; the batch/GPU speedups belong to later phases.
No domain leakageThe contract stays domain-agnostic: grep it for aws/lambda/latency ⇒ zero. Slots, tunables, keys and objectives are opaque ids exactly as the engine treats them today.

Confirm this is in phase-0 scope (it is a real behavior change to Enumerate, though a strictly-more-honest one).
Cancellation is not passed to solvers today Staleness is guarded at the call site / worker epoch (web-workers-concurrency); the solver functions take no AbortSignal. Contract adds an optional signal?: AbortSignal to the async requests; incumbent honors it best-effort (the WASM solves are not always interruptible mid-run, so cancellation may mean "discard the result", not "stop the CPU"). Decide: is best-effort cancellation acceptable for phase 0, with true interruption a later item?
Two optima can differ yet both be valid Float MIP: MiniZinc and the domain solver may pick different assignments achieving the same objective within ε. Referee's equivalent() compares objective value (float-tolerant) + SLO satisfaction, not the raw assignment vector. Confirm the equivalence relation is "same objective + all SLOs hold", not "identical knobs".
Registry binding: request field or adapter construction? Today createEngine(registry, …) binds the registry once; the graph is per-call. Proposed: registry bound at adapter construction (as today), only the graph in the request — keeps requests light and matches the facade. Confirm vs. putting the registry in every request (more explicit, more verbose).
