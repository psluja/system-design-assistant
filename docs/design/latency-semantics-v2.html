Latency SLO Semantics v2

The owner-approved rethink of what a latency requirement means in SDA. The center of the design is the architect's own workflow: set a latency requirement on a service and read it — mean and percentiles — right there, continuously. Every node is a potential beginning: one simulation yields a consistent latency perspective for every node at once. This document replaces the last "accumulated-from-the-origin" latency rule with three positively-defined, anchored quantities that each match a number the interface already shows.
SDA design document · 2026-07-03 · latency semantics v2 · status: for owner review — not started
About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas and the tool computes the design — load, latency, cost, feasibility — against declared requirements, live on every edit. Underneath, SDA is a domain-agnostic engine that turns the drawing into a typed-property graph and a family of solvers, over which a content pack layers all cloud meaning as data. Latency is computed two ways that must agree: a fast scalar pass (queueing-aware means, on every edit) and a discrete-event simulation (the DES — true percentiles). This document redesigns how a latency SLO (a declared requirement like "≤ 200 ms") is anchored and judged, so that the number the tool checks is always the number the architect is looking at — and it makes every node's latency perspective available from a single simulation run.

SLO / banda declared requirement on one property of one place in the design (e.g. latency ≤ 200 ms). Internally a band: a min / target / max for a scalar, or a set of percentile targets for a tail.
verdictthe tool's honest read-back on a band: ok / warning / violation / unknown, with a cause-chain and a ranked remediation. Uncertainty (unknown) is a value, never a guess.
node / flowa node is one component (a gateway, a function, a database). A flow is a request's journey from a source node to a terminal node along the wires.
requirement-bearing nodea node the architect has attached a latency or tail requirement to. The engine computes a latency perspective for every node; the display commits full numbers continuously only where they were asked for — on requirement-bearing nodes and the currently selected node — showing the rest on demand (the no-filler rule, now a display concern).
response latencya node's own queued service time plus the responses of everything it calls synchronously and waits for. An async hop (fire-and-forget, a queue) cuts the wait, so it is excluded. This is the request→response time a caller of that node actually feels.
sojournqueueing-theory name for one node's time-in-system = queue wait + service. SDA computes it from an M/M/c model; it is a node's "own term".
percentile / tailp50 / p95 / p99 — the value below which that fraction of requests complete. The tail (p99) is what users feel; a mean can look fine while the tail is on fire.
async edge · ρan async edge decouples the caller's wait from the callee. ρ (utilisation) = offered load ÷ capacity; at ρ ≥ 1 a queue grows without bound and real latency is infinite.

1The workflow this serves — the design center

Everything below is derived from one concrete way of working, which the owner named as the thing to optimise for: an architect states a latency requirement on a service and keeps that number — mean and percentiles — in view while they edit. The tool's job is to make it trustworthy and legible: judge exactly the quantity the architect pointed at, and show that same quantity back. The deeper move is that every node is a potential beginning — the tool computes a latency perspective for every node from one run, so the architect can point at any service and immediately read it, whether or not a requirement was set there first.

Three properties follow directly, and the rest of this document is their consequences:

  • Every node has a latency perspective; the display decides what to commit to. The engine holds no "watched list": it computes each node's response latency and response percentiles for all nodes. The display carries the no-filler rule (memory ui-no-absent-feature-filler) — it shows full numbers continuously on requirement-bearing nodes and the currently selected node, and every other node's view on demand. Nothing is computed selectively; only presented selectively.
  • What is judged equals what is shown. The verdict on a node must be computed from the very number rendered on its card. Today that is not true (§2), and it is the bug the owner hit live.
  • Percentiles are first-class and continuous. A node shows mean and p50/p95/p99, refreshed after each simulation run — not only an end-of-path global tail.
This is a semantics change, not a new feature bolt-on. SDA already computes every mean v2 needs — per-node response latency is shipped content (content/sda/src/queueing.ts responseLatency), and the DES already produces true percentiles. v2 re-anchors the SLO to those existing quantities, adds one all-node sampling pass to the DES (§4), and closes two honesty gaps. Much of the work is deleting a wrong rule and pointing the verdict at the right, already-computed number.

2The flaw being fixed — "accumulated from the origin"

SDA's scalar latency SLO is judged today against the latency accumulated along the synchronous path from the traffic origin to the node carrying the requirement. That rule is the last remnant of the old "flows begin at the client" bias that the universal-origins work removed everywhere else — and it is wrong on four independent counts.

2.1 Where it lives in the code today

FactIn the code
The latency registry key sums along the synchronous path and an async hop cuts the caller's wait.content/sda/src/registry.ts line 120: aggregate: { series: 'sum', onAsyncEdge: 'cut' } ("accumulates on the synchronous path; an async hop cuts the caller's wait").
The queueing-aware latency the verdict actually judges is accumulated source→node, taking the max over predecessors.content/sda/src/queueing.ts realCumulativeLatency (lines 145–171): upstream + own(id), Math.max(...ps.map(cum)) — a forward fold from the origins.
The latency verdict is computed from that accumulated value.content/sda/src/verdict.ts realAwareVerdicts (line 58 reads realCumulativeLatency; latencyStatus judges it, lines 20–28).
But the node card shows a different number: the node's response latency (own + synchronous downstream subtree).content/sda/src/queueing.ts responseLatency (lines 182–210) → responseLatencyMs per node (doc-15 §4 "one truth"; rendered on canvas, MCP evaluate, and the design doc).
The probe the owner hit. On a real design a node card read responseLatencyMs 120 while the latency verdict on the same node judged an accumulated 200.00000001 and failed. Two surfaces, two numbers, one node — the tool contradicted itself. (The trailing …0001 is a second, independent bug; see §5.)

2.2 Why the accumulated-from-origin anchor is wrong

#DefectWhy
1Ill-defined under universal origins with fan-inSDA no longer assumes traffic "begins at the client": any node can originate load (registry.ts originRps). With several origins fanning into a node, "accumulated from the origin" has to ask which origin — and the honest answer (max over predecessors) is a path the architect never named. The anchor depends on upstream topology the requirement is not about.
2Contrary to industry SLO practiceReal latency SLOs anchor to an operation / endpoint: "the p99 of the response served from this service". They measure from the point of the requirement outward to what it depends on — not from some upstream generator inward. v2's node anchor is the standard one.
3Redundant"End-to-end latency" is already exactly the response latency of the entry node. Carrying a separate origin-accumulated quantity gives two names for one thing at the terminal and a meaningless thing everywhere else.
4Inconsistent with what the UI displaysThe card shows response latency; the verdict judged accumulated latency. A tool whose check disagrees with its own display fails the project's first invariant — the tool must not lie (memory web-is-a-dumb-renderer).

Notably, doc-15 (the response-time model) already foresaw this reckoning: it shipped responseLatency but recorded as an open follow-up whether "response time supersedes the source-sum latency for the caller-facing SLO", and deferred a response-time SLO verdict pending that reconciliation (backlog/docs/doc-15 §5–6). Semantics v2 is that decision: it does.

3Semantics v2 — three anchored quantities

v2 defines latency requirements positively, as three distinct, each-named quantities. An architect chooses the one that matches the question being asked; each maps to a quantity SDA already computes and displays.

QuantityDefinition (positive)Anchored toMaps to today
Node latency SLO
re-anchored		 The node's response latency: its own queued sojourn plus the responses of everything it calls synchronously, combined by its latencyComposition knob (sequential = sum · parallel = max · hedged = min). An async boundary cuts the wait and is excluded. A saturated (ρ≥1) synchronous dependency makes it ∞. the node carrying the SLO (an endpoint / operation) responseLatency — the exact number the card shows (queueing.ts 182–210). The verdict just needs to read this instead of realCumulativeLatency.
Flow-scoped lag SLO
new anchor		 The full journey latency from a chosen source node to a chosen terminal node, including async queue waits along the way. Answers data-propagation questions: "a change written here reaches Aurora in ≤ X" (a CDC / replication lag). a named flow (source→terminal) reuses the guarantee-SLO's flow keying (source, terminal) from content/sda/src/guarantee-slo.ts — but as a numeric quantity, and unlike the node SLO it deliberately includes the async waits the latency key cuts.
End-to-end
falls out		 Not a new mechanism: it is a flow-scoped SLO whose source is the entry node. Equivalently, the response latency of the entry node. The number the design doc's "end-to-end flow row" already reports. the entry node (as source) / the terminal read-back the flow terminal-read latency in content/sda/src/doc-model.ts (the end-to-end flow row, 222–227).
Why lag is a separate quantity, not just "the node SLO on the terminal". The node/response anchor exists to answer "what does a caller of this service wait for?" — so it correctly cuts at the async boundary (a caller does not block on a fire-and-forget). The lag anchor answers a different question — "how long until this data lands there?" — whose whole point is the async queue wait (retention, drain time). The same design legitimately carries both: a response SLO on the API tier and a lag SLO on the CDC→Aurora replication path. Two questions, two anchors; neither is redundant with the other.

4Per-node percentiles — one run, N perspectives new simulator capability

Today the simulator answers exactly one tail: the whole-system end-to-end sojourn, and a percentile SLO is honest only on a terminal node. v2 makes the DES report a response tail for every node from a single run — because a node's latency perspective is nothing but the suffix of the very same job journeys the run already produces. There is no second simulation and no per-node model: it is bookkeeping over one run.

4.1 The key insight: a node's view is a suffix of the same journey

A simulated request already flows through the whole design once. Node X's response latency, on that same request, is simply "the clock from when the job enters X to when X's synchronous subtree completes" — a sub-interval of the journey the DES is already tracking. So one run yields N consistent perspectives at once: entering a stamp at each node the job passes through and recording the elapsed time when that node's synchronous work finishes. The async boundary cuts exactly as it does for the scalar responseLatency (a fire-and-forget hop is not part of the caller's wait), so every node's mean and tail are twins of one definition — the same quantity, measured analytically (scalar) and empirically (DES).

4.2 Cost: trivial by construction

Memoryone reservoir per node — a few thousand samples × node count. Trivial for any drawable design.
CPUa constant, small per-event overhead (record an entry stamp, close a completed sub-interval into the node's reservoir). No extra passes, no extra runs.
Runsexactly one — the same run that already produces the end-to-end sojourn now also emits every node's suffix. All-node sampling is cheaper than a watched-list would be to maintain, and needs no plumbing of "which nodes to watch" into the engine.

4.3 What exists today, and the one change

FactIn the code
The DES keeps one reservoir of end-to-end sojourns and exposes a single sojournPercentile(p).engine/sim/src/des.ts: one sojourns[] array (146), pushed at terminate as sojournEnd − req.systemEntry (235), read by sojournPercentile (427). v2 adds per-node reservoirs alongside it.
A tail verdict is attached only to sink (terminal) nodes; a percentile band on a non-terminal node stays unknown.app/mcp/src/simulate.ts line 38 (sinks.has(String(node))) and app/presenter/src/sim-verdicts.ts 40–46 — the sink gate v2 removes.
tailLatency is a node-local, DES-only key with a percentiles band, verified by checkTailBands, which already iterates any node's band — only the provider was terminal-only.registry.ts 123 (local: true); engine/solve/src/verdict/tail.ts checkTailBands.

The verdict machinery needs no change; the tail provider simply drops its sink gate and returns any node's own response percentile:

// today (sink-only, one global tail):
const tail: TailProvider = (node, key, q) =>
  key === keys.tailLatency && sinks.has(String(node)) ? ms(sim.sojournPercentile(q)) : undefined;

// v2 (every node — its own synchronous-subtree response tail, from the one run):
const tail: TailProvider = (node, key, q) =>
  key === keys.tailLatency ? sim.responsePercentile(node, q) : undefined;  // undefined only ⇒ no run yet ⇒ honest `unknown`

A node's tailLatency SLO is judged against its own response tail; end-to-end stays available as the entry node's view. Every node's mean + p50/p95/p99 is available after each run — the display (§1) decides which to show continuously.

Inherited honest limit to state plainly: downstream composition is serial today. The suffix views compose a node's synchronous subtree serially (sum), which is exact for sequential calls and the conservative bound otherwise. Parallel / fork-join (max) remains the recorded idealization — the scalar pass approximates it with max(means), which under-estimates the true tail by Jensen's inequality (doc-15 §3). Because a node's view is just a suffix of the real journey, it inherits true parallel composition automatically the moment the DES models fork-join — no per-node change is needed. v2 does not close that gap; it states it and structures the sampler so the eventual fix propagates for free.
Invariant kept: scalar and DES are twins, differential-tested. The per-node response mean already exists (responseLatency); the per-node response tail is its DES counterpart under the identical async-cut, serial-composition definition. That gives a clean differential test — the DES per-node mean of the response reservoir must track the analytic responseLatency within tolerance on unsaturated designs — exactly the "scalar approximation + DES truth" discipline SDA already holds for the system sojourn (memory simulation-modeling, testing-strategy).

4.4 Boundaries — when a response perspective is well-defined R1 scope, owner discussion

A response is a suffix of a request's journey (§4.1), so it is well-defined exactly when that journey has a finite synchronous end. Four topologies mark the honest edges of v2 — stated positively, as what each is and how far v2 reaches. The R1 sampler implements the first three; the fourth is named as the next extension.

TopologyWhat its response perspective isv2
Acyclic synchronous request
the norm		 Every node's response is the finite clock from its entry to its synchronous-subtree completion. The suffix bottoms out at the leaves and folds up; the entry node's suffix is the whole journey (= end-to-end). Fully supported. This is the case the sampler measures every run.
Synchronous cycle within one request (A waits for B, B waits back for A on the same request) No finite response exists — a caller waiting on its own reply is a deadlock, and the "suffix until the synchronous subtree completes" never terminates. The honest read-back is that there is nothing to measure. Honestly rejected. The numeric folds are cycle-guarded (both realCumulativeLatency and responseLatency stop at a revisited node; the solvers reject cyclic synchronous flow), so v2 inherits that rejection rather than inventing a number. Uncertainty is a value, never a guess.
Cycle broken by an async edge, or a probabilistic back-edge (a retry / feedback loop with p < 1) The suffix stays finite. An async back-edge is fire-and-forget — it cuts the caller's wait (§4.1), so the loop never enters the response. A probabilistic loop re-enters with probability p < 1, contributing a geometric series that converges; its expected work is bounded and its tail is a real, finite distribution. Supported. The async cut is exact today; a probabilistic loop is what the DES already simulates naturally over time (it plays out each re-entry as a real event), so its response tail falls out of the same run — no special case.
Full-mesh / any-to-any topology, each request using an acyclic subset
named future extension		 The drawing has cycles but no single request does — each request threads an acyclic path through the mesh. A well-defined response then depends on which request we mean: the perspectives are per-request-class, not per-node-in-isolation. Out of v2 — named, not silent. Distinguishing the acyclic path a class of requests actually takes is a multi-commodity (request-class) flow problem: one flow per class, each acyclic, sampled independently. v2 scopes to a single commodity; request classes are the explicit next extension, at which point the same suffix sampler runs once per class.
Why this is a clean boundary, not a gap. Every case above resolves to one of three honest outcomes — a finite measured response, an explicit rejection (a synchronous self-wait has no answer), or a named future scope (request classes). None of them asks the sampler to fabricate a number for a journey that has no finite synchronous end. That is the same discipline as the rest of SDA: where a quantity is undefined, the tool says so.

5Boundary epsilon — an independent bug found alongside honesty fix

The probe's second surprise was the trailing noise: computed 200.00000001246752 versus a band max of 200 read as a violation. That ~1e-8 is floating-point residue from summed queueing terms, not a real breach — yet band judging compares with a bare >, so the design "failed" its SLO by a rounding artefact.

Judge siteToday
Engine scalar band checkengine/solve/src/verdict/check.ts statusForBand (74): if (band.max !== undefined && v > band.max) return 'violation' — exact. (Its point shape already tolerates EPS = 1e-9, line 79; minTargetMax does not.)
Content real-aware latencycontent/sda/src/verdict.ts latencyStatus (23): v > band.max — exact (this is the one that surfaced in the probe).
Tail bandsengine/solve/src/verdict/tail.ts (47): if (v > target) — exact.
Goodput / error-rate ratescontent/sda/src/verdict.ts statusForRate (166): v > band.max — exact.

The fix: band judging adopts the contract's shared float tolerance — closeEnough (engine/solver-contract/src/bindings.ts 47, ε = 1e-4, relative+absolute). A value within ε of a bound is at the bound (ok), while a real breach beyond ε still fails honestly. This is not a new invention: the solver oracle harness already judges bands with the same closeEnough (engine/solver-contract/src/harness/oracle.ts 156–158) — so adopting it in the verdict path makes the live judgement and the referee agree, closing a latent divergence rather than opening one.

Scope discipline. ε is a boundary tolerance, not a slack budget: closeEnough(v, max) only rescues values within 1e-4 (relative) of the bound. A design that misses its SLO by any meaningful margin still reads violation — the tool must not lie in the other direction either. This is a shared-constant change applied at each judge site above; it is orthogonal to the anchoring change and can ship independently.

6Migration — honest, and mostly more generous

The change re-reads existing SLOs; it must do so transparently and without silently tightening or loosening anyone's intent. Every existing latency band means one of two things today, and each has a defined destination.

Existing SLO (today's meaning)v2 destinationEffect on the verdict
A terminal-anchored latency band meant as end-to-end (the finale's ≤ 250, the worked examples, tests, the generated design doc's end-to-end row, the MCP set_slo latency docs). Converts to a flow-scoped SLO with the entry node as source (i.e. explicit end-to-end). Same number, now named for what it is. No change at the terminal (response of the terminal = accumulation to the terminal). Behaviour-preserving for terminal SLOs.
A latency band on a non-terminal (mid-path) node. Re-reads as that node's response latency (own + synchronous downstream), the number its card already showed. Usually more generous — response (own + downstream) is typically smaller than accumulation-from-origin (which dragged in all upstream hops). State this plainly to the architect; it is a correctness fix, but it can flip a red node green.
A percentile (tailLatency) band. Moves consistently: judged against the node's own response tail (§4) instead of only being answerable at a sink. Non-terminal tail SLOs that read unknown today become real verdicts. No terminal regression.

Documents to update in lockstep (so no surface contradicts another — memory web-is-a-dumb-renderer): doc-8 (the Coverage Map) latency rows; the sensitivity-matrix rows that reference end-to-end latency; the generated design-doc latency section (content/sda/src/design-doc.ts / doc-model.ts); and the MCP tool docs for set_slo / evaluate / simulate that today describe latency as an end-to-end/terminal quantity.

8Open questions & design tensions for the owner

QuestionWhere it comes fromProposed lean
How does set_slo express flow scope? app/mcp/src/tools.ts set_slo (313–337) sets a band on node.key — it has no source→terminal notion. A flow-scoped lag SLO needs both endpoints. There is a flow-keyed surface already: set_guarantee_slo (340), but it is categorical. Add an optional {source, terminal} to set_slo for the lag/end-to-end case (node-only stays the response SLO); reuse the guarantee-SLO's (source, terminal) serialisation. Confirm vs. a dedicated numeric flow-SLO tool.
Is the display commitment rule right? The engine computes every node's view; the display must decide which to show continuously vs on demand (§1, §7 R3). Default: full mean + percentiles continuously on requirement-bearing nodes and the currently selected node; every other node one hover/click away. This keeps the surface calm without hiding any computed number. Confirm that default (e.g. should a hovered node also commit, or only a selected one).
Is lag computed analytically or DES-only? Lag must include async queue waits (retention/drain). The scalar pass can approximate a stable queue's wait (backlog / drain); a saturated or bursty queue's true wait is the DES's. Scalar shows an analytic lag approximation live (like the queueing-aware mean); the DES gives the honest lag tail. Mirrors the existing mean/tail split. Confirm the approximation is wanted live, or lag should read unknown until a sim run.
Keep realCumulativeLatency at all? Once the node SLO reads responseLatency and end-to-end is a flow SLO, the source→node accumulation has no verdict consumer — but it may still drive a display or the ρ-cause chain. Retire it from judging; keep it only if a non-SLO display genuinely needs it, else delete (no speculative code). Decide whether any surface still wants the origin-accumulated number.
When does true fork-join composition land, and is serial-now acceptable? The suffix views compose downstream serially (sum); parallel-max remains a recorded idealization (§4 limit, doc-15 §3). Because a view is a suffix of the real journey, it upgrades automatically when the DES models fork-join. Ship serial composition now (exact for sequential, conservative otherwise), stated plainly on the surface; schedule true fork-join in the DES as its own item, at which point every node's tail sharpens with no per-node change. Confirm serial-now is an acceptable honest bound for v2.

9Non-goals

No engine domain leakageResponse latency and the queue model stay in content (doc-15 §4 ADR); the engine remains domain-agnostic. v2 changes which content quantity the verdict reads, not where the computation lives.
No new solverThe scalar quantities already exist; the only new computation is the DES per-node response reservoir (R1). No MiniZinc/clingo change.
No slack budgetsThe epsilon (§5) is a boundary tolerance for float noise, not headroom on an SLO. A real miss still fails.
No engine watched-listThe engine computes a latency perspective for every node from one run — there is no "which nodes to watch" plumbing in the engine. The no-filler rule is purely a display concern (requirement-bearing + selected nodes continuous; the rest on demand). Nodes with no requirement still gain no verdict they were not asked for.
