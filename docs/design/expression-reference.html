The Relation Expression Language

A component in SDA has no code. Its entire behaviour is a handful of one-line arithmetic strings — its relations — and this document is their reference: the exact grammar the parser accepts (and what it refuses, with the real error messages), what an expression means when the engine runs it, a cookbook of twelve proven patterns lifted from the shipped catalogs, and the honest boundary of what this language must never be asked to model.
SDA reference · 2026-07-05 · the manifest relations.expr language · grounded line-by-line in the shipping parser, evaluator and catalogs
About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas and the tool computes the design — load, latency, availability, cost — against declared requirements, live on every edit. Every building block on the canvas is described by a manifest: a pure-JSON data record of its ports, its configuration knobs, its SLO bands, and its relations. A relation is one line of arithmetic — for example concurrency / (perRequestDuration / 1000) — that tells the engine how one quantity of the block is derived from others. This reference is for the architect who opens the ✎ component editor (the pencil button in the palette, or “✎ Edit type” on a selected node) to author a custom component, reaches the relations array, and asks: what language is this, and what can I model with it?

manifestthe pure-JSON description of one component type: ports (typed connectors), config (numeric knobs with defaults and units), relations (derived quantities — the subject of this document), bands (built-in SLO checks). Manifests reference a shared vocabulary by id and never each other.
registry keya governed name for one kind of quantity — throughput, latency, cost, concurrency — declared once with its unit and its composition algebra. Every key a manifest mentions must exist in the registry; there are no free-form fields.
relationone entry of a manifest's relations array: { key, reads, expr }. key is the registry key it produces, expr is the expression string this document specifies, reads declares which keys it consumes.
node · wirea placed instance of a component on the canvas, and a typed connection between two ports. Values flow along wires; a relation only ever computes within its own node.
inflow · self · outflowthe three reference primitives of the language: what arrives from upstream, this node's own value, and what downstream consumers pull. Defined precisely in §3.1.
fixpointhow the engine evaluates: all relations of all nodes become one equation system, solved by iterating until the values stop changing. Authoring order never matters (§3.3).
band · verdicta band is a declared acceptable range on a key (an SLO); a verdict is the honest read-back: ok / warning / violation / unknown / did-not-converge. Uncertainty is a value, never a guess.

1What this language is — and why it is deliberately small

The expression language is a closed arithmetic subset: numbers, key references (in three flavours), + − × ÷, unary minus, n-ary min/max, and comparisons that yield 1 or 0. Nothing else — no loops, no conditionals, no user functions, no exponent, no strings. That smallness is not a temporary limitation; it is the design, and it buys three guarantees.

1.1 Pure data, no dependency hell

A component's behaviour must survive being serialized, exported, versioned and re-imported as plain JSON. An expression string over registry-key names is exactly that: it references the shared vocabulary by id and can never import, call into, or depend on another component. The manifest contract (backlog doc-3 §1) states the consequence plainly: a component affects the world in exactly two ways — its relations (its arithmetic) and its ports (what it connects to). The engine never “understands” a component; it evaluates the relations and matches the ports. throughput = concurrency / (perRequestDuration / 1000) is what a serverless function does — there is no hidden semantics behind the string.

1.2 Solver-invertible by construction

SDA does not only run your arithmetic forward (given the knobs, compute the throughput). Its search modes — optimize, repair, synthesize — run it backward: free a knob, add an objective, and let an exact solver (MiniZinc) find the smallest concurrency that still meets the SLO. Every operator in the language was admitted because a mixed-integer solver can invert it: linear arithmetic, min/max, and 0/1 indicator comparisons all translate cleanly. This is why there is no exponent: when the catalog needed availability-by-redundancy it deliberately chose a sourced step function over the parallel-machines formula 1−(1−a)ⁿ, because “the relation language has no exponent for [it], and [a] variable exponent is hostile to the MIP solver” (content/sda/src/behaviors.ts, availabilityByDeployment). A richer language would make your component pretty in the forward pass and a dead weight in every search.

1.3 One meaning, three compilers — the differential guarantee

Because the operator set is closed, the whole numeric surface of every design — built-in and custom components alike — is expressible in three independent execution engines, and SDA tests that they agree rather than hoping:

CompilerWhere it runsAgreement proven by
JS evaluator (float64)the hot path: every edit, live verdicts (engine/solve/src/relation/evaluate.ts)the reference — the other two are measured against it
MiniZinc emissionthe cold path: exact optimize / repair / synthesize (engine/solve/src/minizinc/project.ts)engine/solve/src/minizinc/differential.test.ts (forward eval vs the native solver; “a disagreement is a P0 — it means the tool would lie”), .../transform.differential.test.ts (wire-transform arithmetic), .../chain.property.test.ts (property-tested simplification on random expression trees)

The GPU compiler makes the closedness argument concrete: its decline reasons are all structural (too many cells, a build error) — there is deliberately no “unsupported arithmetic” reason, “the Expr op set is fully expressible” (engine/solver-contract/src/gpu/compile.ts, DeclineReason). Every construct you are allowed to write is guaranteed to mean the same thing in all three engines. That guarantee is the price of admission for new operators, which is why the language grows only by a deliberate meta-model evolution (the closed-framework rule, doc-3 §0) — never by a shortcut.

Precedence, loosest to tightest: comparison → + − → * / → unary minus → primary (number, reference, call, parentheses). So a + b * c >= d parses as (a + (b*c)) >= d, and -a * b as (-a) * b. Whitespace (spaces, tabs, newlines) is free between tokens — inflow (throughput) parses the same as inflow(throughput).

One honest gotcha: the built-in names are not reserved words. inflow + 1, min * 2, or a bare self all parse — without the ( the name is read as an ordinary key reference named inflow/min/self. The mistake is then caught one stage later, at the network build, as unknown key "inflow" (no such registry key exists). If you meant the primitive, the parenthesis is what selects it.

3Semantics — what an expression means when it runs

A relation is one equation in a system. The engine compiles every node's relations, configs and wires into one flat cell network and solves the whole thing to a fixpoint (engine/solve/src/network/build.ts, engine/solve/src/fixpoint/solve.ts). You author locally; the meaning is global. Three rules cover it: how references resolve, how your value combines with the network, and how the system settles.

3.1 Reference resolution — the four ways to name a value

Inside a relation on node N, a reference to key j resolves by exactly these rules (engine/solve/src/network/build.ts, resolve):

FormResolves toUse it for
j (plain)N's own config value of j if the node declares one; otherwise the value arriving from upstream. A node-local key (one the registry marks as never flowing, e.g. requiredUnits) always resolves to N's own value.reading your own knobs: concurrency, drainRate, deploymentMode. The config-first rule is what makes this safe and readable.
inflow(j)the aggregate of upstream nodes' emitted j across N's incoming wires — the demand offered to N, read before N clamps anything to its capacity. Wire transforms (fan-out ratios, batching, caps) have already been applied.anything driven by offered load: pay-per-use cost, Little's-law demand, overflow.
self(j)N's own local value of j: its relation for j if it declares one, else its config — independent of anything incoming. This is how one relation reads another relation of the same node.reading your own capacity where a plain ref would read the incoming value instead — the distinction that decides correctness in the overflow and queue recipes (§4, R4/R7).
outflow(j)the sum of the j emitted by N's downstream nodes — the demand consumers actually pull from N.drain-side logic: a queue asking “how fast is my wired consumer taking work?”

A reference that resolves to nothing — a name that is no node's config, no relation, and no key in play anywhere in the design — fails the build with unknown key "<name>", named to your node and key. Note the message is the same whether the name is a typo or a self() on a key the node genuinely has no local definition for.

3.2 Your value inside the network — the registry algebra

Your relation computes node N's local contribution to key k. What N then emits downstream is the combination of that local value with what arrived, by the algebra the registry declares for the key — the same declaration that fixes its unit:

Key (registry)UnitAlgebra along a pathSo a node's emitted value is
throughputreq/smin down the path, sum at a fan-in; carried across async wiresmin(your capacity, offered load) — the bottleneck clamp
latencymssum on the synchronous path; an async hop cuts the caller's waityour per-hop latency added to upstream's
availability, durabilityratioproductyour ratio multiplied in — every dependency must be up
cost, egressCostUSD/monthsum, sync or asyncyour monthly line added to the design's bill
overflowreq/smaxthe worst rejection at-or-upstream of this point
backlogmsg/ssumnet queue growth added up

Two consequences worth internalizing. First, you never write the cross-node plumbing — declaring a throughput relation is enough for the min-clamp, the fan-in summing and the async-carry to happen around it. Second, a node with no incoming wire simply is its local value (a traffic source is not zeroed by an empty fan-in), and an empty inflow reads as the algebra's identity: 0 for sum, 1 for product, +∞ for min, −∞ for max. That +∞ is legitimate (“no inbound constraint”) — and it is why capacity must be read with self(), not a plain ref, in any relation that might run on an unwired node (§3.4).

3.3 Evaluation — a fixpoint, so order never matters

All cells are solved together by Kleene iteration from zero (Gauss-Seidel sweeps, at most 1,000 iterations, convergence at 1e-9; engine/solve/src/fixpoint/solve.ts). Practical meaning for an author:

  • Declaration order is irrelevant. Relations may read each other (via self()) in any order, across the relations array and across nodes; the solver settles the system, not a script.
  • Cycles are allowed if monotone. A feedback loop built from min/max/+/non-negative multiplication settles on the natural (least) steady state — the same one the MiniZinc encoding provably finds (backlog doc-4 §3a, Knaster–Tarski). Non-monotone feedback (retry amplification, hysteresis) has no such guarantee and belongs to the simulator, not to relations (§5).
  • Arithmetic is IEEE-754 float64 on the hot path; comparisons yield exactly 1 or 0. The cold and GPU paths agree by the differential contract (§1.3).

3.4 Honesty at the edges — division by zero, infinity, NaN

The tool must not lie, and the language's edge cases are wired into that rule rather than papered over:

SituationValueWhat the engine does
x / 0 with x > 0+∞legitimate — infinity is a valid lattice value (it is also the empty-min identity). Downstream min absorbs it.
0 / 0, or 0 * ∞NaNthe whole evaluation is marked not converged: any NaN anywhere flips the result to the honest did-not-converge verdict (engine/solve/src/fixpoint/solve.ts). No partial number is ever shown as truth.
a value nobody computes (an SLO on a key with no relation)—the verdict is unknown, pointing at what would answer it — never a default, never a guess (engine/core/src/verdict.ts).
A real NaN war story, preserved in the catalog. The queue-backlog relation once read the node's ingest ceiling with a plain throughput ref. On a store whose throughput is a relation (not config), the plain ref resolved to the incoming value — +∞ when disconnected — and queueMode × … produced 0 · ∞ = NaN: an honest but useless did-not-converge on an idle node. The fix is the self() primitive, and the comment stands in the source as the cautionary tale (content/sda/src/common.ts, QUEUE_REL). Rule of thumb: capacity reads are self() reads.

3.5 Units — the discipline is yours

The language is unit-blind: a number is a number, and no dimensional analysis runs over your expression. The registry documents each key's unit (perRequestDuration in ms, throughput in req/s, cost in USD/month, payloadBytes in bytes), and the author converts explicitly at the point of use. The shipped catalogs model the discipline:

  • milliseconds to seconds: concurrency / (perRequestDuration / 1000) — req/s needs seconds;
  • bytes to gigabytes and a month of seconds: inflow(throughput) * payloadBytes * egressUsdPerGb * 2592000 / 1000000000 — req/s × bytes × USD/GB, normalized to USD/month with an explicit 30-day constant;
  • a comment-worthy constant is a named constant in the authoring source, never a bare magic number in the string (the catalogs interpolate MONTH_SECONDS at build time).

3.6 Which keys a relation should produce

A relation's key must be a registry key, and the meaningful targets are the registry's derived keys — the quantities the engine computes and the verdict layer judges. The natural targets, with the role each plays:

Target keyYou are declaring
throughputthis component's capacity (the ceiling the min-clamp and the search's headroom rule read)
cost, egressCostits monthly bill lines — always multiplying a visible unitCost-style knob, never a buried literal, and always reading the same knob that grows capacity (a free capacity dial would let the optimizer meet SLOs for $0)
availability, durabilityits published or derived ratio
overflow, backlog, concurrencyOverflow, poolOverflow, payloadOverflowa violation quantity: zero when healthy, positive when a documented limit is exceeded — pair it with a max: 0 band so it becomes a verdict
requiredUnits, concurrencyNeeded, poolConnectionsNeededdemand-implied sizing, for read-back and for the overflow pair

The reads array is declared metadata: the engine parses your expr and resolves its references from the string itself, so a stale reads does not change the math — but tooling trusts it (the design-doc generator finds a cost relation's driver by scanning reads; the catalog integrity suite asserts every listed key is governed). Keep reads exactly the set of keys the expression references.

4The cookbook — twelve proven patterns

Every recipe below ships in a production catalog today. Each gives the expression, what it says in plain words, and where it lives — copy the shape, swap the numbers.

R1 · Capacity from concurrency and service time

throughput = concurrency / (perRequestDuration / 1000)

Little's law read as a ceiling: a box that can hold concurrency requests in flight, each occupying it for perRequestDuration milliseconds, completes at most concurrency / serviceSeconds per second. This is the single most-used relation in the project — the serverless function, the app server, the search cluster and the stores all declare it (content/sda/src/catalog.ts, common.ts, voice.ts).

R2 · Horizontal scaling with replicas

throughput = concurrency / (perRequestDuration / 1000) * replicas

The same per-instance ceiling times a count knob. Because replicas is an ordinary config the search can free, optimize can size the fleet against an SLO — and because the cost relation reads the same knob (R7), scaling out is never free (content/sda/src/common.ts, the replicated app server).

R3 · A Little's-law resource pool — demand vs budget

poolConnectionsNeeded = inflow(throughput) * (self(connectionHeldMs) / 1000)
poolOverflow          = max(0, self(poolConnectionsNeeded) - self(connectionPool))

Two relations working as a pair. The first is Little's law run forward: offered load times hold-time equals connections the load implies. The second compares demand to the budget and reports only the excess, floored at zero; a max: 0 band on poolOverflow turns it into a verdict. Note the composition: the second relation reads the first through self() — same-node relations chain freely (content/sda/src/behaviors.ts, connectionPool; the identical shape covers the Lambda account-concurrency quota via concurrencyNeeded/concurrencyOverflow).

R4 · The universal overflow balance

overflow = max(0, (inflow(throughput) + self(assumedRps)) - self(throughput))

Offered minus served, floored at zero: what arrives from upstream, plus what this node originates itself (assumedRps, default 0), minus what it actually emits. Every catalog component that receives work carries exactly this relation — applied by one wrapper, never hand-copied — so no component can silently forget its own saturation. The critical touch is self(throughput): the node's own served value, not the incoming one (content/sda/src/behaviors.ts, OVERFLOW / withOverflow).

R5 · Capacity that emerges from a budget

throughput = self(connectionPool) / (self(connectionHeldMs) / 1000)

R1 inverted onto a pool: if every pooled connection is held for connectionHeldMs, the proxy passes at most pool / heldSeconds req/s. The component declares no throughput number at all — its ceiling emerges from the same two knobs its overflow pair (R3) checks, so the two can never disagree (content/sda/src/common.ts, the RDS-proxy manifest).

R6 · Queue backlog with a wired-consumer drain gate

backlog = queueMode * max(0, min(inflow(throughput), self(throughput))
          - (outflow(throughput) + drainRate * (outflow(throughput) <= 0.0001)))

The densest shipped expression, and every primitive earns its place. Ingest is what producers write, capped by the queue's own ceiling: min(inflow, self). Drain is what the wired consumer actually pulls: outflow(throughput) — but when no consumer is connected the outflow is ~0, and the indicator gate (outflow <= 0.0001) flips to 1, substituting the manual drainRate knob as the fallback. The queueMode 0/1 knob switches the whole relation off for components not used as a queue, and the max(0, …) floor reports only growth. Read it as: backlog grows at ingest minus drain, where a wired consumer always outranks the manual knob (content/sda/src/common.ts, QUEUE_REL).

R7 · The four cost shapes — and the never-free-capacity rule

cost = self(unitCost)                       // flat: no sizable knob — you scale by swapping the box
cost = self(throughput) * self(unitCost)    // provisioned: you pay for the ceiling you reserve
cost = replicas * self(unitCost)            // per-driver: replicas / requiredUnits / concurrency
cost = inflow(throughput) * self(unitCost)  // pay-per-use: ~0 at rest, rises with traffic

One mechanism, four billing realities: a visible base rate (unitCost, always a config knob, never a literal buried in the string) times the driver that matches how the real service charges. The cardinal rule: whatever knob grows a node's capacity, its cost relation must read that same knob — otherwise the backward search meets your SLO by turning a free dial, and every cost figure becomes fiction (content/sda/src/behaviors.ts, flatCost / provisionedCost / costPer / payPerUseCost).

R8 · The surcharge wrapper — multiply a base by a stepped factor

cost = (base cost expression) * (1 + 1 * (deploymentMode >= 1) + 0.3 * (deploymentMode >= 2))

Redundancy is not free: flipping a database to Multi-AZ doubles the bill (the billed standby), multi-region adds ~30% more. The pattern wraps the existing cost expression in parentheses — so its own arithmetic binds first — and multiplies by a factor built from cumulative indicator gates. The base model (what the cost multiplies) stays untouched; the surcharge layers on (content/sda/src/behaviors.ts, withDeploymentCost).

R9 · Piecewise values via indicator gates — the if-statement this language doesn't have

availability = 0.995 + 0.0045 * (deploymentMode >= 1) + 0 * (deploymentMode >= 2)

A step function in an integer knob: the single-AZ baseline, plus the uplift to the Multi-AZ SLA when the mode reaches 1, plus the further uplift at mode 2. Each comparison is 1 or 0, so exactly the steps at-or-below the current mode contribute. This is the general recipe for “if mode is X, the value is Y”: encode differences between consecutive steps, gated on >=, and any mode picks up the cumulative sum. Prefer gates on integer-valued knobs with >=/<=; == is exact floating-point equality and belongs only on values that are genuinely integers (content/sda/src/behaviors.ts, availabilityByDeployment).

R10 · Cap and floor

value = min(x, ceiling)      // cap: never exceed a limit — e.g. min((capacity expr), self(assumedRps))
value = max(0, x)            // floor: report only the excess / never go negative

The two bread-and-butter clamps. The engine itself uses the cap when a traffic source declares an origin — its emitted throughput becomes min((its capacity relation), self(assumedRps)), so a source never emits more than it could serve (content/sda/src/manifest.ts, the origin fold). The floor appears in every overflow-style relation (R3, R4, R6): a healthy design reads exactly 0, and any positive value is actionable.

R11 · Demand-driven sizing — the pair every autoscaling fleet uses

throughput    = maxUnits * concurrency / (perRequestDuration / 1000)
requiredUnits = inflow(throughput) / (concurrency / (perRequestDuration / 1000))

Two views of the same per-unit capacity: the ceiling (the most the fleet may scale to) and the demand-implied count (how many units this load actually needs). requiredUnits is what the cost reads (costPer(requiredUnits), R7) — you pay for what the load needs — while maxUnits bounds the ceiling; overflow past it is R4, applied automatically (content/sda/src/behaviors.ts, sizingRels, used by the Fargate/ECS-style manifests).

R12 · A monthly bill line with explicit unit conversion

egressCost = inflow(throughput) * payloadBytes * egressUsdPerGb * 2592000 / 1000000000

Data-transfer cost, the most-missed line on a real bill: requests per second, times bytes out per request, times the price per GB, times the seconds in a 30-day month, divided down from bytes to GB. Every conversion is visible in the expression — the reader can audit the dimensional chain req/s → B/s → GB/month → USD/month without leaving the string (content/sda/src/behaviors.ts, EGRESS).

6Authoring & testing a custom component — the loop

Relations are validated in two stages: a shape check when you save, a deep check when the design evaluates. The loop below gets a custom component from idea to verified in minutes, and every error names its exact location.

6.1 In the web app — the ✎ editor

  • Open: the ✎ button in the palette (new component, pre-filled with a starter manifest that already contains an R1 throughput relation), or “✎ Edit type” on a selected node. Editing a built-in loads it as a starting point and saves as a project-scoped override; the original catalog entry is never mutated.
  • Save: checks the JSON shape (a non-empty type, a ports array). Deliberately shallow — deep validation belongs to the engine at evaluate time (app/core/src/commands.ts, defineComponent). Custom components persist and export with the project and appear in the palette.
  • Place one instance and read the canvas: the evaluate pass parses every relation and resolves every reference. A bad expression surfaces as node <id> key <key>: <parser message> (the exact messages of §2.2); a typo'd reference as unknown key "<name>"; an ungoverned key as key "<name>" not in registry.
  • Test in isolation first: one instance, no wires. This is precisely how the project's own integrity suite proves every shipped manifest (“a single isolated node evaluates cleanly”, content/sda/src/integrity.test.ts) — with no wires, the only possible errors are yours. Expect your capacity relation to read its config-driven value and inflow-driven relations to sit at their identity. Then wire a traffic source in front and watch throughput clamp, overflow floor at zero, and cost move with the driver.
  • Make it judgeable: add a bands entry (e.g. max: 0 on your overflow-style key) so your relation produces a verdict, not just a number.

6.2 Over MCP — the AI-driven loop

The bridge exposes the same loop as self-correcting tools (app/mcp/src/tools.ts): list_protocols for the port vocabulary → define_component with the manifest JSON → describe_component to read back the full card (ports, knobs with defaults and units, built-in bands) → add_component / connect / set_config to stage a probe design → evaluate for the {feasible, violations, system, verdicts} read-back. Every refusal names the valid alternatives (the closest knob, the node list, the port list), so the loop converges without reading any source.

The quality bar your component is held to. A relation is credible when its numbers are sourced or marked as estimates, its cost reads the knob that grows its capacity (R7), its limits produce banded overflow keys instead of silent clamping (R3/R4), and its capacity reads are self() reads (§3.4). Meet those four and your hand-authored component participates in every mode of the tool — forward verdicts, backward sizing, Monte-Carlo confidence, GPU batches — with exactly the same standing as the shipped catalog.
