Readiness Audit — from empty canvas to a defended document

A read-only walk of the full professional architect's path through System Design Assistant, both surfaces: the canvas a human drives and the MCP tool surface an AI agent drives. Measured against the owner's bar — “a professional, with no instructions, goes from a blank canvas to a defended document in one session, and every number along the way is computed, honest, and points to the next move.” This report is findings-table-heavy by design.
SDA audit · 2026-07-04 · committed HEAD 31f7709 (“assumption model R1 — the envelope, named worlds, schema 7”) · read-only · no commits
About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas and the tool computes the design — load, latency, availability, durability, cost, feasibility — against declared requirements, live on every edit, and can run backwards to size or repair the design. Under the canvas is a domain-agnostic engine (a typed-property graph plus a family of solvers); all cloud meaning is layered on top as content (components as data). SDA has two front doors onto the same engine: the canvas (this is what a human uses) and an MCP tool surface (this is what an external AI agent uses, e.g. GitHub Copilot over a bridge). This audit walks both, as a first-time senior AWS architect who has read no documentation, and records where the path is smooth, where it is honest, and where it snags.

the barthe owner's single acceptance test, quoted above: blank canvas → defended document, one session, every number computed & honest & actionable. Every finding below is scored against it.
the two pathscanvas path = the web app a human drives with a mouse; agent path = the MCP tools an AI drives with no UI. A readiness gap on one is not automatically a gap on the other; several findings are asymmetric.
SLO · verdict · ρan SLO is a declared requirement on one property of one place (a throughput floor, a latency ceiling). A verdict is the honest read-back: ok / warning / violation / unknown. ρ (utilisation) = offered load ÷ capacity; ρ ≥ 1 means a tier is saturated and its queue grows without bound.
the envelopethe computed boundary of a design: the maximum sustained demand it can carry with every SLO still green, plus which promise breaks first. HEAD 31f7709's headline feature — the “default answer” that needs no declared demand.
provenancewhere a number came from, shown as a badge: documented (sourced) · estimate (credible catalog default) · architect (set by hand) · default (untouched). The tool's honesty axis: it never invents infra numbers.
severityblocker stops the path · major a professional stalls or is misled · paper-cut friction that annoys but does not block.
CLOSED / PARTIAL / OPENused in the doc-10 delta (§7): a prior-evaluation gap now fully addressed / substantially but not fully / untouched.

1Verdict, up front and honest

The engine has crossed a line since the last evaluation. A professional on the canvas path can, in one session with no instructions, build a real system, watch every number compute honestly, size it backwards to meet an SLO, declare an uncertainty range and get a distribution, and press one button to get a genuine, defensible design document — C4 diagram, capacity table, tail latency, sourced reliability, an assumptions register with provenance, a cost breakdown with per-component derivation, and risks with remediations. Two years of “a fast calculator that authors nothing” is over: SDA now authors the deliverable, and it is good.

Where the bar is not yet met, the gaps cluster in three honest places. (a) The agent path is more fragile than the canvas path — the same journey that flows on the canvas can strand an AI on step one (an ambiguous wire aborts the whole build; an empty design returns a green with no hint), which is exactly the “Copilot wandered” behaviour the owner saw. (b) This week's headline feature, the envelope, is not yet reachable — it has no web surface, and the committed MCP build predates it. (c) Two polish snags — there is no “New / blank design” affordance (the literal “empty canvas” start), and the deliverable ships titled Untitled. None of these is a deep design flaw; all have cheap, honest fixes. The single highest-leverage move is to make the agent path as guided-and-guarded as the canvas already is.

2Method — the walk, both surfaces

Persona: a senior AWS architect, first session, no docs. Storage cleared (IndexedDB sda deleted). Canvas path: driven with the Playwright/Edge browser tools against my own dev server at localhost:5188 — discover how to add components, read the live verdicts, set an SLO, run Improve, declare a ± range, watch uncertainty, generate the design doc, export & reload. Agent path: a headless JSON-RPC harness driving the committed built server app/vscode/dist/mcp-server.cjs over stdio — the same journey via tools only (list_components → apply_design → evaluate → set_slo → simulate → repair → generate_doc), plus deliberate probes of the failure surfaces. Findings cross-checked against committed source and against backlog doc-10 (the June 30 brutal architect evaluation).

First surprise: the “empty canvas” is not empty. With storage cleared and the page reloaded, SDA does not open blank — it seeds a working checkout demo (Web client → NGINX → Checkout API → Postgres) with a throughput SLO already set and Postgres already saturated (ρ 250%, dropping 3,000 req/s), overlaid by a “Getting started” card. This is a reasonable onboarding choice — a professional has something real to react to in the first second — but it is in tension with the literal bar (“from a blank canvas”), and there is no one-click way back to blank (F7). The seed is intentional: app/web/src/onboarding.ts treats untouched-seed OR empty as first-run material.

4The agent path — where an AI stalls

The headless harness against the committed mcp-server.cjs tells a rougher story, and it maps precisely onto “Copilot wandered.” The tool ergonomics are excellent — guided errors name the nodes, ports and types, and even rewrite the single most common malformed call — but three silent-wrong or hard-abort behaviours can end a session on the first move.

1major Agent path · apply_design build A whole design to build in one call; if a wire is under-specified, the tool resolves or asks — without discarding the rest. One ambiguous out-port (svc has [db, out, cache]) aborts the entire atomic build to 0 nodes, reporting one ambiguity at a time; the session then dead-ends on “node not found.” This is the literal “wandering” mechanism. Empty / no-origin design returns feasible:true, violations:0, flows:[] with no note — a false green an agent can mistake for done. 6What works well (fair credit)

The deliverable is real. The generated Design Document is the single biggest change since doc-10. On a design with declared demand it produces, from the verified model: an honest Summary verdict, a Requirements table, an assumptions register with provenance, a C4 container SVG, a capacity/ρ table with charts, a DES tail (p50/p95/p99), a sourced Reliability section (AWS Well-Architected links), a cost table with per-component derivation including Multi-AZ billing, and Risks with remediations. On an empty/no-demand design it degrades honestly (“a 0-component design…at an unstated load…No SLOs declared…an empty register is an honest register”) and flags the author-required sections (security, resilience narrative, alternatives, rollout) rather than faking them.
  • Numbers are computed and honest, everywhere. Saturation yields ∞, not a soothing average; the real queueing-aware latency is primary and the ideal is a parenthetical; ρ, drops, and per-node cost are on every tier; the tail is a DES percentile, not the mean.
  • Backward solve that points to the next move. The Improve panel names the binding constraint and shows the exact before→after sizing with Apply — the bar's “every number points to the next move” made literal.
  • Uncertainty is ambient and reachable. Declaring a range instantly produced a Monte-Carlo distribution with a seed and a confidence interval — no source-reading, no separate “run” ritual.
  • Guided MCP errors are best-in-class. Unknown node lists the nodes; unknown type suggests near matches; an omitted-but-ambiguous port lists the options; the fused "node,port" mistake is detected and rewritten. This is real anti-wander ergonomics — the abort scope (F1) is the flaw, not the messaging.
  • Empty states are honest, not blank. The tail panel with no origin says “set a client throughput to simulate the tail” (app.tsx:2022); the flow footer says “no traffic origin” and explains any node can originate via originRps.
  • Persistence you can trust. Autosave round-trips a full reload including declared ranges; the schema-7 export is the real, diffable backup.
  • The MCP reliability tool returns AWS-sourced availability & DR tiers with RPO/RTO — a genuine reliability-pillar contribution, on the agent path.

8The fresh-sore probes

Empty-section guidance — does anything say WHY / what next?

Canvas: yes, honestly. The tail-latency panel without an origin reads “set a client throughput to simulate the tail”; the flow footer reads “no traffic origin” and explains any node can originate via originRps (the source comment: “the flow figures above are vacuous — say WHY”). No silent blanks. Agent: no. The same empty state on evaluate returns a false green with no note (F2). The guidance exists as UI copy but not as tool output — the asymmetry is the fix target.

Discoverability of SLO entry

Good. SLOs live in the System panel's Requirements (end-to-end) block as inline, always-visible inputs with live read-backs — a professional finds them by reading the panel, not by knowing a command. (The seed even ships one set, modelling the interaction.)

Is the envelope / uncertainty story reachable without reading source?

Uncertainty: yes. The ± popover is self-explaining and applying a range lights the ambient Monte-Carlo surface immediately (seed, sample count, confidence interval). Envelope: no — it has no web surface (F4) and is absent from the committed MCP build (F3). The one place it partially surfaces is inside the generated document's capacity section. For a feature billed as “the default answer that needs no demand,” it is currently the least reachable of the three views — and, tellingly, is the thing being built right now.

Doc generation on a design with no declared demand

Honest and graceful. generate_doc on the empty design produced: “a 0-component design carrying 0 request flow(s) at an unstated load at about $0/month. No SLOs are declared yet…”, an empty-but-honest assumptions register, and a completeness table marking security/resilience/alternatives/rollout as ⚠ author required. It never invents a demand or a number. The one thing it does not do on a no-demand design is offer the envelope (“how far could this go?”) in place of the absent demand — which is exactly the envelope's reason to exist, and the F4 gap seen from the document side.
