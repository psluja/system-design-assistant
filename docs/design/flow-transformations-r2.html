Flow Transformations — R2: the surface

Making the engine’s new truth visible and touchable: rates on the wires, transform knobs on every component, in both shells — native-first in VS Code, popover-first on the web.
SDA design document · 2026-07-02 · R1 (engine) shipped in 1f10a05 · status: for owner review
About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas — services, databases, queues, connected by wires — and the tool computes that design instead of merely drawing it: it derives each component's load, latency, availability and cost, checks them against the architect's requirements, and can search for better configurations. This document is self-contained; earlier design documents are optional further reading, not prerequisites.

the two shellsSDA ships as a web application and as a Visual Studio Code extension; both render the same canvas from shared code.
presentera shared library that turns raw engine results into display-ready data, so both shells show identical numbers by construction.
transforma declared function on a port or wire (introduced in the companion document) that changes the traffic rate an edge carries — e.g. 'this port emits 100 log lines per request'.
pillthe small rounded label this document introduces on a wire, showing the rate the wire actually carries (e.g. '×100 → 100k/s').
Inspectorthe side panel showing the selected component's editable properties.

1What R2 must achieve

R1 made the engine compute transformed flows. R2 has one job: an architect glancing at the canvas must see what each wire really carries, and change any transform in two clicks — on every component, exactly as the owner required.

2One brain, two faces (the anti-drift rule)

Per the “web is a dumb renderer” doctrine, all edge-rate math lives in ONE new presenter module, consumed verbatim by both shells:

Anti-drift invariant: the presenter never re-derives engine math. R1 deliberately placed transforms in the shared expression seam; R2 extends the same discipline to pixels: one function, one import path, zero copies. A unit test pins presenter edge rates against a full engine evaluation.

3The canvas language — rates on wires

RuleRationale
Identity edges show no persistent label; exact rate appears on edge hover (tooltip-pill)50 wires × permanent numbers = noise; the interesting places are the non-1:1 ones
A transformed edge gets a persistent pill: function + resulting rate (×100 → 100k/s, ÷100 → 10/s, cap 500/s, p=0.01 → 10/s)the whole point of the feature — the surprise must be visible at a glance
Tone: amber for amplification (k>1), teal for reduction, gray for cap/window, dashed pill for probread the SHAPE of the system without reading numbers
IN-side transforms render at the target end of the edge (small chevron pill), OUT-side mid-edgemirrors ownership: sender’s emission vs receiver’s consumption
Rates come from the LIVE evaluation; with no traffic origin the pills show the function only (no fake numbers)the tool must not lie — no rate without a declared load

4Editing — two clicks on every component

Web (popover-first)

1
Click the pill on an edge (or the port row in the Inspector) → a small popover: function dropdown (none · ratio · batch · cap · window · prob) + value field with unit hint (× / n / req/s / ms / p).
2
Apply → studio.dispatch(setTransform) (shipped in R1, undoable). Pills and downstream verdicts update live; a manifest-sourced default shows a “reset to catalog” affordance.

VS Code (native-first)

1
Selected Node tree gains a Ports section: each port row shows direction, protocols and the active transform (override marked modified, GitLens-style). Inline $(edit) → QuickPick (function) → validated InputBox (value); inline $(discard) clears an override.
2
The edit is a document text edit (new pure setTransformText in document-edits.ts, same pattern as setSloText) → native undo/dirty/save. The canvas pill click posts the existing select and reveals the Ports row — editing stays native (doctrine), the webview never grows its own form.
Frozen protocol untouched. The webview already receives the document (pills render from it + evaluation); edits flow through existing seams (web: dispatch; vscode: document edit → docExternal). No new host↔webview message types.

everything elseidentityfan-outs, log ratios, cache-miss ratios are workload facts, not component facts — inventing defaults would lie; the per-instance knob is the feature

doc-8 gets the coverage row + this honesty rule; the design-doc generator (doc-7 deliverable) lists every ACTIVE transform in the capacity section (“log pipeline receives ×100 of checkout traffic”).
