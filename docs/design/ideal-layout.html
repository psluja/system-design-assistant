The Ideal Layout

Twenty years of being irritated by layout algorithms, and one question: “czy możemy być najlepsi? marzy mi się idealny algorytm” — can we be the best? This document argues we can, and says exactly how. The thesis is one sentence: a generic engine (dagre, ELK, React Flow Pro) lays out an abstract graph — anonymous nodes and edges; SDA knows the diagram — its flows, its roles, its sync/async wires, its request classes, its groups — and so it can lay out meaning. That is an advantage no generic tool can copy, because it has thrown the meaning away before layout begins. Here is the objective it optimises, the search that spends real GPU compute to find the optimum, and the benchmark that makes “best” a measured claim rather than a felt one.

About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas and the tool computes the design — load, latency, availability, cost, feasibility — live on every edit. Because the drawing is the primary surface, how it is laid out is not cosmetic garnish: a diagram an architect trusts must read cleanly — flows left-to-right, tiers aligned, the read path visibly distinct from the write path. Today SDA auto-arranges a diagram with Tidy (a classic layered layout, glossary below) in under 16 ms, and routes the wires with a pure-TypeScript orthogonal router. Both are good. This document is about the ceiling: what the best-possible layout would be, why SDA is uniquely positioned to reach it, and a design that spends seconds of background GPU compute to search for it — without ever making the canvas jump, and without changing a single computed number (a layout is beauty, never truth — §3.5).

Tidy / layered layoutSDA's current auto-arrange (app/presenter/src/layout.ts): the textbook Sugiyama recipe — put each node in a column = its longest-path depth from the sources (so flow reads left→right), order rows within a column by the barycenter (average height) of their upstream neighbours to reduce crossings, then centre each column. Fast, deterministic, dependency-free. The seed this design refines, never discards.
the routerSDA's orthogonal edge router (app/presenter/src/edge-routing.ts): draws right-angle wires that avoid node boxes — canonical straight/Z/L shapes first, a Hanan-grid A* only when blocked. Pure TS, deterministic, ~2 ms median. Critically, it produces the real geometry that ships, so bends and crossings can be measured on it, not estimated.
crossing · bendA crossing is where two wires cross; a bend is a 90° corner in one wire. Fewer of each reads cleaner. These are the metrics every generic layout engine optimises — and the only ones it can, because they need no knowledge of what the nodes mean.
dagre · ELK (elkjs)The two dominant graph-layout libraries the industry uses (dagre = the layered layout behind most JS diagram tools; ELK = the Eclipse Layout Kernel, elkjs is its JS port). Both take a bare node/edge graph and optimise crossings/bends/length. They are the benchmark SDA must beat (§6) — and the proof of the thesis, because they cannot optimise the semantic terms §2 adds.
flow · role · group · classThe meaning SDA has and a generic engine does not. A flow is one request path from a traffic origin to its terminal (content/sda/src/system.ts, requestFlows). A role is a node's tier — client / edge / compute / store — read from its type. A group is an explicit visual container (Group, member ids). A request class is a named commodity with its own acyclic wire membership over a shared topology (content/sda/src/request-class.ts). A wire is sync or async (semantics on the edge).
the three-layer disciplineSDA's honesty rule for GPU compute, established by TASK-81: the GPU proposes in fp32 (fast, approximate, preview-grade), the CPU proves in fp64 (exact, verdict-grade), and fp32 is never presented as final truth. This document adds a third proof for geometry: the router arbitrates — the winner's bends and crossings are measured on real routed wires before it is applied.
annealing / beam · seedA search over layouts: try a candidate, perturb it (swap two rows, nudge a node onto a guideline), keep it if it scores better, occasionally accept a worse one to escape a local optimum (simulated annealing); or keep the best k at each step (beam). A recorded seed drives the randomness so the same design always yields the same layout (reproducible — §5.2).
pin / anchorA node the architect hand-placed. The optimiser must treat it as fixed — a hard constraint it lays out around, never over (§5.3). The tool polishing a diagram must never undo a deliberate human placement.
resting handshakeTASK-81's UX pattern: while the design is being edited, show a fast preview; when it comes to rest, deliver the confirmed result. Here: Tidy applies instantly; “polishing…” refines in the background; the better layout arrives without a canvas jump.

1The thesis — SDA lays out meaning, not an abstract graph

Every generic layout engine solves the same problem: given anonymous nodes and anonymous edges, place them to minimise crossings and bends. It is a good problem and these engines solve it well. But they solve it blind — the moment a diagram is handed to dagre or ELK, everything that made it a system (which node is a client, which path is the write, which hop is asynchronous) has been erased down to circles and lines. SDA never erases it. The design on the canvas is a typed-property graph with flows, roles, sync/async wires, request classes and groups still attached — so SDA can place a node because it is a store on the read path, not merely because a crossing-count says so. That is the unfair advantage, and it is structural: a generic tool cannot copy it without first becoming SDA.

The one-sentence thesis. A generic engine optimises the graph (crossings, bends, length — the terms that survive when meaning is thrown away); SDA optimises the diagram (those same terms plus flow lanes, role layering, read/write separation, branch symmetry, async offset, group compactness — the terms only a tool that kept the meaning can even express). Best-in-industry is therefore not a matter of a cleverer crossing-minimiser; it is a matter of optimising a richer objective nobody else can write down.

2The objective — hard constraints, then weighted beauty the owner reviews these weights

A layout is chosen by maximising a score. That score has two parts: hard constraints that a candidate must satisfy to be legal at all (a candidate that violates one is rejected, never merely penalised), and a set of weighted soft terms whose sum is the aesthetic quality. Every soft term is normalised to [0,1] so the weights are the only place preference lives — and that vector is exactly what the owner should review, because it encodes what “beautiful” means for SDA.

2.1 Hard constraints — legality, not preference

These are booleans. A candidate layout either satisfies all of them or it is discarded before scoring. They are cheap to check and they guarantee the optimiser can never “win” the score by cheating geometry.

#Hard constraintHow it is enforced
H1No wire through a non-endpoint node. The property the router already guarantees and the CQRS regression test pins at 0 cuts.The winner is routed before it is applied (§3.4); any candidate the router cannot route cleanly is infeasible. Reuses segmentBlocked / pathHitsBox (edge-routing.ts).
H2No node overlap. Two node boxes, each inflated by the minimum clearance, are disjoint.Pairwise AABB test; a proposed move that would overlap is rejected at candidate-generation time (cheap, O(n) with a spatial hash).
H3Minimum spacings. Column pitch, row gap, lane gap and group padding stay ≥ the constants Tidy already uses (COLW, VGAP, LANE_GAP, PAD_X).Candidate moves quantise to a grid of those spacings; the spacings are constants shared with layout.ts so Tidy and the optimiser agree.
H4Group containment. A node that belongs to group G sits inside G's band; a node that does not, does not.Group membership (Group.members) is a placement partition, exactly as Tidy's lanes are; a move that would eject a member is illegal.
H5Pins are fixed. A hand-placed (pinned, §5.3) node's position is immovable.Pinned nodes are removed from the free set; the optimiser lays out only around them.

2.2 The soft terms — normalised, weighted, summed

Let a layout be a set of node rectangles and the routed wire polylines. Each term below maps that geometry to a penalty p ∈ [0,1] (0 = perfect, 1 = worst plausible). The quality is score = 1 − Σ wₓ pₓ, with the weights summing to 1. The first three are the terms a generic engine also has; the rest are SDA's semantic terms — the ones that make us best.

The proposal above front-loads crossings and alignment (the two an architect notices first) and gives real weight to the four semantic terms that are our moat — but the split is a policy, not a law.

Read the worked score honestly, because it is the whole argument. On crossings — the metric dagre and ELK live and die by — SDA and raw Tidy tie at zero. If layout were only crossings, we would have nothing to add. The entire 25-point gain comes from alignment, lane coherence, symmetry and async offset — terms a generic engine cannot score because it does not know the CQRS design has a write path, two read paths, and asynchronous fan-out. This is the thesis, quantified: we are not a better crossing-minimiser; we optimise a strictly larger objective.

3The search — Tidy seeds, the semantic pass shapes, GPU proposes, CPU & the router prove

Finding the layout that maximises §2 is a search over node positions, port orders and lane orderings — a large, non-convex space. The design spends real background compute on it, structured as a pipeline that is always answering: an instant seed, a cheap deterministic shaping pass, then a GPU-accelerated refinement that proposes thousands of candidates and a CPU-plus-router stage that proves the winner. The three-layer discipline from TASK-81 governs every use of the GPU.

3.1 Stage 0 — Tidy, the instant seed (unchanged)

The existing tidyLayout runs first and is applied immediately, exactly as today. It is deterministic, dependency-free, and well under 16 ms; it gives a correct, readable layout with zero crossings on the committed examples. Nothing below replaces it — the search seeds from it, so the worst case of the whole feature is “you get Tidy, instantly,” which is what ships today. That is the safety floor.

3.2 Stage 1 — the semantic pass (deterministic, cheap, no randomness)

Before any stochastic search, a single deterministic pass rewrites the seed using the meaning Tidy ignores: assign each flow (requestFlows) to a horizontal lane; bias each node's column toward its role (§4.1); separate read from write paths into distinct bands; give sibling branches mirrored offsets; push async wires onto offset spurs; snap columns and rows to shared guidelines. This is pure, cheap, and reproducible — and it alone captures most of the §2.3 gain (alignment, lane, symmetry, async). It is the part of “best-in-industry” that needs no GPU at all; the search that follows only polishes what it cannot decide deterministically (exact row orders, tie-breaks, jog removal).

3.3 Stage 2 — GPU proposes: batch-scoring thousands of candidates in fp32

From the shaped seed, a bounded search (simulated annealing, or a beam over the best k) generates candidate layouts by local moves: swap two rows in a column, nudge a node onto a nearby guideline, flip a branch's offset, reorder a node's ports, shuffle group-lane order. The geometry score of §2 is embarrassingly parallel and fp32-friendly — crossings are pairwise segment-intersection counts, length and alignment and area are sums over coordinates — so a WGSL compute kernel scores thousands of candidates in one dispatch, one candidate per invocation. This is the exact shape the existing GPU backend already runs (one scenario per invocation, engine/solver-contract/src/gpu/webgpu.ts); here the payload is candidate coordinates rather than override tables, but the discipline is identical.

Reuse the gpu-module DISCIPLINE, author a new geometry kernel. The existing kernel scores the engine's cell network (capacity/latency math) — different arithmetic from layout geometry, so this is a new WGSL shader living with the layout code, not a call into the solver kernel. What it reuses is the pattern proven by TASK-81: reach WebGPU only through a lazily-imported entry (bundle separation — the driver stays out of the static graph), probe honestly with gpuDevice() (Chrome/Edge/webview yes, else null), fall back to CPU silently when there is no device, and report which backend ran so the surface can tag preview vs confirmed. The device plumbing (gpuDevice, the error-scope handling, device-loss re-probe) can be shared verbatim; only the shader body and the buffer layout are new.

3.4 Stages 3–4 — CPU proves, the router arbitrates (fp32 never final)

The GPU's fp32 ranking is preview-grade: good enough to pick the top handful of candidates, never good enough to apply. So the top-k are re-scored on the CPU in exact fp64 — and then the decisive step: the deterministic router runs on each finalist, so its bends and crossings are measured on the actual wires that will ship, not the straight-line estimate the batch kernel used. This matters because the two can disagree: a candidate the estimator loved may route into an ugly staircase, and one the estimator ranked third may route cleanest. The router is only ~2 ms (median 1.9 ms, per the routing round), so routing a handful of finalists per refinement step is affordable — the budget spends where it earns. The router is the final arbiter; the applied layout is the finalist that wins the fp64 score computed on real routes.

The three-layer discipline, extended to geometry. TASK-81 established two layers: GPU proposes (fp32), CPU proves (fp64). Layout adds a third, because the aesthetic that ships is a property of the routed geometry, not the placement alone: GPU ranks → CPU re-scores exactly → router measures reality. An fp32 score never decides which layout the architect sees. This keeps the “tool must not lie” invariant trivially satisfied here (§3.5) and mirrors, one-form-per-kind, the honesty pattern the whole codebase already uses for compute.

3.5 Layout is beauty, never truth — the honesty this feature owes

One structural fact makes the GPU use here low-stakes in a way the solver's is not: a layout never changes a computed number. Load, latency, cost and verdicts are functions of the graph, not its coordinates; the worst a bad layout can do is look ugly, never report a wrong capacity. So “the tool must not lie” is satisfied by construction — there is no infra number to fabricate. The three-layer proof above is therefore not about correctness of truth (there is none at risk) but about correctness of the claim we make about beauty: we do not present an fp32-ranked layout as “the best we found” until the CPU and the router have confirmed it is. And the seed guarantees the honest floor: if the GPU is absent, the device is lost, or the budget expires with nothing better than Tidy, the architect keeps Tidy — a good layout, instantly, always.

3.6 Budget, determinism, persistence

The polisher runs under the TASK-81 ambient loop: event-driven (a design mutation restarts it), debounced, latest-wins (a new edit supersedes an in-flight search), idle = zero work (no polling timer, battery-honest). A wall-clock budget (proposed 2–5 s, tunable) bounds the anneal; it may stop at any time and keep the best-so-far. The search is seeded (§5.2), so the same design yields the same layout. The applied result persists in doc.layout exactly as today — no schema change (AC#3): the optimiser writes the same {id→{x,y}} map Tidy writes.

4Hierarchy detection — the structure induces itself, no user input

The semantic pass needs a hierarchy to lay out to: which nodes are a lane, which column is a tier, which box is a group. The owner's requirement is that the architect never has to declare any of this — it is read from the design the tool already holds. Here is how flows, roles and groups induce the hierarchy, and what happens when the topology refuses to be a hierarchy at all.

Role from type is a heuristic — and that is a choice to confirm. The tier map above reads a node's role from its type prefix, which is content data the layout may inspect (the layout lives in app/presenter, which is allowed to be domain-aware; the domain-agnostic engine is untouched). This is honest but heuristic: a novel component with an unmapped prefix falls to a neutral tier and is placed by flow depth alone — never mis-placed, just un-hinted. The alternative is an explicit tier field on the registry key metadata (one-form-per-kind, like the role axis the assumption model adds). Which source of truth to use is an open question (§6).

4.2 When the topology is not a hierarchy — ambiguous, cyclic, mesh

Not every design is a clean DAG, and the tool must not invent a hierarchy that isn't there (that would be a layout that lies about the architecture). Three honest fallbacks:

  • Ambiguous depth (a node reachable at several depths): the role-layering term breaks the tie toward role order; where role is also unknown, the barycenter rule (already in Tidy) decides. Deterministic, never arbitrary.
  • Cyclic flow (A→B→…→A): a single implicit river has no acyclic layering — and SDA already names this rather than faking a fixpoint (cyclicFlowDiagnosis). The layout answer is the same as the engine's: request classes carve the mesh into per-class acyclic slices, and each class becomes its own lane laid out acyclically. Shared nodes appear once, positioned by a barycenter consensus across the classes that use them. The mesh loops; every commodity's lane is clean.
  • Full mesh / no gradient (everything talks to everything, no role slope): there is genuinely no hierarchy to draw. The layout falls back to a force-directed / circular arrangement scored by the same objective — but with the role-layering and lane terms going to zero because no structure exists to reward. Honest: when there is no hierarchy, we optimise crossings and symmetry and say nothing false about tiers.

5Tensions & the hard cases, resolved honestly

The interesting design is where the objectives fight, where determinism and search collide, and where the optimiser must keep its hands off. Stated plainly — some resolved with a lean, some genuinely for the owner.

5.1 Alignment vs wire length — they pull in opposite directions

The sharpest internal tension. Snapping a node onto a shared guideline (alignment↓) can drag it away from its neighbours and lengthen its wires (length↑). A naive optimiser oscillates between the two. Resolution: alignment operates within a snap tolerance — a node only snaps to a guideline when the displacement is ≤ τ (proposed 0.3×COLW), so alignment can never yank a node across a lane to chase a distant axis. Above τ, the length term dominates and the node stays put. The two terms are thus coupled by a dominance rule, not just summed: a large length increase vetoes a small alignment gain. The weights (§2.2) then arbitrate the residual near the tolerance boundary. This keeps “aligned” from ever meaning “stretched.”

5.2 Determinism vs annealing — a seeded schedule

Annealing is stochastic, but the project demands reproducibility (“seeded → reproducible layout”) and no surprise motion. These reconcile the same way the Monte-Carlo module already reconciles them: a recorded seed drives a fixed PRNG, and the entire schedule — temperature ladder, iteration count, the sequence of neighbour moves, tie-breaks — is a pure function of (seed, design-hash). Same design + same seed → byte-identical layout. The seed can be derived from a content hash of the graph, so it need not even be stored to be reproducible; storing it (in layout provenance, not a schema change) additionally lets the architect re-roll for a different local optimum on demand. This is exactly the seeded-determinism discipline of content/sda/src/uncertainty.ts, applied to geometry.

5.3 When the optimiser must NOT move things — pin semantics

A hand-placed node is a decision, and the polisher must never silently undo it. Detecting a manual placement without a new schema: a node whose stored position diverges from what tidyLayout of the current topology would produce, beyond a small ε, is inferred to be hand-placed — a pin. Pins are hard constraints (H5): the optimiser removes them from the free set and lays out only around them, routing wires to and from them like any obstacle. Proposed pin UX: dragging a node implicitly pins it (a small pin glyph appears); an explicit “re-tidy this node” unpins it back into the free set; a group can be pinned as a block. The invariant: the polisher improves the diagram the architect is building, it never fights the architect for control of a node. The exact affordance (implicit-on-drag vs an explicit pin toggle; whether a global re-tidy clears pins) is an open question (§6).

5.4 Performance at 50+ nodes, and fp32 score ties

  • Scale. The committed examples run to 23 nodes (cqrs-production-large, 4 groups); “best in industry” must hold at 50+ (design-for-growth: the socket before the load). The search cost is dominated by candidate scoring, which is exactly where the GPU earns its keep — TASK-81 measured the GPU flat in batch size and ~5× the CPU at N=10 000. So more nodes → a larger candidate space → a bigger batch, which is the GPU's best case. The wall-clock budget bounds the search regardless; at worst the architect gets the semantic pass (stage 1) result, which is already a large gain over Tidy with no search at all.
  • fp32 ties. Two candidates whose true scores differ by less than fp32 can resolve may tie or mis-rank on the GPU. That is precisely why fp32 only proposes: the CPU re-scores the top-k in fp64 and the router measures real bends, so ties are broken exactly and deterministically (the total-ordered tie-break already used by the router's priority queue extends to the finalist ranking). An fp32 tie is never applied; it is merely a slightly wider top-k sent to the exact stage.

6Benchmark — making “najlepsi” a measured claim

The owner's word is best, and best must be measured, not felt. So the deliverable includes a comparison harness that scores SDA's layout against the industry's engines on our own committed designs, with an objective table — crossings, bends, length, alignment, lanes — so the claim is falsifiable.

6.1 The harness

For each committed example, run four layout engines — raw Tidy, dagre, elkjs (layered), and SDA-ideal — then route all four with the same deterministic router and score all four with the same §2 objective. The engines that ship in the product are Tidy and SDA-ideal only; dagre and elkjs are test-only devDependencies of the harness (or their published outputs vendored as golden fixtures), never runtime deps — the “dependency-free layout” invariant holds. 7Non-goals

No new schemaThe applied layout is the same doc.layout {id→{x,y}} map Tidy writes today (AC#3). Pins and seeds are inferred or content-hashed, not new persisted structure — every existing project loads and lays out unchanged.
No required dependencyThe shipped runtime layout stays pure TypeScript — no dagre, no elkjs, no WASM in the product path. dagre/ELK appear only in the benchmark harness as test-only devDeps (or vendored golden output). The GPU module is lazily imported and optional; absent a device, the CPU path is silent and complete.
fp32 never decides the applied layoutThe GPU proposes a ranking; the CPU re-scores exactly and the router measures real routes. The layout the architect sees has always been proven in fp64 on real geometry (the three-layer discipline, §3.4).
No invented hierarchyWhere the topology is a mesh with no role gradient, the tool does not fake tiers — it falls back to a force-directed arrangement scored by the same objective with the structural terms zeroed (§4.2). A cyclic flow is carved by request classes, exactly as the engine already carves it.
No fighting the architectHand-placed nodes are pins the optimiser lays out around, never over (§5.3). The polisher improves the diagram being built; it never seizes a node the human placed.
No change to any computed numberLayout is beauty, not truth (§3.5). The optimiser moves pixels; it never touches load, latency, cost, availability or a verdict. A worse layout can be ugly, never wrong.
No engine domain leakageThe optimiser lives in app/presenter (already domain-aware, already the home of Tidy and the router). The domain-agnostic engine and the solver contract are untouched; the GPU geometry kernel is a new presenter-side shader, not a solver-contract change.
