Verification & Validation — the evidence that SDA does not lie

SDA is meant to be a system for serious people, competing with the best, that does not lie or feel flimsy. This document turns that intent into a discipline with a name the rest of serious simulation engineering already uses — V&V. Verification asks "are the equations solved correctly?" (the engine agrees with THEORY); Validation asks "are they the RIGHT equations for reality?" (the engine agrees with MEASURED systems). The organizing principle extends the project's hardest invariant — the tool must not lie — to the proof itself: a coverage matrix that makes every gap visible, so for every parameter and every metric we know exactly what is proven, to what error, against what source, and what is not. Green only where there is evidence. This is the credibility backbone, and it is honest about how much is not yet built.

About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas and the tool computes the design — throughput, latency, availability, durability, cost, feasibility — against declared requirements, live on every edit, and can run backwards to size or repair it. Under the canvas is a domain-agnostic engine: a typed-property graph, a fixpoint solver for the forward pass, a discrete-event simulator (DES) for tails and transients, and a native search for optimization. All cloud meaning is content (components as data). A tool that computes numbers a professional will stake a design on must be able to show its work — not just that it computed something, but that what it computed is right, and right about the real world. This document is the strategy for that proof. Definitions, stated positively:

Verification"are the equations solved correctly?" — the engine's output agrees with the closed-form theory it claims to implement (Little's law, Erlang-C, the M/M/c and P–K formulae) and with an independent solver on the same problem. It is a purely internal, mathematical check; no real system is involved. Established today by analytic anchors, differential tests, and the solver oracle harness (§2).
Validation"are they the RIGHT equations for reality?" — the engine's prediction of a measured system (a published benchmark, a real bill) lands within a stated error of the ground truth. It is the check verification cannot make: a perfectly-solved wrong model is still wrong. Seeded today by the calibration harness (§3).
anchoran analytic ground truth a capability is checked against — a textbook closed form (M/M/1 sojourn, Erlang-C wait), a metamorphic law (scaling load scales ρ), or a property (a flat generator is byte-identical to no generator). An anchor is verification: it needs no measured data, only mathematics.
calibration casea corpus entry: a real, measured system (a saved .sda.json model), the workload sweep, the tunable component-defaults treated as free variables, and the measured ground truth with cited source URLs — pure DATA under calibration/corpus/ (corpus.ts:77). A calibration case is validation: it holds SDA against reality and reports the residual.
coverage matrixthe generated table that crosses every capability/parameter, architecture and metric×regime with its evidence, one cell at a time. Each cell is validated (an anchor AND a calibration error %), verified (an anchor but no measured case yet), or unvalidated (an honest gap). It is generated from the code + the corpus so it cannot drift (§6).
residualthe error that REMAINS after the best fit — the structural gap no tunable can remove. It is the calibration harness's real output: not "we matched reality" but "here is exactly how far, and why" (fit.ts:1–5).
structural limita thing SDA fundamentally does NOT model and will not — deep allocator/GC economics, detailed consistency protocols, network-partition dynamics, cache-eviction internals. Flagged permanently unmodeled so the fidelity report is honest about its boundaries, not silent (§9).
fidelity reportthe published deliverable: per-capability (anchor ✓ + calibration error % or UNVALIDATED), per-metric×regime, the residual/structural-limits map, and a single honest headline — "X of Y capabilities calibrated, aggregate fidelity Z %, N documented structural gaps" (§7).

1The idea — V&V is the credibility backbone, and honesty is its rule

Every discipline that stakes real decisions on a simulation — computational fluid dynamics, nuclear reactor safety, network capacity planning — separates two questions and never conflates them. Verification: did we solve the equations right? Validation: are they the right equations for the real world? SDA already lives half of this instinctively (a differential-tested engine); this document names it, audits it, and builds the missing half into a discipline you can watch grow as a number.

The reason to adopt the V&V frame explicitly is that it is the vocabulary of the audience SDA wants to earn — the staff architect who has used a load tester and read a capacity study, and who trusts a tool that can say "this number is verified against Erlang-C and validated against TechEmpower to within 1 %" over one that just renders a confident chart. It is also the discipline that keeps the team honest: the two questions have different failure modes and different fixes, and mixing them is how simulators quietly start lying. A model can pass every property test (verified) and still predict a system that does not exist (invalid); a model can match one measured system (validated once) and be solved by a buggy solver on the next (unverified). You need both, tracked separately.

The organizing principle: the tool-must-not-lie invariant extends to the proof.

SDA's charter forbids inventing infra numbers — uncertainty is a value (unknown), never a guess. This document applies the same rule one level up, to validation itself. The measure of solidity is not "how many green checks" but: for every parameter and every metric, do we know exactly what is proven, to what error, against what source, and what is NOT? The instrument is a coverage matrix where a cell is green only where evidence exists — an anchor for verification, a measured residual for validation — and every remaining cell reads UNVALIDATED in plain sight. A visible gap is not a failure of the tool; a hidden gap is. Solidity is the honesty of the map, not the absence of white space on it.

2Verification today — the audit ("are the equations solved correctly?")

SDA is already strongly verification-anchored, in three distinct engines each with its own oracle. This section audits exactly what is anchored — and, in §2.4, exactly what is not. Every claim is a named file:line at HEAD adc55bf.

2.1 The simulation oracle — closed forms are the ground truth the DES is checked against

The continuous, analytic half of the time engine is a file of pure, textbook closed forms — and it exists expressly as "the oracle the discrete-event simulator is validated against" (engine/sim/src/analytic.ts:2–3). It implements M/M/1 (analytic.ts:31), M/M/c via the numerically-stable Erlang-B recursion lifted to the Erlang-C wait probability (analytic.ts:50–63), the Pollaczek–Khinchine mean wait for M/G/1 parameterised by the service SCV (analytic.ts:70–74), and Little's law (analytic.ts:77–79), with cited sources (Erlang 1917; Pollaczek 1930 / Khinchine 1932; Little 1961; Cooper 1981) at analytic.ts:10–11. The DES itself (engine/sim/src/des.ts) states the differential invariant in its header — "validated differentially against the closed forms (M/M/1, M/M/c, Little, P-K) within confidence tolerances" (des.ts:15) — and the content-side analytic twin the canvas actually renders (content/sda/src/queueing.ts) calls the same mmc (queueing.ts:187) and asserts "@invariants Agrees with the DES within tolerance" (queueing.ts:1–17). Because both the analytic twin and the DES read one shared station definition (graph-read.ts:91, queueStation), the two cannot silently drift apart.

2.2 The solver oracle harness — the executable specification (the verification template)

The backward capabilities — optimize, repair, explain-infeasible, enumerate — cannot be checked against a hand-written answer table (that would only re-encode a human's belief). Instead engine/solver-contract/src/harness/ is an executable specification (framed at docs/design/solver-contract.html:49,74 — "'adapter X implements capability C' is not a claim in a README — it is a green test run"). It is the template every future verification effort should copy, and it has four clean seams:

The oraclethe incumbent adapter (native MiniZinc/COIN-BC MIP + clingo/ASP), whose answer is taken as certified (oracle.ts:17–22), plus two engine-independent recomputations: a whole-graph objective total by algebraic inversion of the cumulative out-cells (oracle.ts:151–179) and an anti-lie SLO re-check that re-runs evaluate on the solved assignment and checks every band within ε (oracle.ts:197–231). The candidate solver is passed in as a parameter, so oracle vs candidate is "just which adapter a caller binds" (oracle.ts:31–35).
The generatora seeded, deterministic generator of opaque typed-property graphs (mulberry32, no Date.now/Math.random; each instance carries the seed that produced it — generator.ts:122–137,189), over three topologies (chain/fan-out/fan-in), two regimes (sat/unsat, true by construction against the design's own analytically-known ceiling — generator.ts:184–185,660–691), and 17 named coverage axes (generator.ts:162–179). Default batch ≈ 54 numeric + 6 enumerate cases (generator.ts:1254–1260).
The metamorphic lawsproperties whose effect on the optimum is known a-priori, so they catch a bug both solvers share: scale equivariance — scaling every rate by k scales the optimum by exactly k (metamorphic.ts:46,81–91); permutation invariance (metamorphic.ts:122–126); monotone tightening — raising an SLO floor never lowers the optimum, and feasibility is downward-closed (metamorphic.ts:143–156); repair coherence — a repair from a feasible point is zero-distance, and repair's L1 never exceeds the distance to the from-scratch optimum (metamorphic.ts:166–186). Plus determinism-under-seed and monotonicity in the harness core (harness.ts:186–218).
The comparator + CIequivalence on the honest surface — objective value + SLO satisfaction for optimize, total L1 for repair, the shortfall set for explain, the exact selection set for enumerate (harness.ts:94–119) — with a per-instance time budget (a correct-but-slow candidate FAILS, harness.ts:73–76) and a one-paste repro command per divergence (harness.ts:129–132). It runs as standing tests with seed rotation (SDA_HARNESS_SEED offsets each base seed into a disjoint region) and a DEEP lane (~707 tests, ~85 s) — native/index.test.ts:65–75 — plus a live referee that throws on any incumbent-vs-native divergence (native/index.test.ts:181–200).

2.3 The engine-algebra property layer — the forward pass over adversarial designs

Below the solvers, the forward fixpoint and content algebra are fuzzed over random and adversarial designs with fast-check: the named engine laws — fixpoint converges, no NaN masquerades as converged, deterministic and order-independent, load-monotone, async decouples wait — hold over hundreds of generated real designs (content/sda/src/laws.property.test.ts:35–107, seed pinned at :15), and the robustness suite proves instantiate/evaluate never throw on cycles, self-loops, orphans and pathological configs (robustness.property.test.ts:105–179). The CPU-station primitive carries a 300-run byte-identity property: any node with no CPU config keeps exactly its original station (cpu-station.e2e.test.ts:41–52).

3Validation today — the calibration seed ("are they the RIGHT equations?")

Verification cannot tell you the model is right about the world — only that it is solved correctly. The calibration harness (@sda/calibration, TASK-93) is the seed of the validation half: a DATA corpus of measured ground-truth systems plus a deterministic fitter that predicts each with SDA's own engine, fits the tunable component-defaults, and reports the residual — the structural gap no tunable can remove (calibration/src/index.ts:1–4).

3.1 How it works — predict, fit, report the residual, guard against over-fit

Corpus (DATA)each entry is a JSON file under calibration/corpus/: a saved .sda.json model, the workload sweep, the tunables (a component default treated as a free variable with a search range, or a pinned sourced hardware fact), and the measured ground truth with cited source URLs. A datapoint may be measured: null — unpublished — in which case the harness predicts and reports but does NOT score it (the tool must not invent a number to grade against) — corpus.ts:52–85.
Predictthe SAME content path the canvas uses: instantiate → evaluate → nodeQueues (analytic M/M/c) → responseLatency, plus a seeded DES corroboration for the tail (predict.ts:1–5). The capacity ceiling is read by linear extrapolation from a sub-saturation probe (predict.ts:74–87); a latency share is the numerator hop's sojourn over the operation's end-to-end response (predict.ts:94–105). Injection-resistant: a config key from the corpus is refused if it is a prototype-pollution vector (predict.ts:33–36).
Fit & residuala deterministic coordinate descent over a log-spaced grid (no RNG, fixed order) minimises the RMS of signed relative errors over every scored point (fit.ts:41–132). Two entries declaring the same (selector, key) SHARE one free variable, so a joint fit can be over-determined — which is exactly how an irreducible structural residual is surfaced (corpus.ts:34–48).
Leave-one-outthe honest over-fit guard: fit on the rest, report the error on the held-out entry, and state plainly whether the held-out tunables were actually constrained by the fold-in set (a genuine out-of-sample test) or fell back to catalog defaults (fit.ts:178–204).
Report, never applythe harness READS, FITS and REPORTS but NEVER mutates a shipped default — applying a recommendation is a separate, owner-reviewed step, emitted as PROPOSED-DEFAULTS.json (report.ts:1–5,199–211). pnpm calibrate writes the report; pnpm calibrate:check adds a CI fidelity-regression gate against a pinned baseline (run.ts:53–61, baseline.ts:27–37).

The calibration finding — validation drove a real engine primitive.

The TechEmpower single-vs-20-query pair originally shared one db.postgres service time and could not fit both ceilings — a ~6 % irreducible residual that WAS the framework's per-request CPU, a resource the model could not express. Adding a separate CPU station (cpuCores + cpuTimePerRequestMs, an M/M/cores tier, graph-read.ts:48) let single-query bind on the framework CPU (~104.5k) and 20-query on the DB (5,858) independently — collapsing the aggregate from 5.2 % to 0.6 % (baseline.ts:17–34). This is validation working as intended: the residual pointed at a missing primitive, and closing it was physically grounded (the CPU width is the sourced 28 hardware threads), not an over-fit. Externally corroborated — TechEmpower later lifted the single-query result 25× with framework-side fixes, never touching the DB.

Honest limit of the seed — three systems is a beginning, and leave-one-out says so.

The full-fit aggregate (0.6 %) is excellent but the out-of-sample generalization is 12.1 % (CALIBRATION-REPORT.md:10): holding out TechEmpower single-query reproduces a ~+13 % over-prediction because that point is the only one constraining the framework-CPU free variable, and DeathStarBench's tunable is disjoint from the TechEmpower fold-in set entirely (harness.test.ts:67–86). This is not a flaw in the harness — it is the harness being honest that three architectures with mostly-disjoint tunables cannot yet cross-validate each other. It is the single strongest argument for the corpus roadmap (§8): validation solidity is bounded by architecture diversity, and right now that diversity is thin.

4The organizing principle — three coverage axes

To make "what is proven, and what is not" a computable fact rather than a feeling, the claim surface is decomposed into three orthogonal axes. Every unit of evidence lands somewhere in their product, and every empty region is a visible gap.

4.1 Axis (c) in full — metrics × regimes, and where the evidence is thinnest

A capacity model that matches reality at ρ=0.3 tells you little about ρ=0.95; the tail explodes near the knee, not below it. So each metric is tracked in three regimes: below-knee (ρ ≲ 0.7, comfortable), at-knee (ρ ≈ 0.8–0.95, where queueing bites), and past-saturation (ρ ≥ 1, where the honest answer is drop/overflow/Infinity). 6The coverage matrix — representation, data model, and generation (no drift)

The matrix is not a hand-maintained spreadsheet — that would rot within a week. It is a generated artifact, walked from the code and the corpus by a script, exactly as docs/ALGORITHMS.md (54 algorithms) and docs/FEATURES.md (20 features) are generated from @algorithm/@feature headers and asserted byte-identical by scripts/generate-catalogs.test.ts. That proven anti-drift mechanism is the template.

6.1 The cell data model

Each cell is one (capability-or-parameter × metric × regime) coordinate with its evidence. It is pure data, so the report, the freshness test and the CI gate all read one structure:

interface CoverageCell {
  capability: string;         // e.g. 'queueingTail:p99' | 'cost:egress' | 'optimize'
  metric: MetricKind;         // 'throughputCeiling'|'tail'|'cost'|'availability'|'bottleneck'|'transient'
  regime: 'below-knee' | 'at-knee' | 'past-saturation';

  verification: {
    anchored: boolean;
    oracle: 'analytic-closed-form' | 'solver-incumbent' | 'metamorphic' | 'property' | 'none';
    test:  string | null;     // file:line of the @anchor test that proves it, or null
  };
  validation: {
    calibrated: boolean;
    corpusEntry: string | null;   // which calibration/corpus/*.json entry
    errorPct:   number | null;    // post-fit residual %, or null if UNVALIDATED
    source:     string | null;    // the ground-truth URL
  };

  // derived, never authored — the single source of a cell's colour:
  status: 'validated'   // anchored AND calibrated (errorPct is a number)  -> GREEN
        | 'verified'    // anchored, not yet calibrated                     -> AMBER
        | 'unvalidated';// neither, OR a documented structural limit        -> GREY
}

The rule is mechanical and unforgeable: status = 'validated' iff verification.anchored && validation.errorPct !== null; 'verified' iff anchored but not calibrated; else 'unvalidated'. A cell can never be green on a promise — only on a residual number with a source URL behind it. A structural limit (§9) is a first-class 'unvalidated' that additionally carries unmodeled: true, so the report can distinguish "not yet" from "never".

6.2 How it is generated — three walks, one emit

Walk 1 — the claim surfaceenumerate the rows: the 6 capabilities (capability/index.ts), the 8 families (instructions.ts) and the 26 input knobs (registry.ts), crossed with the 6 metrics × 3 regimes. This is axis (a) × axis (c), read straight from the code so a new knob appears as a new UNVALIDATED row automatically.
Walk 2 — the anchorscollect the verification evidence from @anchor headers on the anchor tests (a new tag, sibling to @algorithm): each header names the capability/metric/regime it proves and the oracle it uses. A capability with no @anchor is emitted verification.anchored: false — the matrix cannot claim an anchor that no test carries.
Walk 3 — the residualsread calibration/corpus/*.json + the fitted residuals from the calibration report structure (report.ts:31–38): each scored ground-truth point contributes validation.errorPct, corpusEntry and source to the cell it measures.
Emitscripts/generate-coverage.mjs writes docs/COVERAGE.md (the matrix) deterministically; a freshness test asserts the committed file is byte-identical to what the walks produce, so a capability added without an anchor-or-flag fails CI rather than passing silently — the same discipline generate-catalogs.test.ts already enforces.
Why generation is the whole point.

A hand-kept matrix drifts the moment someone adds a knob and forgets the row; then the map lies, which is worse than no map. A generated matrix inverts the incentive: the only way to make a cell green is to write the anchor test and the calibration case that the walks will find. The matrix becomes a forcing function — you cannot ship a capability into it without either the evidence or an explicit, visible UNVALIDATED. The map cannot lie because nobody writes the map by hand.

7The fidelity report — the published deliverable

The coverage matrix is the ledger; the fidelity report is its published face — the artifact that gives an architect an honest sense of how solid the tool is, in one glance and then in full detail. It extends the existing CALIBRATION-REPORT.md from "three residuals" to a whole-tool fidelity picture, generated by pnpm calibrate + pnpm coverage, living beside its inputs under docs/ (a candidate docs/FIDELITY.md, or an HTML twin for the deliverable surface).

7.1 The four parts

The headlineone honest sentence: "X of Y capabilities calibrated · aggregate fidelity Z % · N documented structural gaps · M architectures in corpus." Today that reads, truthfully: 2 of 8 modeling families calibrated · aggregate fidelity 0.6 % (over 3 scored points) · out-of-sample 12.1 % · 3 architectures · N structural gaps documented. The headline never averages away the gaps — the unvalidated count sits right next to the fidelity %.
Per-capabilitya row per capability/family: anchor ✓/✗ (with the oracle), the calibration error % or UNVALIDATED, and the corpus entry + source. Reusing the report's existing per-entry table shape (report.ts:144–169), extended left with the anchor column and down with the un-calibrated families.
Per-metric×regimethe §4.1 grid, generated — six metrics × three regimes, each cell coloured by status, so the white-spaces (cost/availability/transient past-saturation) are literally visible as grey.
The residual & structural-limits mapthe honest tail: the fitted residual per scored point (the structural gap that remains — report.ts:191–195), the leave-one-out generalization, and the §9 permanent-unmodeled list. This is where "we know exactly what is NOT proven" is written down.

7.2 Format & provenance discipline

Generated, deterministic, reproducible with one command (the @algorithm/pricing-source idiom already in the codebase). Every fidelity number carries its basis label the way SDA already labels every rendered number — measured (transient) vs analytic (quasi-static) (two-tier.ts:61, time-sweep.ts) — so no fidelity figure is ever shown without saying what evidence produced it. The report RECOMMENDS but never applies (a shipped default changes every design's numbers — report.ts:1–5); the fidelity report inherits that firewall.

9Known structural limits — what SDA will NOT model, permanently

Honesty about boundaries is what keeps the fidelity report from being silently wrong at the edges. These are not gaps to close — they are things SDA deliberately does not model, flagged unmodeled so the report is explicit about where its predictions stop meaning anything. The calibration report already names several (CALIBRATION-REPORT.md:70: "the DEEP CPU economics … stays deliberately out of domain, flagged unmodeled").

Not modeled (permanent)Why it is out of domain
Deep allocator / GC / lock-contention economicsSDA models a tier as an M/M/c station with a service time and a core count; it does not model heap pressure, GC pauses, false sharing or lock convoys. The fitted per-request CPU time absorbs their average, but the dynamics (a GC pause that spikes p99 under memory pressure) are outside the queueing abstraction — named unmodeled at CALIBRATION-REPORT.md:70.
Detailed consistency / consensus protocolsSDA propagates a qualitative guarantee token over a lattice (engine/core/src/lattice.ts) — it says "this flow's end-to-end consistency is X and here is the weakening hop." It does not simulate Raft/Paxos round-trips, quorum read/write latencies as a function of replica placement, or the CAP trade under partition. Guarantees are a type, not a timed protocol.
Network-partition dynamicsAvailability is a series/parallel product with documented uplifts per deployment mode (behaviors.ts:184–195); it is not a partition-and-heal simulation. Split-brain, failover storms and partial-partition behaviour are not modeled — the tool answers steady-state availability nines, not partition transients.
Cache-eviction internalsA cache can be modeled as a hit-ratio that scales effective service time (roadmap #4), but LRU/LFU/ARC eviction dynamics, working-set shifts and cold-cache stampedes are not simulated. The hit ratio is an input, not an emergent property of a modeled eviction policy.
Two in-series resources as a true tandemWhere a tier binds on the MIN of two stations (DB vs framework CPU), SDA takes the binding minimum, not a full tandem-network interaction — explicitly flagged at CALIBRATION-REPORT.md:70 as out of domain. Adequate for capacity; not a model of coupled queueing between the two.
The rule.

A structural limit is a permanent UNVALIDATED · unmodeled cell — it never turns green, and the fidelity report lists it under "what no tunable can remove, ever." Where a design genuinely needs one of these, SDA's honest output is unknown, never a fabricated number — the same line the tool already holds everywhere. Making these explicit is what lets the rest of the report claim solidity without over-claiming reach.

10The discipline — the CI gate, the landing rule, and the over-fit guards

A strategy is only as real as the routine that enforces it. Three mechanisms turn V&V from a document into a standing property of the codebase.

The CI fidelity gatepnpm calibrate:check already fails on any residual regression beyond 1.5 pp against the pinned baseline (run.ts:53–61, baseline.ts:46–56), and the standing test harness.test.ts runs it under pnpm test. Phase 1 adds pnpm coverage's freshness assertion to the same gate: fidelity cannot silently regress, and the matrix cannot silently drift.
The landing ruleevery new capability lands with (a) an @anchor-tagged analytic anchor test AND (b) a calibration case — OR an explicit UNVALIDATED flag in the matrix. Because the matrix is generated (§6), (a)-or-flag is mechanically enforced: an un-anchored capability that is not flagged fails the freshness test. This is the forcing function that keeps the map complete.
The over-fit guardsfour, layered: (1) train/test split — leave-one-out is reported alongside the full fit, and the honest 12.1 % is not hidden (fit.ts:178–204); (2) fit, don't apply — tunables are recommended in PROPOSED-DEFAULTS.json for owner review, never written into shipped content (report.ts:199–211); (3) residual-as-output — the goal is the residual, not a match, so a suspiciously-perfect fit on few points is read as under-constraint, not success; (4) architectures > free variables — the corpus roadmap (§8) is sized so the number of distinct architectures grows faster than the number of shared free variables, keeping the fit over-determined. Guard (4) is the one the current corpus fails, and the one the roadmap exists to fix.

11The three hardest tensions

11.1 Over-fit with few systems — the coverage paradox

Every new capability that needs a calibration case tends to introduce a new free variable (a new service time, a new pool size). If the corpus grows one architecture per new tunable, the fit stays exactly as under-determined as it is today — three points, three shared free variables, 12.1 % out-of-sample. The tension is that breadth (more families calibrated) and depth (each family over-determined) pull against each other with a fixed research budget. The honest resolution is guard (4) as a hard rule: a family is not marked validated until at least two architectures share its tunable, so the fit for that tunable is over-determined and the leave-one-out for it is a genuine out-of-sample test. Until then it is verified with a single sourced point — anchored, plausibly-calibrated, but honestly not cross-validated. The matrix distinguishes those two states so "validated" always means "over-determined," never "matched once."

11.2 Retrievable ground truth — most systems publish ratios, not curves

The deepest practical obstacle is that real published systems rarely give a clean absolute p99-vs-QPS or throughput-ceiling curve. DeathStarBench — the standard benchmark — publishes relative shares and ratios, so the corpus honestly marks its mixed-workload ceiling measured: null and scores only the one clean latency-share anchor (deathstarbench-social.json:38–41). This is not a defect to engineer around; it is a fact of the source material. The resolution is to make measured: null a first-class, respected state — the harness predicts and reports it for context but refuses to score it (the tool must not invent a number to grade against) — and to gate corpus growth on retrievability, preferring benchmark suites that publish absolutes (TechEmpower, TFB cached, redis-benchmark, OpenMessaging) over prestige systems that publish only prose. Speculative entries still earn inclusion for topology diversity, but never inflate the scored fidelity number.

11.3 Capabilities with no measurable real-world analog

Some capabilities have no "measured system to fit" in the queueing sense, and forcing a fit-error onto them would itself be a lie. Cost is deterministic: cost = driver × sourced unit price (behaviors.ts:342) — there is no residual to fit, only a source to keep fresh; "validation" for cost means the unit price matches the vendor's published rate today, not that a regression converged. Availability is a series product against published SLAs (reliability.ts); measured end-to-end availability of a real DR setup is rarely published, so it is validated against the SLA nines the vendor commits to, not against observed incidents. Delivery guarantees are qualitative tokens over a lattice — there is no numeric ground truth at all. The resolution is that the matrix carries a fourth evidence kind beside anchor/residual: sourced — a cell whose correctness is a cited, current source rather than a fitted error. It is honestly not the same as a calibrated residual, and the fidelity report must not blend them into one "fidelity %." A cost cell is green when its price source is current; a queueing cell is green when its residual is small; the report says which is which.

12Phasing — framework first, then corpus in waves you watch grow

Ordered so the smallest honest slice ships first and each phase makes the next cheaper. Effort is stated honestly per phase.

  1. The V&V framework. P1 The coverage-matrix data model (§6.1) + scripts/generate-coverage.mjs emitting docs/COVERAGE.md, freshness-asserted like the algorithm catalog; the generated fidelity report (§7) extending CALIBRATION-REPORT.md; and an analytic-anchor pass filling the highest-value verification gaps (§2.4) — an evaluate differential oracle, a P–K test at intermediate SCV, a realCumulativeLatency end-to-end DES differential. Effort: moderate — mostly wiring the walks over structures that already exist (registry, capability index, corpus, report), plus ~3 anchor tests. No new solver math. Gate: the matrix generates green-where-evidence and grey-everywhere-else; CI fails on drift or fidelity regression.
  2. Corpus Wave A — the highest-value white-spaces. P2 Roadmap #5 Kafka (streaming/backlog), #6 Redis + #7 Cassandra (datastore capacity + tail), #4 cache hit-rate. Each: research → model → retrieve ground truth → land with its residual. Effort: high — this is real modeling + sourcing work, ~1 system per focused slice. Gate: each family flips from UNVALIDATED to verified or, with two architectures, validated; the fidelity headline's calibrated-family count rises.
  3. Corpus Wave B — cost, availability, serverless. P2 #8 DynamoDB (documented ceilings + bill), #10 Lambda (concurrency + cold start), #11 a published bill, #12 a DR reference. Introduces the sourced evidence kind (§11.3) for cost/availability. Effort: high; some quantities land measured: null honestly. Gate: the cost/availability past-saturation white-spaces stop being empty; the fidelity report distinguishes sourced from fitted.
  4. Corpus Wave C — fan-out, CQRS, multi-region diversity. P3 #9 Discord fan-out tail, #13 CQRS, #14 managed web stack — chosen for topology diversity to drive the leave-one-out generalization down. Effort: high, and partly speculative — scored where retrievable, included for the over-fit guard where not. Gate: leave-one-out RMS falls materially below today's 12.1 % as the fold-in set diversifies.
  5. Each wave ends with a published fidelity report. every wave Regenerate docs/FIDELITY.md; the headline's "X of Y calibrated · aggregate Z % · N gaps · M architectures" is the number the owner watches grow. Solidity becomes a tracked metric, not a claim — the point of the whole exercise.

13Open questions for the owner

1. Where does the fidelity report live? A new generated docs/FIDELITY.md (+ an HTML twin for the deliverable surface), or fold the coverage matrix and per-metric grid into the existing CALIBRATION-REPORT.md? The former keeps calibration (validation) and coverage (verification+validation) as separate artifacts; the latter is one file. Recommendation: separate — the fidelity report reads from the calibration report, so it should not be it.

2. Is a single "aggregate fidelity %" the right headline, given it averages heterogeneous metrics? A capacity residual and a latency-share residual are not the same kind of error, and cost has no residual at all. Options: (a) one aggregate % over queueing-family scored points only, with the other families as an explicit calibrated-fraction; (b) a small vector (capacity X %, tail Y %, N sourced, K unvalidated). Recommendation: (b) — a vector is more honest than a blended scalar, and the tool's whole ethos is not blending a violation away.

3. Does a sourced cost/availability cell count toward "calibrated"? §11.3 argues it is a distinct evidence kind — green when its source is current, not when a fit converged. Confirm it appears as its own column ("sourced") rather than being counted in the calibrated fraction, so the headline never implies a fit that did not happen.

4. What is the corpus growth commitment? A fixed wave of 3–4 systems per milestone (predictable solidity growth), or opportunistic as retrievable ground truth surfaces (honest but lumpy)? The over-fit guard (§10) wants steady architecture growth; the retrievability tension (§11.2) makes it lumpy. Recommendation: commit to the family-coverage order (Wave A→C), let the rate float with retrievability.

5. Do speculative (measured: null-heavy) entries belong in the corpus? They add topology diversity for the leave-one-out guard but score nothing. Include them (they shrink the 12.1 % by diversifying the fold-in set) or keep the corpus scored-only (cleaner, but slower to over-determine)? Recommendation: include, clearly badged, never counted in the scored aggregate.

6. CI budget for the growing corpus. calibrate:check runs the full fit each CI; the standing guard already turns DES corroboration off for speed (harness.test.ts:13–15). With 14 systems and DES tails, is a fit-only gate in CI + a full report (with DES) nightly the right split? Recommendation: yes — mirror the solver harness's DEEP-lane pattern (a fast standing gate, a heavy nightly).
