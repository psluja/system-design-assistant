The Assumption Model

One assumption space, three views. The owner rejected a tool that demands a demand — “what counts is what I CAN do, what I WANT, and the RESULT of the simulation.” Then he found the missing shape himself: scenarios — named worlds (typically pessimistic / real / optimistic) that hold every assumption across every node, authored scenario-first, and created for you by default with values derived from the design's own capacity. This document unifies three answers over one set of parameters: the envelope (the boundary — the default answer, no assumptions needed), a scenario (a named point — absolute bill, utilization, verdicts), and a range / Monte Carlo (a distribution around a point — confidence). Same space, three views.
SDA design document · 2026-07-04 · the assumption model (envelope · scenarios · ranges) · status: for owner review — not started
About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas and the tool computes the design — load, latency, availability, cost, feasibility — against declared requirements, live on every edit. Underneath, SDA is a domain-agnostic engine that turns the drawing into a typed-property graph and a family of solvers; a content pack layers all cloud meaning on top as data. Every number the tool computes rests on some input the architect (or a catalog default) supplied. This document is about those inputs that are genuinely uncertain about the world — how much traffic will arrive, what fraction of reads hit the cache, how long a query really takes — and how the tool answers without forcing the architect to invent numbers, then sharpens the answer as real numbers arrive. It defines a single assumption space and the three views that read it.

assumptionan input that expresses a belief about the outside world, not a fact about the design: the offered traffic, a cache-hit ratio, a per-request duration. Formally, a config value whose role is fact-assumption (below). Distinct from a limit (a ceiling the design commits to) and a promise (an SLO the design must meet).
the role axisthe shared axis on which every node quantity is classified (one-form-per-kind): fact-assumption (a belief about the world) · resource-limit (a ceiling this design sets) · computed (a derived result) · promise (a declared SLO). The role decides how a quantity is presented — and, here, whether a scenario may override it.
capacity envelopethe computed boundary of the assumption space: the maximum sustained demand at each origin under which all declared SLOs still hold, plus the breaking order (which node/promise fails first as load grows) and the latency knee. Needs no declared demand — it is the default answer (doc: this document §3; TASK-85).
scenario (named world)a named point in the assumption space: a name plus a set of overrides node.key → value on the fact-assumption inputs. Because the point is concrete, the answers become absolute: a monthly bill, an actual utilization, a per-scenario verdict. Typically authored as a trio: pessimistic / real / optimistic.
range · Monte Carloan assumption declared as an interval, not a point ({lo, hi} uniform or {lo, mode, hi} triangular). Sampling many draws turns every conclusion into a distribution — a confidence (“the latency SLO holds in 97% of scenarios”). Shipped today as content's uncertainty module.
provenancewhere a number came from, derived mechanically and shown as a badge: documented (a sourced quota/SLA), estimate (a credible catalog default), architect (set by hand), default (an untouched default). This document adds one value: derived — computed from the design's own envelope, awaiting a measured replacement.
SLO / band · verdict · ρan SLO is a declared requirement on one property of one place, internally a band. A verdict is the honest read-back: ok / warning / violation / unknown. ρ = offered load ÷ capacity; the tool sizes to ρ ≤ 0.8 (the headroom convention) so a queue never runs to the ρ→1 knife-edge. Uncertainty is a value, never a guess.

1The workflow this serves — no demanded demand, then scenarios

This design has one origin and it is the owner's, stated on 2026-07-04. First the rejection: a tool must not open by demanding a load before it will say anything — “what counts is what I CAN do, what I WANT, and the RESULT of the simulation — where in that is there room for assumptions?” Then, from the same conversation, the shape that does belong: not a single mandatory number but named worlds. An architect thinks in a pessimistic case, a realistic case, and an optimistic case; each world is not just a traffic figure but a full set of beliefs — demand, cache-hit ratio, per-request durations — held together and named. The tool should let him author those worlds, and, because “architects are lazy — let’s do everything for them,” it should create the trio by default with sensible values already filled in. His verdict on that last idea: “this could be a hit.”

Two requirements fall out immediately, and the rest of this document is their reconciliation:

  • The default answer needs no assumption. With nothing declared, the primary computed answer is the capacity envelope — how far this design can be pushed before an SLO breaks, and what breaks first. The simulated surfaces are never empty-and-mute (TASK-85).
  • Assumptions, when wanted, are named worlds — and pre-filled. A scenario is an optional refinement lens: declare a world (or accept the auto-created trio) and the curves collapse to absolutes for that world. The tool authors the trio itself, from the envelope, with honest provenance — never invented market numbers.
The one-sentence thesis. The fact-assumption inputs of a design span an assumption space; the tool reads that space three ways — the envelope is its computed boundary (the default, no point needed), a scenario is a named point in it (absolute answers), and a range / Monte Carlo is a distribution around a point (confidence). One space; three views; each honest about what it needs.

2The role axis — the boundary drawn mechanically meta-model addition

Before the three views, one structural decision governs the whole model: which inputs a scenario may touch. The answer is not a hand-picked list — it is read off a single classification the project already needs for a deeper reason, and this is where the “originRps confusion” that started this work gets fixed at the root.

2.1 One form per kind: every node quantity carries a role

The owner's consistency law (memory one-form-per-kind) is that same-kind things must share one form of naming, description and presentation, so their differences stand out as values on a shared axis — “you don't describe a dog biologically and a cat as a companion; then you can't see the difference.” Applied to node quantities, the shared axis is a role: every quantity is one of four kinds.

The root cause of the originRps confusion, named. The owner hit it directly (memory one-form-per-kind): a fact-assumption (offered load) was rendered in the visual and naming form of a limit — so a belief about the world looked like a ceiling the design sets, and the difference that matters (this is uncertain about reality; that is a design commitment) became invisible. The role axis is the fix: it makes the kind a first-class value every surface renders by, and it is the same axis that mechanically answers “what may a scenario override?” — only role fact-assumption.

2.2 Status of the role axis, honestly not yet in code

The role axis is proposed, not built. Today the registry (content/sda/src/registry.ts) tags each key with an aggregate algebra and a kind of 'input' | 'derived' — a coarse computed-vs-authored split, not the four-way role. The richer classification exists so far only as data in two places: the design-doc's assumptions register already derives a provenance per input mechanically (content/sda/src/doc-model.ts line 139, configProvenance 535–540), and the sensitivity matrix already declares each knob's expected direction as reviewable data (content/sda/src/sensitivity-matrix.test.ts line 80). Introducing role as an explicit registry field — the “role on registry keys that surfaces render by” the memory calls for — is the meta-model step this design assumes and TASK-85 AC#4 commissions (“one-form-per-kind role axis applied to all node quantities”). Everything below is written to that axis; where a claim depends on it, it is flagged.

3View one — the envelope (the default answer) TASK-85

With no scenario declared, the tool answers with the boundary of the assumption space along the demand axes: the maximum sustained load each origin can carry with every SLO still green, which promise breaks first as load grows, and where the latency knee sits. This is a computed answer that needs no assumption — so the simulated surfaces are never empty.

3.1 What the envelope is, and how it is computed

The envelope is a maximisation: free the origin demand, hold the design fixed, and push demand as high as the SLOs permit. The machinery exists. The native solver evaluates in ~1 ms and finds optima by per-knob inversion on the monotone capacity/flow class (engine/solver-contract/src/native/search.ts §“the optimum is found by per-knob inversion”), driven by an Objective { node, key, direction: 'min' | 'max' } (.../conformance/corpus.ts line 27). Max-demand-subject-to-SLOs is exactly optimize with the demand key freed and direction: 'max'. The breaking order is a by-product: as demand rises, the first SLO whose inversion binds is the first break, the next is second, and so on. The knee is read from the load sweep, which already scales the design's origins and reads the queueing-aware end-to-end latency — today over [0.5, 0.75, 1.0, 1.25, 1.5]× nominal (content/sda/src/sweep.ts line 27, SWEEP_FACTORS), generalised here to sweep up to the envelope edge. The DES confirms the envelope edge (percentiles at the critical point), as it confirms every other queue-dynamics quantity.

Tension resolved — the envelope inverts the “no-cheating” rule, and that is correct. The backward-search discipline (memory search-tunables-no-cheating) fixes the workload on a source and makes throughput raise-only, because optimize(cost, min) with a free workload would “minimise cost” by serving no traffic (set the load to ~1). The envelope does the opposite: it frees the demand and maximises it. There is no cheating to prevent — a maximiser cannot game the objective downward, and the design (limits, sizing) stays fixed. So the two coexist cleanly: the no-cheating rule governs optimize(cost) where demand is a fixed requirement; the envelope is the distinct inverse problem where demand is the free objective. This is why demand carries role fact-assumption and not resource-limit: only a fact-assumption is legitimately free-and-maximised here.

3.2 The honest boundary the envelope draws

The envelope answers relative questions with no assumption: how far can this design be pushed, and what gives first. It cannot answer absolute questions — a monthly bill for pay-per-use compute, an actual utilization, a %-of-worlds confidence — because those require knowing where in the envelope you actually sit. That is the honest line stated everywhere (TASK-85 AC#2): absolute cost / utilization / confidence require a declared demand; envelope answers never do. This is not a limitation to apologise for; it is the two-view split made visible, and it is exactly what motivates view two.

4View two — a scenario is a named point the owner's discovery

A scenario names a concrete world: pick a point in the assumption space and every curve collapses to a number for that world. This is the refinement lens the owner described — and its model is small, because the contract already has the shape.

4.1 The model — a named world over the existing Scenario seam

The solver contract already defines a Scenario: a set of numeric overrides keyed by the "node|key" coordinate, evaluated in a batch by the EvaluateBatch capability (engine/solver-contract/src/capability/evaluate-batch.ts lines 22–24, 44). It is how Monte Carlo already substitutes drawn values onto a fixed design at ~1 ms per evaluation (content/sda/src/uncertainty.ts). A named world is precisely that contract Scenario, promoted to a first-class authored object: a name, plus overrides restricted (at authoring time) to role fact-assumption keys.

// A named world — pure DATA on the project document (export schema 6 → 7, additive).
interface AssumptionScenario {
  readonly id: string;                       // "pessimistic" | "real" | "optimistic" | any name — the canvas label
  readonly name?: string;                     // optional friendly name (mirrors RequestClassDecl.name)
  readonly overrides: readonly ScenarioOverride[]; // only role=fact-assumption keys; DELTAS over the base layer
}
interface ScenarioOverride {
  readonly node: NodeId;
  readonly key: string;                       // a fact-assumption registry key (originRps, cache-hit, …)
  readonly value: number;
  readonly provenance: 'derived' | 'architect'; // derived-from-envelope vs replaced by a measured value (§5)
}
// Absent everywhere ⇒ no named worlds: the base layer IS the design, evaluated once — today, bit-for-bit.
// The ambient batch lowers each AssumptionScenario to a contract Scenario: overrides → { "node|key": value }.

Two structural properties keep this small and honest:

  • Deltas over a shared base, never full copies. A scenario stores only the fact-assumption values it changes from the base layer (the design's own config). Three scenarios must not mean 3× the maintenance: edit a service time on the base and all three worlds inherit it; a scenario overrides only where its world genuinely differs. This mirrors how a range rides inside an instance (Instance.ranges) rather than duplicating the design.
  • Only fact-assumption keys. The authoring surface offers a scenario override only for role fact-assumption quantities (§2). A limit or an SLO is not a “world” — changing it is a design variant or a requirement change, which have their own surfaces. The role axis draws this boundary mechanically, so it can never drift.

4.2 Serialization — additive, following the lag-SLO discipline

The project document is at schema 6 today (app/core/src/document.ts line 23), the request-classes container added as R2 lands. Named scenarios are the schema-7 addition, following the exact discipline of the additive containers before them — guaranteeSlos, lagSlos (the schema-4 addition, app/core/src/document.ts line 47), and requestClasses (schema 6): a new top-level container, plain arrays (no Map), absent ⇒ empty ⇒ today bit-for-bit, and every older export loads unchanged (the migration chain 1..6 extends to 7). One decision to make deliberately: an override naming a node/key the design no longer contains. A request class is structural (it defines a commodity), so a dangling ref is a hard load error; a scenario override is a value substitution (the same shape Monte Carlo's applyOverrides tolerates by ignoring overrides on absent/non-fixed cells, content/sda/src/uncertainty.ts 463–479). The lean is to match that: a stale override is reported and skipped, not fatal — a scenario is a soft lens, not a structural claim.

5The hit — honest auto-created defaults “this could be a hit”

This is the move the owner flagged as the payoff: don't ask a lazy architect to invent three worlds — create the trio for him, pessimistic / real / optimistic, with values already filled in. The entire credibility of the feature rests on where those values come from: never invented, always derived from the design's own computed capacity, and badged as such.

5.1 Demand keys — fractions of the envelope

The envelope (§3) already computes the maximum sustainable demand at each origin. The trio's demand values are proposed fractions of it, so they are grounded in this design's capacity rather than any market figure:

WorldProposed demandWhy (the honest rationale)
real≈ 60% of envelopeComfortably inside the ρ ≤ 0.8 headroom band the tool already sizes to (TARGET_UTILIZATION = 0.8, content/sda/src/behaviors.ts line 23) — a plausible steady-state that leaves burst headroom.
optimistic≈ 30% of envelopeA quiet-launch / early-adoption world — low utilization, everything green with wide margin.
pessimistic≈ 110% of envelopeDeliberately past the edge: a stress world that should break something, so the architect sees which promise fails first under overload before it happens in production.
The exact fractions are a proposal, not a law. 30 / 60 / 110% are defensible anchors (real inside headroom, pessimistic past the edge), but they are a policy choice the owner should ratify (§9). What is not negotiable is the principle: every derived demand is a fraction of a number the tool computed for this specific design — there is no hard-coded “1,000 req/s” anywhere.

5.2 Ranged / non-demand keys — the polarity problem, honestly

A world is more than demand: pessimistic should also mean the unfavourable end of every soft assumption (a worse cache-hit ratio, a slower query), optimistic the favourable end, real the mode. Where an assumption is already declared as a range ({lo, hi} or {lo, mode, hi}, content/sda/src/manifest.ts 82–106), the trio picks its ends: pessimistic = the unfavourable bound, real = the mode (or midpoint), optimistic = the favourable bound. That requires knowing each key's polarity — which direction is “worse”.

Tension — the registry does NOT declare polarity today. There is no polarity flag on a key. A band's shape (minTargetMax) and an objective's direction encode a requirement's or a search's direction, but not the intrinsic “higher is worse” of a fact-assumption. Polarity is currently captured in two indirect places: the sensitivity matrix's declared expected-direction data (.../sensitivity-matrix.test.ts line 80), and empirically by the native solver's own per-knob sign probing (.../native/search.ts ~351–355 — it measures which way a metric moves with a knob). Resolution: derive the favourable/unfavourable end from that same sign the solver already computes against the design's SLOs (a key whose increase pushes a promise toward violation is “worse”), or add polarity to the role-axis metadata as an explicit field. Crucially: where polarity is genuinely undefined, the key stays at base in every world — no guessing. A world only diverges on an assumption whose direction is known. This is an open question for the owner (§9).

5.3 Provenance — derived, rendered distinctly, flipped on edit

Every auto-filled value carries provenance derived — a new fifth value on the existing provenance axis (documented | estimate | architect | default, content/sda/src/doc-model.ts line 139). This is one-form-per-kind at work: rather than invent a new badge form, the derived-default reuses the assumptions register's existing badge axis and adds one member. It renders distinctly — proposed wording: “wyprowadzone z pojemności projektu — podmień na zmierzone” (“derived from the design's capacity — replace with a measured value”). The moment the architect types a real number over a derived one, provenance flips derived → architect — which is exactly what the existing mechanical derivation already does for any value that diverges from its default (configProvenance: an instance value ≠ the default reads architect, .../doc-model.ts 536–537). No new mechanism; one new starting badge.

The honesty rule, stated once and plainly. SDA must not invent infrastructure numbers (project invariant: the tool must not lie). The auto-created trio contains no market data, no fabricated traffic figures, no guessed prices — only values derived from this design's own computed envelope and its declared ranges, each badged derived and openly labelled as a placeholder for a measurement. A world with no envelope to derive from (no origin, no SLOs) gets no demand override — the feature stays silent rather than fabricate one.

6View three — ranges & Monte Carlo (the cloud around a point)

The third view is already shipped. A scenario is a point; a range makes an assumption an interval, and sampling turns every conclusion into a distribution — a confidence. It composes with scenarios rather than competing with them.

Content's uncertainty module (content/sda/src/uncertainty.ts) draws N seeded samples from every declared Instance.ranges, evaluates each through the same EvaluateBatch a named world uses, and rolls the results into percentiles, histograms, per-SLO confidence (judged by the same queueing-aware verdict every surface reads), and a tornado. The relationship to scenarios is compositional: a range is a cloud around a point. In the assumption space, a scenario fixes the point; ranges declared on that scenario's inputs blur it into a distribution, and the Monte-Carlo run reports how often the SLOs survive the blur. The auto-created trio and Monte Carlo are two honesties that stack: the trio says “here are three worlds I sized for you”; Monte Carlo says “and here is how confident the real world's verdict is, given what you told me is uncertain about it.”

7Scenario-first authoring & ambient evaluation

The owner named the gesture: “I create scenarios in System, and on the nodes I assign parameters under that name.” The scenario is chosen first, in one place; editing an assumption on a node then writes into whichever world is active. Every world is evaluated continuously, so the comparison is always on screen.

7.1 The active-world lens — the same form as the class lens

The System panel (the whole-design roll-up every shell already shows, app/presenter/src/summary.ts) owns the scenario list + an ACTIVE selector. This is deliberately the same form as the planned request-class lens (docs/design/request-classes.html §7 — a filter that scopes what the canvas shows): one-form-per-kind says a “named overlay you select to scope the view” is one kind, so scenarios and classes share it. The rule while authoring:

  • A world is active ⇒ editing a fact-assumption parameter on a node writes the override into that world (and flips its provenance to architect, §5.3). The canvas shows that world's computed results.
  • No world is active (the base lens) ⇒ editing writes the shared base layer; the canvas shows the base evaluation — and, with no demand pinned, the envelope (§3).

7.2 All worlds evaluated ambiently → a comparison matrix

Because a named world lowers to a contract Scenario, evaluating all of them is a single EvaluateBatch call — the very batch Monte Carlo already runs at ~1 ms per evaluation, under the TASK-81 ambient loop (event-driven, debounced, latest-wins, idle = zero work). A three-world design costs three sub-millisecond evaluations after each edit; the result is a matrix view (metrics × scenarios) plus a per-scenario verdict column, with the active world highlighted on the canvas. This is the natural home for the north-star guidance loop (TASK-83): the next-move hint can be computed per world, or against the worst world, without new machinery.

11Non-goals

No invented world numbersThe auto-created trio derives every value from this design's own envelope and declared ranges — never a market figure, a fabricated traffic count, or a guessed price. No envelope / no range ⇒ no override (the feature stays silent). This is the invariant, not a nicety.
No mandatory demandThe tool never demands a demand to answer. The envelope is the default; a scenario is opt-in refinement. The whole design exists to avoid forcing a number.
Scenarios override only fact-assumptionsA world is a belief about the outside world, not a design variant (that is Improve / limits) or a requirement (that is an SLO). The role axis enforces this mechanically; a scenario can never silently retune a limit or loosen a promise.
