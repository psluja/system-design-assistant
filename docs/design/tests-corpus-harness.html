How the code proves itself

A guide to three parts of this repository — the tests, the harness, and the corpus. Read this before you read the code. It explains the one idea they share and where each part lives, so you can find your way and trust what you find.
System Design Assistant · developer onboarding · docs/design/tests-corpus-harness.html

1The one idea

This tool computes numbers that a professional will stake a real design on: how much traffic it serves, how fast, how much it costs, how reliable it is. The tool has one promise — it must not lie. Every number it shows is either backed by a source or marked unknown. It never guesses.

A promise like that cannot be kept by good intentions. It has to be proven. So the codebase proves its own correctness at three levels, and those three levels are exactly the three parts of this guide.

PartThe question it answersIn plain words
The testsDoes each piece behave as intended?The everyday safety net. Thousands of small checks, next to the code they check.
The harnessAre the equations solved correctly?Two independent engines must agree with each other, and both must agree with textbook mathematics. This is called verification.
The corpusAre they the right equations for the real world?Our predictions are compared against real, measured systems. This is called validation.

Verification and validation are different questions, and mixing them is how simulators quietly start lying. A perfectly-solved wrong model is still wrong. A right model solved by a broken engine is still wrong. You need both, and they are kept separate on purpose.

2The testsDoes each piece behave as intended?

Tests live next to the code they test, not in a separate folder. A file named optimize.ts is tested by optimize.test.ts in the same directory. There are about 220 such files across the project. When you open any source file, its test is one step away.

The most useful thing to know is that the file name tells you what kind of proof it is. The suffix is the signal:

File name ends with…What kind of testWhat it proves
.test.tsOrdinary unit testOne piece does what it should on chosen examples.
.property.test.tsProperty testThrows hundreds of random inputs at the code and checks that a law always holds — for example, "the result never becomes negative". It looks for the example you did not think of.
.differential.test.tsDifferential testTwo independent ways of computing the same answer must match. If they ever disagree, one of them is wrong — and we find out.
.e2e.test.tsEnd-to-end testA whole real design is run from start to finish, the way a user would. There are about 46 of these, including named reference architectures (a Netflix-style playback path, an Uber-style dispatch, a serverless checkout).
.golden.test.tsGolden testA generated file (such as a design document) must be byte-for-byte identical to a saved reference. If the output drifts, even by a space, the test fails.
Why the property and differential tests matter most.

Ordinary tests check the cases you imagined. Property tests and differential tests check the cases you did not imagine. They are the reason a claim in this project can be trusted rather than merely hoped for.

3The harnessAre the equations solved correctly?

The harness lives in engine/solver-contract/src/. Its job is to prove that the engine solves problems correctly — not on a handful of examples, but on hundreds of generated ones.

The idea is a simple three-step loop: make a problem, make the certified answer, then grade a candidate against it.

The parts

  • harness/generator.ts
    Builds the random designs. It does not just scatter noise — it aims at hard cases on purpose: requirements placed exactly on the edge of what is possible, extreme numbers, very deep chains, several requirements at once, and more.
  • harness/oracle.ts
    Computes the answer we treat as certified. It uses the reference engine and then double-checks that answer a second, independent way.
  • harness/harness.ts
    Runs a candidate engine and compares it to the oracle. A disagreement fails the test and prints a one-line command to reproduce that exact case.
  • harness/metamorphic.ts
    Checks laws that must hold no matter what — for example, "double the traffic, and the answer doubles too." These catch bugs that both engines might share.
  • native/
    The fast engine — the candidate, the one the product actually runs.
  • incumbent/
    The reference engine (well-tested industrial solvers). It is slower but trusted, so it is the oracle's source of truth.
  • capability/
    The six things the engine can do — evaluate a design, size it to meet a goal, repair it, explain why it is impossible, list valid options, and run many scenarios at once. Each has its own test.
One detail worth knowing early.

Every generated design carries the number (the "seed") that produced it, and the code never uses the clock or true randomness. So any result is exactly reproducible: given the seed, you rebuild the same design, byte for byte. A bug found at night can be replayed in the morning.

4The corpusAre they the right equations for the real world?

The harness proves we solve the equations correctly. It cannot tell us the equations are right about reality — only measurement can do that. That is the corpus, which lives in calibration/.

The corpus is a small collection of real, measured systems. For each one, we hold a saved design, the workload, the source of the published numbers, and the numbers themselves. The tool predicts each system with its own engine, and we report how far off we are.

The data

  • calibration/corpus/*.json
    One file per measured system. Today there are eight, covering four real architectures: DeathStarBench (a microservice social network), TechEmpower (a web-framework benchmark), Redis (an in-memory store), and Kafka (a streaming platform). Each file names who measured it, what they measured, and the source link.
  • calibration/*.sda.json
    The same systems drawn in our own design format — the model the tool actually runs.

The code

  • src/predict.ts
    Runs our engine over a model and reads back the number we want to compare (for example, the throughput ceiling).
  • src/fit.ts
    Tunes the handful of unknown component defaults to fit reality, and reports the residual — the error that remains after the best fit. That leftover error is the honest output: not "we matched", but "here is exactly how far off, and where". It also holds out each system in turn and predicts it from the others, which tests how well the model generalizes to systems it has not seen.
  • src/report.ts → CALIBRATION-REPORT.md
    Writes the full derivation: measured versus predicted, per system.
  • src/coverage.ts + src/fidelity.ts → docs/FIDELITY.md
    Generate the honest headline: which behaviors are validated against real systems, which rest on theory, and which are cited facts. Green only where evidence exists; every gap is shown in plain sight.
  • src/baseline.ts
    The pinned numbers a standing guard watches. If a change makes a prediction worse than the pinned value, the guard fails and names the system.
Adding a measured system is data, not code.

You write one new JSON file and run pnpm calibrate. The report re-fits and updates itself. No model change is required for a new datapoint, and no AI is involved.

5How to start reading

If you have thirty minutes and want to understand the idea from the code itself, read in this order:

  1. docs/FIDELITY.md — the honest state of the whole thing, in one page. Start here to see what is proven and what is not.
  2. engine/solver-contract/src/harness/harness.ts — the grading loop. It shows how a candidate is held to the oracle.
  3. engine/solver-contract/src/harness/generator.ts (the top comment) — what kinds of hard designs are generated, and why.
  4. calibration/src/fit.ts — how a real measurement becomes a residual, and how the leave-one-out check works.
  5. Any *.e2e.test.ts under app/mcp/ — a real design exercised end to end, which ties everything together.
