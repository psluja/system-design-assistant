About this document & the project (context for the outside reader).

SDA (System Design Assistant) is a fully client-side tool in which a software architect draws a system architecture on a canvas — services, databases, queues, connected by wires — and the tool computes that design: each component's load, latency, availability and cost, checked against declared requirements. One of its features exports a design document — the written deliverable an architect hands to reviewers. Today that export is a modest Markdown file; this document designs its grown-up successor.

verified modelthe canvas design plus everything the engines computed about it — the single source every sentence of the generated document must come from.
assumptionany input number the computation rests on: a declared traffic level, a timeout, a cache-hit ratio. Mature engineering documents list them explicitly, with their origin.
provenancewhere a number came from: an official document, an estimate, the architect's own hand, or a catalog default.
C4 container viewan industry-standard diagram style showing a system's deployable pieces and the connections between them.
DES / simulationSDA's discrete-event simulation — replays the system over time to obtain latency percentiles, retry behavior and failure rates that averages cannot give.

1Why, and why now

Two reasons. First, our own honest assessment: SDA verifies designs excellently but barely delivers — architects are judged by documents, not canvases. Second, an unfair advantage no competitor has: hand-written design documents drift into fiction the week after they're written; a document generated as a pure function of the verified model is true by construction, every number computed, every assumption carrying its origin.

Industry research (Google's design-doc culture, the arc42 template, RFC practice at HashiCorp/Twitter and peers) converges on a canon of sections — summary, context, goals and explicit non-goals, requirements, an assumptions register, architecture views, detailed design, reliability, cost, alternatives considered, risks and open questions. The two hallmarks of a production-grade document are exactly the two we can now generate better than a human: the assumptions register (we know every knob and its provenance) and alternatives considered (we can compute the ranked alternatives, not recall them).

3The assumptions register — the heart of v2

Every number the computation rests on, one row each, with a provenance badge. This is what turns “trust me” into a reviewable document:

Rule inherited from the whole product: the register lists what IS; it never pads. A design with three assumptions gets three rows. Retry policies, flow transforms (with their level: wire / instance / catalog), traffic origins, deployment modes, SLO bands — all land here automatically because they are all just config the generator can enumerate against the catalog.

4Architecture — one document model, two renderers

DecisionChoice & why
Self-containmentONE .html file: inline CSS, inline SVG charts/diagram, zero external requests — mailable, archivable, works offline, safe under any CSP. Print stylesheet → clean PDF from the browser.
Chartshand-rendered SVG from chart series (bars, waterfall, line) — no chart library (bundle size, CSP, determinism). Our design docs already prove the aesthetic.
C4 diagramgenerated from the canvas layout the architect already arranged (tidy positions), so the document mirrors what they see; protocols + rates label the edges.
Load→latency curvethe generator may run extra forward evaluations (a 5–7-point load sweep, ~6 ms each) — computed exclusively at GENERATION time, never persisted as fake state.
Alternatives sectionoptional + async (clingo/MiniZinc solvers): generated when the caller passes compare results or opts in; the section states its method honestly.
Surfacesweb “Design doc” button → downloads .html (Markdown stays in a dropdown); VS Code command → opens the .html beside the canvas + saves next to the .sda.json; MCP generate_doc{format?: 'html'|'markdown'}.

6Non-goals

Anything outside the computed domain — security reviews, rollout plans, organizational contextOWNER RULING (2026-07-03): not skeletons, not placeholders — completely absent. The document covers only what SDA computes well, and says so in ONE honest scope sentence in the summary ("this document covers the quantitative envelope of the design: capacity, latency, availability, cost and their assumptions"). A scope statement is honesty; an empty template section is filler.
