Calibration #1 — DeathStarBench Social Network vs SDA

The first ground-truth entry in SDA's calibration corpus. We build the DeathStarBench social-network microservice architecture in SDA, feed it the benchmark's own verified workload, and hold SDA's predictions against the published measured behaviour — honestly separating tunable defaults from structural gaps. Update (re-calibration): the #1 structural gap this entry found — the DES tail blind at datastores — has now been fixed in pure content config; §4.5 holds the before/after and the honest re-assessment.
SDA calibration corpus · 2026-07-06 (re-calibrated 2026-07-07) · entry #1 · research + modelling + analysis + fix · content/sim config change (uncommitted)
What this is (context for the outside reader).

SDA (System Design Assistant) is a client-side tool that turns an architecture drawing into a typed-property graph and computes it: a fast scalar forward pass (throughput, overflow, cost, queueing-aware mean latency) and a discrete-event simulation (DES — true p50/p95/p99 tails). The question this corpus answers, one real system at a time, is blunt: can SDA model reality, and where does it diverge? DeathStarBench (Gan et al., ASPLOS 2019) is the standard open-source microservices benchmark; its social network is a broadcast-style Twitter clone of ~11 Thrift services over MongoDB / Redis / Memcached. It is an ideal first datapoint because its architecture is fully open-source and several papers publish its measured behaviour.

Headline finding (original run). SDA independently rediscovers the capacity/flow skeleton of the real system — it names the same high-load bottleneck the paper does (the back-end post-storage database), reproduces the fact that compose-post is the longest-latency operation, and lands MongoDB's share of end-to-end latency in the right ballpark (~11% predicted vs 8.5% measured). But its time/tail model did not reproduce DeathStarBench's single defining measurement — the p99-vs-load blow-up at saturation — because SDA's DES treated fixed-throughput datastores (Mongo/Redis/Memcached) as infinite-server pure delays that never queue. The residual was roughly half TUNABLE (generic service-time/concurrency defaults, not Thrift-microservice-calibrated) and half STRUCTURAL (no datastore queueing in the DES, no CPU-bound front-end tier, no persistence-fork / thread-overhead / cache-dynamics primitives).

Re-calibration finding (after the fix). The #1 structural gap is now closed. A pure-content change — give each fixed-throughput datastore a connection-pool config so SDA's existing poolStation mechanism forms a finite-server M/M/c in the DES, plus a one-line trigger tweak so the pool queues by physics (no caller deadline required) — makes the datastore p99 rise with load and blow up at saturation exactly where the scalar pass already flagged it. The store's DES utilisation now tracks the analytic ρ (0.28 → 0.97 across the sweep) instead of sitting at ≈0, and the end-to-end p99 climbs from a flat 165 ms to 1,810 ms at ρ≈1.1 (≈ the paper's 2 s SLA) and ~7,000 ms past it, with goodput now collapsing to the Mongo-limited capacity instead of falsely matching the offered load. Capacity is byte-identical (the scalar/analytic pass is unchanged; only the DES tail gained queueing). The other structural gaps (CPU-bound front-end, persistence-fork stall, thread-execution economics, cache dynamics, fan-out tail amplification) remain — see §4.5 for the honest residual. Full before/after in §4.5.

3The SDA model

The model is calibration/deathstarbench-social.sda.json (29 nodes, 35 wires), built entirely from the current SDA catalog. Component capacities are catalog defaults — this is deliberately the honest "out-of-the-box" SDA prediction, so the calibration measures SDA as it ships, not a hand-fitted version.

3.1 Mapping & substitutions

  • Services → compute.service (default: 500 concurrency / 20 ms handler ⇒ 25,000 RPS capacity). Where the C++ uses std::async (ComposePost phase 1, TextService), the node's latencyComposition is set to parallel (max) to match; all others keep the default sequential (sum).
  • nginx → proxy.nginx with a CPU station (cpuCores 3 × cpuTimePerRequestMs 1 ms ⇒ ~3,000 RPS). substitution note SDA now has a CPU-bound tier primitive, so nginx is modelled as CPU-bound (the low-load bottleneck the paper reports) rather than a thin fixed-latency hop; the CPU time equals its 1 ms latency so the low-load sojourn — and MongoDB's latency share — is unchanged.
  • MongoDB / Redis / Memcached → db.mongodb / cache.redis / cache.memcached (defaults: 10,000 op/s·5 ms / 100,000 op/s·0.5 ms / 200,000 op/s·0.5 ms). All are fixed-throughput (no concurrency knob) for the scalar pass — and, after the fix, each carries a connection-pool config that gives it a finite-server M/M/c in the DES so its p99 can queue (§4.5), with the scalar capacity unchanged.
  • Mixed workload → one client + probability splits. SDA's simulate declines request-classes, so the 60/30/10 mix is a single client at the load point with prob(0.6/0.3/0.1) wire splits at nginx. A real service that answers several RPCs with different downstreams (HomeTimeline read vs write, PostStorage read vs write) is modelled as per-RPC nodes so each operation's call graph stays exact.

4.5The fix & re-calibration — datastore queueing in the DES

The gap above was not a missing primitive — it was missing config. SDA already has a poolStation mechanism (content/sda/src/sim.ts) that turns a fixed-throughput node into a finite M/M/c that queues, driven entirely by two data knobs; the datastores simply never declared them. The fix is therefore pure content, no engine type-branch, honouring SDA's domain-agnostic invariant.

4.5.1 What changed (content + one config-read trigger)

  • Datastores now declare a connection pool. db.mongodb / cache.redis / cache.memcached (and the archetype db.sql / db.cheap) get a connectionPool (c = physical service parallelism) + connectionHeldMs (server-occupancy time). One form: held = how long a request holds its server, and c = throughput × held is the Little's-law in-flight concurrency — so c / held equals the store's existing throughput ceiling and capacity does not move. Mongo ⇒ M/M/50 (10,000 op/s × 5 ms query, within maxPoolSize 100); Redis ⇒ M/M/1 (single-threaded command loop); Memcached ⇒ M/M/4 (4 worker threads). Config-only — not the poolOverflow relation, because the universal overflow band already flags saturation at the same threshold; adding it would duplicate the verdict and disturb the byte-identical scalar pass.
  • The queue trigger was decoupled from the caller deadline. poolStation used to form the M/M/c only when a borrow-timeout (maxQueueWaitMs) was set. A finite pool queues by physics whether or not a deadline is armed, so it now forms on the pool alone; the deadline is layered on top for reneging, unchanged. (The RDS Proxy, which ships a 120 s borrow timeout, already queued — it is byte-identical; a design with no pooled node is byte-identical.)
Byte-identity, verified. The scalar forward pass and the analytic queueing twin read the datastore's capacity (its throughput ceiling) exactly as before — the pool knobs feed only the DES. The full repo test suite (2,535 tests) stays green with the change; only the DES tail moves.

4.5.3 Did it close the #1 gap? Yes — and here is what it honestly does NOT do

The re-calibration verdict on the defining metric flips from DIVERGES to REPRODUCED: SDA's DES now bends the tail at the saturating tier. But honesty demands the boundaries of the fix:

  • No drops, only unbounded growth, past saturation. → Since closed (opt-in, §4.5.4). A datastore here has no query/borrow timeout (maxQueueWaitMs), so beyond ρ=1 the queue grows without bound and the p99 is a window-dependent number (the ~7,000 ms figures scale with the measured window), not a steady state — there is no steady state past saturation. Reneging (bounded tail + a real errorRate/drops) needed a datastore query-timeout knob — and that knob is now expressed by exactly that maxQueueWaitMs station wait-deadline, the SAME machinery the RDS Proxy's borrow timeout uses (§4.5.4). Set it and past-saturation queries renege (bounded drops + a real errorRate); unset (the default), it is byte-identical to today.
  • The median goes bimodal past saturation. At 15k RPS the p50 falls to ~2 ms while p95/p99 sit in the seconds: the fast paths that never touch the saturated Mongo keep completing quickly, so the median tracks them and only the upper percentiles carry the mongo-bound tail. Realistic, but it means the median is not a saturation signal — the p95/p99 are.
  • The hot-path (analytic) latency treated the datastore as a pure delay. → Now CLOSED (§4.5.4). The original fix was DES-only: queueing.ts's nodeQueues reported the datastore's sojourn as its flat service time (rising ρ, flat latency), so the live canvas showed saturation as ρ but not as queue-inflated latency. The analytic twin now forms the SAME M/M/c pool station the DES does, reading the SAME pool primitives through one shared definition (graph-read.ts queueStation, called by both engines), so its sojourn rises with load and answers ∞ at ρ≥1 in lock-step with the DES (§4.5.4) — the RDS Proxy's analytic pass is fixed by the same change.
  • The OTHER structural gaps from §5.2 remain (the CPU-bound front-end is since resolved separately). No persistence-fork stall, no thread-execution economics, static (not dynamic) cache hit-rate, and fan-out that amplifies rate but not tail. Those remain the real residual — this fix closes datastore queueing; the CPU-bound front-end tier is now closed by the CPU-station primitive (§5.2), which makes nginx the low-load bottleneck the paper reports.

4.5.4 Follow-ups landed — the analytic twin queues too (residual #3), a datastore query timeout (residual #1)

The two residuals §4.5.3 flagged as open — the DES-only nature of the fix, and the lack of past-saturation drops — have now both been closed, in the same pure-config spirit: no engine type-branch, capacity byte-identical, and designs with no pooled node untouched (the sacred pin).

  • Residual #3 — the analytic asymmetry — CLOSED. The hot-path analytic twin (content/sda/src/queueing.ts nodeQueues) now forms the SAME finite-server M/M/c a pooled store gets in the DES: c = the connectionPool slots, μ = 1/connectionHeldMs, so c·μ = pool/held equals its declared throughput ceiling (ρ and capacity byte-identical) and its per-hop sojourn is now the Erlang-C wait, not a flat pure delay. Both engines read those pool primitives through ONE shared definition — graph-read.ts queueStation, called by both nodeQueues and the DES poolStation — so they can never drift (a new analytic-⇄-DES differential test pins a pooled datastore's wait within tolerance). The live canvas therefore shows queue-inflated latency, not just rising ρ.
  • Residual #1 — past-saturation drops — CLOSED (opt-in). A datastore query timeout is expressed by the SAME station wait-deadline the RDS-Proxy borrow timeout already uses — maxQueueWaitMs (one form for one kind: a wait deadline before reneging). Set it on a datastore and past capacity a query whose wait for a pooled connection exceeds it RENEGES: bounded drops, goodput capped at the pool ceiling, a real errorRate — instead of the unbounded, window-dependent queue. Default (unset) = today, byte-for-byte. This is why the raw §4.5.2 overload figures (~7,000 ms) are window-dependent: they are the NO-timeout regime; arming a query timeout replaces that unbounded tail with a bounded-tail-plus-drops outcome.

5Residual analysis — tunable vs structural

The honest split. A tunable miss is a default that is simply mis-set for this domain and an auto-fit could correct. A structural gap is a behaviour SDA has no primitive to express — the real research finding.

5.2 STRUCTURAL — gaps SDA cannot express (the real residual)

  • No queueing at fixed-throughput datastores in the DES. RESOLVED (§4.5) Mongo/Redis/Memcached had no concurrency knob ⇒ PURE_DELAY, so the p99-vs-load blow-up — the whole point of a tail benchmark — was unreachable at the tiers that actually saturate. This was the #1 finding, and it is now fixed: a connection-pool config gives each store an M/M/c in the DES, and its p99 now blows up at saturation (§4.5). The two sub-residuals this raised are also closed (§4.5.4): the analytic hot-path twin now forms the same pool M/M/c (its sojourn rises with load and answers ∞ at ρ≥1, in lock-step with the DES), and a datastore query timeout (maxQueueWaitMs, opt-in) makes past-saturation queries renege — bounded drops instead of unbounded growth.
  • RESOLVED (CPU primitive) CPU-bound front-end tier. nginx-thrift (OpenResty + Lua + php-fpm-style processing) is where the paper says the capacity bottleneck sits at low load. SDA now models this: cpuCores + cpuTimePerRequestMs give nginx a CPU M/M/c station (capacity = cores ÷ cpuTime), so with a modest core budget it becomes the low-load bottleneck the paper reports — the calibration model gives it 3 cores × 1 ms ≈ 3,000 RPS, just below the post-storage Mongo tier, and at 1000 RPS it is the busiest tier (ρ≈0.33). The CPU time equals its 1 ms proxy latency, so the low-load per-hop sojourn — hence MongoDB's measured 8.5% latency share — is unchanged; only the ceiling moves. The core count is an estimate (the paper gives no absolute front-end RPS). What stays out of scope is a cascade/back-pressure model (a back-end stall saturating the front-end, Sinan) and the deep CPU economics (allocator/GC/lock contention; a true tandem queue rather than the binding MIN).
  • No background-persistence / stateful-stall primitive. Sinan's root cause is Redis' periodic fork + copy-on-write snapshot stalling request serving every minute. SDA has no notion of a periodic self-inflicted stall — a time-domain, stateful failure mode.
  • Capacity is Little's law, not OS-thread economics. SDA sizes a tier by concurrency ÷ service-time. Eyerman's result — thread-per-RPC clone/exit burning ~23% of CPU, and fibers unlocking ~6× — is an execution-model cost SDA cannot represent; the 15k→90k jump is invisible.
  • Static cache hit-rate, not dynamics. SDA uses a fixed miss fraction. Real hit rate shifts with warmup, eviction and working-set-vs-load — and that shift changes which tier saturates as load rises (the bottleneck-shift the paper reports).
  • Fan-out amplifies rate but not tail. The follower fan-out is modelled as a rate multiplier (ratio(k)); the callee's latency is added once, not k×. Tail-at-scale from k parallel/sequential writes (the classic fan-out tail) is not captured.
  • Mixed workload collapsed to one river. Because simulate declines request-classes, distinct operations become probability splits — which makes the analytic response sum mutually-exclusive branches (an inflated 183 ms at the client). The DES fork-join handles it correctly (p99 165 ms), but per-operation tails are not separable in one run.

6Verdict

Does SDA model this real system? Partly — and honestly about the part it doesn't. As a capacity and flow verifier, SDA is genuinely faithful: from nothing but the catalog defaults it independently rediscovers the paper's high-load bottleneck (the back-end post-storage database), reproduces that compose-post is the heaviest operation, puts MongoDB's latency share within ~1.3× of the measured 8.5%, and its scalar overflow verdict correctly declares the design infeasible past ~3,571 RPS mixed. Those are non-trivial hits for a domain-agnostic engine fed a real architecture.

Where it diverged was its time model — and the biggest piece is now fixed. SDA originally did not reproduce DeathStarBench's defining behaviour — the p99 knee at saturation — because its DES could not queue a fixed-throughput datastore. §4.5 closes exactly that: the datastore p99 now rises with load and blows up at saturation, in pure content config, capacity byte-identical. What remains structural is the rest — no CPU-bound front-end, no persistence-fork stall, no thread-execution cost, only static cache behaviour, and fan-out that amplifies rate but not tail. Those are not fixable by tuning a number, and this fix does not touch them.

For the future auto-fit. The free variables that a fit-loop would turn first are: compute.service.perRequestDuration (↓ ~4–10×), compute.service.concurrency (↓ to a real pool so a compute tier can be the bottleneck), db.mongodb.throughput, and the three estimated transforms (miss ratio, page fan-out, follower fan-out). Now that the DES tail can bend at the datastore, tuning those actually re-shapes a real p99 curve rather than a flat line — the fit has a gradient to follow. The single highest-value change this calibration argued for has now been made: fixed-throughput datastores have an M/M/c form in the DES so their p99 rises with load (§4.5). Both follow-ups it argued for have since landed too (§4.5.4): the analytic twin's matching pool form — one shared queueStation definition both engines read, so the live canvas tail rises with load in lock-step with the DES — and a datastore query timeout (the shared maxQueueWaitMs wait-deadline) so overload produces bounded drops rather than unbounded growth.
